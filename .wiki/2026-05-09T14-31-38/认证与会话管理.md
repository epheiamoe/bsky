# 认证与会话管理

## 架构全景

认证系统由三层构成，每层有明确职责：

```
┌─────────────────────────────────────────────────┐
│                    UI 层                         │
│  useAuth (React Hook) ←→ LoginPage / App         │
├─────────────────────────────────────────────────┤
│                   Store 层                       │
│  createAuthStore() — 纯状态机 + 发布订阅          │
├─────────────────────────────────────────────────┤
│                Client 层                         │
│  BskyClient — AT Protocol 会话管理 + JWT 刷新     │
└─────────────────────────────────────────────────┘
```

**Store 层**（`createAuthStore`）是纯逻辑，不依赖 React；**Hook 层**（`useAuth`）将其桥接到 React 组件生命周期；**Client 层**（`BskyClient`）处理与 PDS 的网络通信。PWA 与 TUI 共享前两层，差异仅在凭证来源与会话持久化策略。

[来源](packages/app/src/stores/auth.ts#L4-L17) | [来源](packages/app/src/hooks/useAuth.ts#L6-L23)

## 一、Store 层：createAuthStore

`createAuthStore()` 返回一个 `AuthStore` 对象，包含状态、方法和极简的发布订阅机制。

### AuthStore 状态

| 字段 | 类型 | 说明 |
|---|---|---|
| `client` | `BskyClient \| null` | 登录后创建的 API 客户端实例 |
| `session` | `CreateSessionResponse \| null` | 包含 accessJwt、refreshJwt、handle、did |
| `pdsUrl` | `string \| null` | 用户实际 PDS 地址（登录后自动发现） |
| `profile` | `ProfileView \| null` | 用户个人资料（登录后异步获取） |
| `loading` | `boolean` | 登录进行中 |
| `error` | `string \| null` | 错误消息 |

[来源](packages/app/src/stores/auth.ts#L4-L17)

### login 流程

```
login(handle, password, pdsUrl?)
  │
  ├─ 1. 设置 loading=true, error=null, _notify()
  │
  ├─ 2. new BskyClient({ pdsUrl })  ← 指定 PDS 或使用默认 bsky.social
  │
  ├─ 3. c.login(handle, password)   ← 调用 createSession API
  │      ├─ 返回 session (accessJwt, refreshJwt, handle, did)
  │      ├─ 从 DID Document 发现用户实际 PDS
  │      └─ 更新 this.pdsUrl + 重建 ky 实例
  │
  ├─ 4. store.session = session
  │    store.pdsUrl = c.pdsUrl
  │    store.client = c
  │
  ├─ 5. c.getProfile(handle)        ← 获取用户资料
  │      └─ store.profile = 结果
  │
  └─ 6. loading=false, _notify()
```

关键设计：**PDS 发现机制**。即使指定了入口 PDS（如 `bsky.social`），`BskyClient.login()` 会从 `createSession` 返回的 `didDoc` 中提取用户的真实 PDS，或通过 `com.atproto.identity.resolveDid` 查询。后续所有请求都发往此真实 PDS。

[来源](packages/app/src/stores/auth.ts#L29-L45) | [来源](packages/core/src/at/client.ts#L136-L177)

### restoreSession 流程

不重新输入密码的恢复路径：

```
restoreSession(session, pdsUrl)
  │
  ├─ 1. new BskyClient() + c.restoreSession(session, pdsUrl)
  │       ├─ 设置 this.session
  │       ├─ 设置 this.pdsUrl
  │       └─ 重新创建 ky 实例（携带 JWT 刷新钩子）
  │
  ├─ 2. store.session = session, store.pdsUrl = pdsUrl, store.client = c
  │
  └─ 3. c.getProfile(session.handle)  ← 异步
         ├─ 成功 → store.profile = profile
         └─ 失败 → 检查 isAuthenticated()
                     ├─ 未认证 → 清除状态, error='session_expired'
                     └─ 已认证 → 忽略错误
```

注意 `restoreSession` 的 getProfile 是异步的且不阻塞 UI——store 会先设置 session 和 client，profile 稍后到达。如果 session 已过期（refresh 也失败），`BskyClient.restoreSession` 不会自动刷新（它只设置状态），第一次 API 调用失败后 `_withRefresh` 钩子才会触发刷新尝试。

[来源](packages/app/src/stores/auth.ts#L47-L65) | [来源](packages/core/src/at/client.ts#L615-L624)

### 发布订阅机制

Store 使用最简模式：`subscribe(fn)` 注册监听器，返回取消函数；状态变更时调用 `_notify()`。

```typescript
subscribe(fn) { store.listener = fn; return () => { store.listener = null; }; }
_notify() { if (store.listener) store.listener(); }
```

[来源](packages/app/src/stores/auth.ts#L67-L71)

## 二、Hook 层：useAuth

`useAuth()` 将 `createAuthStore` 桥接到 React：

```typescript
function useAuth() {
  const [store] = useState(() => createAuthStore());  // 惰性初始化，组件生命周期内仅一次
  const [, force] = useState(0);
  const tick = useCallback(() => force(n => n + 1), []);
  useEffect(() => store.subscribe(tick), [store, tick]);
  return { client, session, pdsUrl, profile, loading, error, login, restoreSession };
}
```

**桥接模式**：Store 本身是纯对象，不关心 React。`useState(0)` + `subscribe` 实现了 "store 变更 → React re-render" 的映射。每次 Store 状态更新后，`_notify()` → `tick()` → `force(n+1)` → 组件重新渲染。

这与项目中所有 Hook（`useTimeline`、`useCompose`、`useAIChat` 等）使用的模式一致，详见 [React Hooks 架构与 Store 模式](react-hooks-架构与-store-模式.md)。

[来源](packages/app/src/hooks/useAuth.ts#L6-L23)

## 三、Client 层：BskyClient 与会话管理

`BskyClient` 是 AT Protocol 的完整客户端，详见 [BskyClient：AT Protocol 客户端实现](bskyclient-at-protocol-客户端实现.md)。此处聚焦认证相关部分。

### 会话创建

`BskyClient.login(handle, password)` 调用 `com.atproto.server.createSession`，返回包含 `accessJwt`、`refreshJwt`、`handle`、`did` 的响应。之后通过 DID Document 中的 `#atproto_pds` service 端点发现用户的实际 PDS。

[来源](packages/core/src/at/client.ts#L136-L177)

### 自动 JWT 刷新

核心机制是 Ky 的 `afterResponse` 钩子 `_withRefresh`：

```
请求 → 400 响应
       ├─ 错误类型是 ExpiredToken 或 InvalidToken
       │     ├─ 创建 _refreshPromise（单例，防并发）
       │     │     └─ 用 refreshJwt 调用 com.atproto.server.refreshSession
       │     │           ├─ 成功 → 更新 this.session
       │     │           └─ 失败 → this.session = null
       │     ├─ 等待 _refreshPromise 完成
       │     ├─ 如果 session 有效，用新 accessJwt 重试原请求
       │     └─ 返回重试结果
       └─ 其他 400 错误 → 打印错误日志
```

**并发保护**：`_refreshPromise` 确保多个并发请求同时遭遇 400 时只发起一次 refresh。所有等待者共享同一个 promise，刷新完成后依次重试。

[来源](packages/core/src/at/client.ts#L61-L110)

### 三种 API 端点路由

`BskyClient` 维护三个 Ky 实例，根据请求类型自动选择：

| 实例 | 端点 | 用途 |
|---|---|---|
| `this.ky` | 用户 PDS + JWT | 需要认证的写操作和私有数据 |
| `this.publicKy` | `public.api.bsky.app` + 无认证 | 公开数据（profile、thread、search） |
| `this.chatKy` | `api.bsky.chat` + JWT | DM 聊天 API |

如果当前已登录，`getProfile`、`getPostThread` 等方法会优先使用认证实例（`this.ky`），否则降级为公开实例。

[来源](packages/core/src/at/client.ts#L51-L129)

### 存储层：useSessionPersistence

- `saveSession(session)` — 序列化写入
- `clearSession()` — 移除键

[来源](packages/pwa/src/hooks/useSessionPersistence.ts#L1-L27)

### 生命周期（App.tsx）

PWA 的 `App` 组件管理完整的会话生命周期：

```
应用启动
  │
  ├─ 检查 localStorage 是否有已保存的 session
  │     ├─ 有 → restoreSession({ accessJwt, refreshJwt, handle, did }, pdsUrl)
  │     │         └─ setIsLoggedIn(true)
  │     └─ 无 → 显示 LoginPage
  │
  ├─ 用户手动登录 → handleLogin(handle, password, pdsUrl?)
  │     ├─ 调用 store.login()
  │     ├─ 成功 → saveSession() 到 localStorage
  │     │         └─ setIsLoggedIn(true)
  │     └─ 失败 → 显示错误消息
  │
  ├─ session 变更监听
  │     └─ session + client 有效 → saveSession()
  │
  └─ 认证错误（session 过期）
        └─ 收到 authError + isLoggedIn → clearSession()
                                         → setIsLoggedIn(false)
                                         → 回到 LoginPage
```

`LoginPage` 组件提供表单输入 handle、password 和可选的 PDS URL，提交后调用 `onLogin`。PDS 输入框有醒目的黄色警告样式，明确标注"仅适用于技术用户"。

[来源](packages/pwa/src/App.tsx#L44-L211) | [来源](packages/pwa/src/components/LoginPage.tsx#L6-L135)

### 与 TUI 的区别

| 维度 | PWA | TUI |
|---|---|---|
| 凭证来源 | 用户输入（LoginPage 表单） | `.env` 文件（`BLUESKY_HANDLE` / `BLUESKY_APP_PASSWORD`） |
| 会话持久化 | `localStorage`（`bsky_session`） | 无——每次启动从 `.env` 重新登录 |
| 配置工具 | 设置页 UI | `SetupWizard` 交互式向导 + `bsky-tui.config.json` |
| 重连策略 | session 过期后清除，回到登录页 | `useEffect` 检测 `isAuthenticated()` 变化，自动重新 login |

[来源](packages/tui/src/cli.ts#L64-L93) | [来源](packages/tui/src/components/App.tsx#L149-L163)

## 五、TUI：.env 凭证配置

### 启动流程

```
cli.ts 启动
  │
  ├─ 加载 .env（从项目根目录 + 当前工作目录）
  │     ├─ BLUESKY_HANDLE=your-handle.bsky.social
  │     ├─ BLUESKY_APP_PASSWORD=your-app-password
  │     └─ BLUESKY_PDS=https://your-pds.example.com  (可选)
  │
  ├─ getConfigFromEnv() → 读取 + 合并 tui.config.json
  │     ├─ 必须：handle + password
  │     └─ 可选：pdsUrl
  │
  ├─ 有完整配置 → 渲染 App，传入 config
  │     └─ App.useEffect: 自动 login(config.blueskyHandle, ...)
  │
  └─ 无配置 → 渲染 SetupWizard 交互向导
        └─ 完成 → 写入 .env + bsky-tui.config.json → 重新加载 → 渲染 App
```

[来源](packages/tui/src/cli.ts#L64-L116)

### SetupWizard

交互式向导引导用户逐步输入：handle → 密码 → PDS（可选）→ LLM 提供商 → 模型 → API Key → 语言 → 场景模型配置。完成后：

1. 写入 `.env` 文件（仅凭证：`BLUESKY_HANDLE`、`BLUESKY_APP_PASSWORD`、可选 `BLUESKY_PDS`）
2. 调用 `saveTuiConfig()` 写入 `bsky-tui.config.json`（非凭证设置：模型、语言、场景配置等）

这种"凭证 vs 配置"分离的设计使得 `.env` 可以安全地添加到 `.gitignore`，而 `bsky-tui.config.json` 同样被 gitignore。

[来源](packages/tui/src/components/SetupWizard.tsx#L64-L106)

### 自动重连

TUI 的 `App` 组件包含会话过期检测：

```typescript
useEffect(() => {
  if (!authLoading) login(config.blueskyHandle, config.blueskyPassword, config.blueskyPds);
}, []);  // 初始自动登录

useEffect(() => {
  if (client?.isAuthenticated()) {
    setWasAuthenticated(true);
  } else if (wasAuthenticated) {
    login(config.blueskyHandle, config.blueskyPassword, config.blueskyPds);  // 重连
  }
}, [client]);  // 检测 session 丢失
```

这一机制确保系统休眠后恢复时，TUI 能自动使用 `.env` 中的凭证重新登录，用户无需交互。

[来源](packages/tui/src/components/App.tsx#L149-L163)

## 六、认证类型定义

`CreateSessionResponse` 是 AT Protocol 标准响应：

```typescript
interface CreateSessionResponse {
  accessJwt: string;    // 短期访问令牌（通常 2 小时）
  refreshJwt: string;   // 长期刷新令牌（通常 90 天）
  handle: string;
  did: string;
  email?: string;
  emailConfirmed?: boolean;
  emailAuthFactor?: boolean;
}
```

`DidDocument` 中的 PDS 服务端点用于路径发现：

```typescript
interface DidDocument {
  id: string;
  alsoKnownAs?: string[];
  service?: Array<{
    id: string;          // "#atproto_pds"
    type: string;        // "AtprotoPersonalDataServer"
    serviceEndpoint: string;  // 用户实际 PDS URL
  }>;
}
```
