# 测试模式与调试技巧

本项目的测试体系基于 Vitest，所有端到端测试都调用真实的 Bluesky API 和 AI 提供商（DeepSeek），不做任何 mock。[集成测试体系](集成测试体系.md) 已经概述了整体架构，本文聚焦实操层面：**被注释的创建帖子测试为什么存在、如何激活、以及调试时的实用技巧**。

---

## 被注释的创建帖子测试：设计决策而非遗漏

打开 `packages/core/tests/e2e.test.ts`，你会发现三个被 `/* ... */` 包裹的测试块——`[2] Feed: post, read, search, thread flatten`、`[3] Image: upload blob, embed in post`、`[4] AI: tool calling for post analysis`——以及 `afterAll` 中注释掉的清理逻辑。同样的模式出现在 `feed.test.ts` 和 `ai_integration.test.ts`：创建帖子的代码被整体注释，依赖该帖子的后续测试也随之冻结。

[来源](packages/core/tests/e2e.test.ts#L51-L75)

### `deleteRecord` 存在，但清理不可靠

`BskyClient` 确实暴露了 `deleteRecord` 方法——它调用 `com.atproto.repo.deleteRecord`，并在生产代码中用于删除列表、关注等记录。

[来源](packages/core/src/at/client.ts#L510-L515)

问题出在 AT Protocol 的 **数据流架构** 上。当你调用 `createRecord`，数据写入你的 PDS（个人数据服务器）；但 Bluesky 的 **AppView**（中继索引器）随后会抓取并缓存这条记录。即使你成功调用了 `deleteRecord` 从 PDS 删除数据：

- AppView 的缓存可能不会立即清除。
- 搜索引擎（如 `searchPosts`）可能数分钟甚至数小时后仍返回已删除的帖子。
- 如果后续测试运行试图用相同关键词搜索，会污染断言。

因此测试文件中的注释直白地写：`AT Protocol doesn't support easy post deletion via API`。

[来源](packages/core/tests/e2e.test.ts#L33-L35)

### 三份测试文件中的注释模式

| 文件 | 被注释的测试 | 依赖关系 |
|------|-------------|---------|
| `packages/core/tests/e2e.test.ts` | 测试 2、3、4（共 3 个） | 测试 4 依赖测试 2 创建的 `createdPostUris[0]` |
| `packages/core/tests/feed.test.ts` | 帖子创建 + 图片上传 + 线程展平 + 上下文（共 6 个） | 每个后续测试依赖前一步的 URI/CID |
| `packages/core/tests/ai_integration.test.ts` | beforeAll 中的帖子创建 + AI 分析测试（共 2 个） | AI 测试依赖 `testPostUri` |

所有注释都遵循同一原则：**写操作示例被禁用，读操作测试保持激活**。这样 CI 流水线可以零副作用地运行，不会在 Bluesky 上留下垃圾数据。

## 如何激活并选择性运行测试

当你需要验证写操作逻辑时，可以局部取消注释并只运行相关测试。推荐两种方式：

### 方式一：`it.only` —— 最精确

找到目标测试，在 `it` 后添加 `.only`：

```ts
it.only('[2] Feed: post, read, search, thread flatten', async () => {
  // ...
}, 60000);
```

然后运行单个测试文件：

```bash
pnpm --filter @bsky/core exec vitest run --config vitest.config.ts tests/e2e.test.ts --reporter=verbose
```

Vitest 会跳过所有无 `.only` 的测试。注意同时取消 `createdPostUris` 数组声明和 `afterAll` 日志的注释——如果你还需要后续测试依赖它的话。

### 方式二：`pnpm --filter` + 文件名匹配

如果你只关心某个文件但不指定单个测试，可以靠 `test:e2e` 脚本自带的 verbose 模式：

```bash
pnpm --filter @bsky/core test:e2e
```

这个命令等价于：

```bash
vitest run --config vitest.config.ts tests/e2e.test.ts --reporter=verbose
```

[来源](packages/core/package.json#L17-L19)

### 必须同步取消的关联代码

解除单个 `it(...)` 块的注释不够——检查这些变量是否也在被注释的行中声明：

- `e2e.test.ts`：`createdPostUris`（第 25 行）和 `afterAll` 中的日志。
- `feed.test.ts`：`testPostUri`、`testPostRkey`、`testPostUris`、`uploadedBlobCid`、`uploadedBlobDid`。
- `ai_integration.test.ts`：`testPostUri`（第 28 行）和 `sharedTestPostUri`（第 23 行）。

一个遗漏的声明会导致运行时 `ReferenceError`。

## 调试技巧

### 1. `--reporter=verbose`：看清每个测试的生命周期

默认的 Vitest reporter 只显示通过/失败汇总。Verbose 模式逐条打印测试名称和执行时间：

```
 ✓ [1] Authentication: login, resolve handle, get profile (4822ms)
 ✓ [2] Feed: read timeline (3124ms)
 ✓ [5] AI: translation to Chinese (1432ms)
```

这在排查超时或挂起测试时尤其有用——你能精确知道哪个测试卡住了。

### 2. `console.log`：在真实 API 调用中植入侦察点

测试文件大量使用 `console.log` 输出中间结果。激活被注释的测试前，确保这些日志行也处于激活状态：

```ts
console.log(`Created test post: ${res.uri}`);
console.log(`Search found ${searchRes.posts.length} posts, hitsTotal: ${searchRes.hitsTotal}`);
```

[来源](packages/core/tests/feed.test.ts#L49)

输出会出现在 Vitest 的 stdout 中（在 verbose 模式下尤其清晰）。你可以在此基础上添加自己的日志，比如打印响应头或原始响应体，来诊断 API 错误。

### 3. 添加延迟等待索引

Bluesky 的 AppView 索引不是实时的。创建一个帖子后立即搜索，大概率搜不到。测试中已经预留了 `setTimeout` 模式：

```ts
await new Promise(r => setTimeout(r, 2000));
```

[来源](packages/core/tests/e2e.test.ts#L71)

如果测试环境网络较慢或 AppView 负载高，考虑把延迟增加到 **5000ms**。这个延迟被平衡地用在两个地方：创建帖子后等待搜索索引、上传图片后等待 blob 可下载。

### 4. 环境变量完整性检查

所有端到端测试依赖 `.env` 文件的三个变量：

```
BLUESKY_HANDLE=your.handle
BLUESKY_APP_PASSWORD=your-app-password
LLM_API_KEY=sk-...
```

[环境变量与配置](环境变量与配置.md) 有完整说明。如果某个测试无声跳过，检查 `beforeAll` 中的守卫条件——例如 `ai_integration.test.ts` 会检查 `LLM_API_KEY` 并调用 `it.skip`。

## 测试触发写操作的确认门

即使测试中创建帖子，也需要注意生产代码中的 **写操作确认门**（confirmation pipeline）。[36 个 AI 工具：从定义到执行](36-个-ai-工具-从定义到执行.md) 描述了 `create_post` 等写工具在 `AIAssistant` 调用时会触发确认流程。测试代码直接调用 `client.createRecord()` 绕过了这个门——因此不会触发确认提示，适合自动化。
