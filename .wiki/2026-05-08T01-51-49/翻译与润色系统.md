# 翻译与润色系统

翻译与润色系统是 AI 辅助内容创作的两大支柱功能。翻译子系统采用双模式架构，兼顾 CJK 与非 CJK 文本的不同需求；润色子系统则通过单轮提示词调用实现帖子草稿的快速改写。两者均暴露为无状态的纯函数，由 `@bsky/core` 导出，供 TUI 和 PWA 分别集成。

---

## 翻译引擎：双模式架构

核心函数 `translateText` 位于 `packages/core/src/ai/assistant.ts`，接受 `mode: 'simple' | 'json'` 参数，根据目标语言和文本特征选择输出格式。[来源](packages/core/src/ai/assistant.ts#L725-L730)

### Simple 模式

当目标语言属于 CJK 或文本较短时使用。提示词 `PF_TRANSLATE_SIMPLE` 要求模型"只返回翻译结果，不含解释"：

> `Translate the following text to {targetLang}. Keep the original meaning, output only the translation, no explanations.`

[来源](packages/core/src/ai/prompts.ts#L144-L146)

请求体为标准 `system + user` 消息格式，`temperature: 0.3`，禁用思考链。API 响应的 `choices[0].message.content` 直接作为翻译结果返回，不做 JSON 解析。[来源](packages/core/src/ai/assistant.ts#L741-L754)

`LANG_LABELS` 注册表将语言代码映射为自然语言标签（`zh → '中文'`、`en → 'English'`、`ja → '日本語'` 等），传入提示词模板。[来源](packages/core/src/ai/prompts.ts#L16-L24)

### JSON 模式

当目标语言为非 CJK（如法语、德语）或文本较长（≥200 字符）时使用。提示词 `PF_TRANSLATE_JSON` 要求模型输出结构化 JSON：

> `{"source_lang": "<ISO 639-1 code, use 'und' if unsure>", "translated": "<the translation>"}`

请求体额外附加 `response_format: { type: 'json_object' }`，强制 LLM 输出合法 JSON。[来源](packages/core/src/ai/assistant.ts#L752-L754)

响应经过两层校验：

1. `JSON.parse` 尝试解析
2. 验证 `parsed.translated` 字段非空

解析成功后返回 `{ translated, sourceLang }` 结构；解析失败或缺少 `translated` 字段则触发重试逻辑。若重试用尽仍失败，降级为将原始 content 作为 `translated`、`sourceLang: 'und'` 返回。[来源](packages/core/src/ai/assistant.ts#L782-L804)

**关键设计决策**：模式选择权不在 `translateText` 函数内部，而是由调用方传入。`docs/AI_SYSTEM.md` 描述了自动选择规则（CJK 或短文本 → simple，其余 → json），但代码中该逻辑由 UI 层负责——TUI 的 `UnifiedThreadView` 通过 `translateMode` prop 注入，ProfileView 则硬编码为 `simple`。[来源](docs/AI_SYSTEM.md#L95-L98)

### 重试逻辑与空内容缺陷

`translateText` 实现了指数退避重试机制，解决 DeepSeek JSON 模式偶发返回 `200 OK` 但 `content` 为空字符串的缺陷：

| 尝试次数 | 等待延迟 | 触发条件 |
|---------|---------|---------|
| 1（首试） | — | 直接请求 |
| 2 | 800 ms | 空 content / JSON 解析失败 / 网络错误 |
| 3 | 1,600 ms | 同上 |
| 4（末次） | 2,400 ms | 同上，失败则抛异常 |

[来源](packages/core/src/ai/assistant.ts#L739-L817)

重试逻辑覆盖三种失败场景：

- **空 content**（HTTP 200 但 `content.trim()` 为空）
- **JSON 解析失败**（仅 json 模式）
- **网络/服务错误**（`res.ok` 为 false 或 fetch 异常）

三种场景的重试路径不同：前两者在函数内部 `for` 循环中直接 `continue` 走同一次迭代的剩余逻辑；后者通过 `catch` 块捕获后 `continue`。[来源](packages/core/src/ai/assistant.ts#L765-L813)

### 向后兼容的包装函数

`translateToChinese` 是便捷封装，固定调用 `translateText(config, text, 'zh', 'simple')`，只返回 `result.translated` 字符串，保持与旧代码的兼容性。[来源](packages/core/src/ai/assistant.ts#L823-L826)

## 润色引擎：单轮提示词调用

`polishDraft` 是整个系统中结构最简单的 AI 函数之一——它基于 `singleTurnAI` 通用单轮调用封装而成。[来源](packages/core/src/ai/assistant.ts#L831-L840)

```typescript
export async function polishDraft(
  config: AIConfig,
  draft: string,
  requirement: string,
  modelOverride?: string,
): Promise<string> {
  return singleTurnAI(config, P_POLISH_SYSTEM, PF_POLISH_USER(requirement, draft), 0.7, 2000, modelOverride);
}
```

参数细节：

- **temperature: 0.7**——高于翻译（0.3），允许更多创造性改写
- **max_tokens: 2000**——足够容纳 Bluesky 帖子（300 字符限制）及扩展
- **无重试**——与翻译不同，润色不实现重试逻辑，失败直接抛异常

### 提示词设计

系统提示词 `P_POLISH_SYSTEM` 简洁明确：

> `你是一个文字润色助手，根据用户要求调整以下帖子草稿，只返回润色后的文本。`

[来源](packages/core/src/ai/prompts.ts#L168)

用户提示词模板 `PF_POLISH_USER` 将用户要求和草稿拼接：

> `用户要求：{requirement}\n\n草稿：\n{draft}`

[来源](packages/core/src/ai/prompts.ts#L171-L172)

**设计意图**：要求"只返回润色后的文本"意味着模型输出直接作为结果字符串返回，不需任何后处理。这与翻译的 JSON 模式形成对比——后者需要结构化解析。

### singleTurnAI 通用函数

作为 `polishDraft` 的底层基础设施，`singleTurnAI` 实现了标准的单轮 LLM 调用：

- 拼接 `system + user` 消息
- 支持 `modelOverride` 覆盖 `config.model`
- 根据 provider 类型决定是否发送 `thinking` 参数
- 解析响应为 `ChatCompletionResponse` 后返回 `content`
- 网络错误时给出明确提示（含 URL 检查和配置建议）

[来源](packages/core/src/ai/assistant.ts#L661-L707)

## UI 集成：两种交互模式

### PWA：PolishWidget + 模态弹窗

润色功能通过 **Widget 系统** 注册为 `polish` 组件。[来源](packages/pwa/src/App.tsx#L87-L93)

Widget 注册的属性：

- `views: ['compose']` ——仅在 compose 页面可见
- `defaultOpen: true` ——默认在右侧面板展开
- `titleKey: 'action.polish'` ——多语言标题（中文"润色"、日文"推敲"、英文"Polish"）

`PolishWidget` 组件的核心逻辑：

1. 通过 `widgetContext` 获取 `polishConfig`、`composeDraft` 和 `onComposeDraftChange` 回调
2. 用户输入润色要求（支持 Enter 快捷提交）
3. 调用 `polishDraft(config, draft, requirement)` 
4. 结果显示区提供 **Copy**（复制到剪贴板）和 **Replace**（替换原文）两个操作按钮

[来源](packages/pwa/src/components/widgets/PolishWidget.tsx#L13-L54)

在 `ComposePage` 中，Polsih 按钮有两个呈现途径：

- **大屏幕（`lg+`）**：通过右侧面板 Widget 直接交互
- **小屏幕**：点击顶栏润色按钮 → `showPolishModal` 状态 → 弹出 `WidgetModal`，将 `polishConfig`、当前帖子文本、替换回调注入 context

[来源](packages/pwa/src/components/ComposePage.tsx#L346-L354)

多帖子线程场景下，Polish 的目标帖子通过 `polishTargetPostId` 状态跟踪——默认选中第一个非空帖子，用户在帖子间 Tab 切换时自动更新焦点（通过 `onFocus` 事件设置 `setPolishTargetPostId(post.id)`）。[来源](packages/pwa/src/components/ComposePage.tsx#L87-L93)

### TUI：f 键驱动的键盘交互

TUI 的润色流程完全由键盘驱动，状态机包含四个阶段：

| 阶段 | 触发 | 界面表现 |
|------|------|---------|
| `idle` | 默认 | 正常 compose 模式 |
| `req`（输入要求） | `f` 键 | 紫色边框文本输入框，placeholder 显示润色要求 |
| `loading` | 回车提交 | 青色 `"润色中…"` 文字 |
| `result` | AI 返回 | 绿/红色边框结果框 + 操作提示 |

[来源](packages/tui/src/components/App.tsx#L113-L117)

键盘映射：

- `f` / `F`（compose 模式且有非空帖子）→ `setPolishPhase('req')` [来源](packages/tui/src/components/App.tsx#L471-L474)
- `Enter`（req 阶段）→ 调用 `handlePolishCall` [来源](packages/tui/src/components/App.tsx#L345-L349)
- `Esc`（req/result 阶段）→ 回到 idle [来源](packages/tui/src/components/App.tsx#L353)
- `r` / `R`（result 阶段）→ **Replace**：用润色结果替换当前帖子文本 [来源](packages/tui/src/components/App.tsx#L359-L363)
- `c` / `C`（result 阶段）→ **Copy**：将润色结果写入 stderr（终端回显但不修改帖子）[来源](packages/tui/src/components/App.tsx#L365-L371)

`ComposeView` 根据 `mode` 属性渲染四种不同的输入/结果显示区域：

- `mode === 'polishReq'` → 输入框，绑定 `polishRequirement` 状态
- `mode === 'polishResult'` → 结果框 + 加载/错误/正常三种子状态，底部显示快捷键提示

[来源](packages/tui/src/components/ComposeView.tsx#L137-L159)

`handlePolishCall` 使用 **动态 import** 按需加载 `@bsky/core` 的 `polishDraft`，并通过 `resolveScenarioConfig(config.scenarioModels.polish)` 获取针对润色场景独立配置的模型参数（用户可在设置中为 AI Chat / 翻译 / 润色分别指定不同模型）。[来源](packages/tui/src/components/App.tsx#L199-L214)

## 场景化模型配置

翻译和润色都支持通过 `scenarioModels` 独立配置模型。在 TUI 中，`SetupWizard` 和 `SettingsView` 均提供三个场景的模型选择入口（`aiChat` / `translate` / `polish`），[来源](packages/tui/src/components/SetupWizard.tsx#L55) PWA 的 `SettingsModal` 也同样提供三个场景的配置。[来源](packages/pwa/src/components/SettingsModal.tsx#L316-L317)

这意味着用户可以给翻译指定一个快速便宜的模型，给润色指定一个创造力强的模型——两个函数都通过 `modelOverride` 参数或内部 `config.model` 消费此配置。

## 与 AI 助手引擎的关系

翻译和润色是 AI 引擎的子集，但与主对话引擎 `AIAssistant` 有本质区别：

| 维度 | `translateText` / `polishDraft` | `AIAssistant.sendMessage` |
|------|-------------------------------|--------------------------|
| 记忆 | 无状态单轮 | 多轮对话历史 |
| 工具 | 无工具调用 | 31 个 AT Protocol 工具 |
| 流式 | 不支持 | 支持 SSE 流式输出 |
| 重试 | 指数退避（翻译）| 无重试 |
| 写操作确认 | 不适用 | 四步确认门 |

[AI 助手引擎](ai-助手引擎.md) 文档详细说明了后者的完整架构。
