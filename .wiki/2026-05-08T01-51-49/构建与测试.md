# 构建与测试

## 概览

本项目的开发工作流围绕三个核心环节展开：**类型检查**（TypeScript 编译时验证）、**构建**（跨包编译）、**测试**（真实 API 集成测试）。所有操作通过 `pnpm` 的工作区命令统一编排，根 `package.json` 中的脚本使用 `-r`（recursive）标志将命令广播到所有子包。[来源](package.json#L6-L12)

## 常用命令

### 全仓库类型检查

```bash
pnpm -r typecheck
```

执行所有包的 `tsc --noEmit`，零产出的纯类型验证。`@bsky/core` 包的 `typecheck` 脚本即 `tsc --noEmit`。[来源](packages/core/package.json#L20)

### 构建所有包

```bash
pnpm -r build
```

每个包的 `build` 脚本独立运行。`@bsky/core` 使用 `tsc`（TypeScript 编译器输出到 `dist/`），输出物由 `package.json` 的 `exports` 字段控制。[来源](packages/core/package.json#L15)

### 运行全部测试

```bash
pnpm test
```

等价于 `pnpm -r test`，广播到各包。`@bsky/core` 的测试脚本为 `vitest run --config vitest.config.ts`，以 **run-once** 模式执行。[来源](packages/core/package.json#L17)

当前测试套件按文档记录为 **29 个用例**（`auth` 3 + `feed` 8 + `ai_integration` 8 + `e2e` 10），代码实际已扩展至更多，详情见下方覆盖范围表。[来源](docs/TESTING.md#L25-L32)

### 单独运行 E2E 套件

```bash
pnpm --filter @bsky/core test tests/e2e.test.ts
```

或使用预置别名：

```bash
pnpm --filter @bsky/core test:e2e
```

`test:e2e` 脚本额外指定了 `--reporter=verbose`，输出详细测试名称和耗时，便于追踪长流程中的每一步。[来源](packages/core/package.json#L19)

### 其他常用命令

| 命令 | 用途 |
|------|------|
| `pnpm --filter @bsky/core test:watch` | 监听模式，文件变更自动重跑 |
| `pnpm --filter @bsky/core test tests/auth.test.ts` | 仅运行认证测试 |
| `pnpm --filter @bsky/core test tests/ai_integration.test.ts` | 仅运行 AI 集成测试 |
| `pnpm -r lint` | 全仓库 lint 检查 |

## 测试框架与配置

**运行器**：Vitest 3.x，配置文件位于 `packages/core/vitest.config.ts`。

```typescript
// packages/core/vitest.config.ts
export default defineConfig({
  test: {
    globals: true,        // 全局 describe/it/expect，无需 import
    testTimeout: 60000,   // 单测试超时 60s（默认 30s 对真实 API 不够）
    hookTimeout: 30000,   // beforeAll/afterAll 超时
  },
});
```

[来源](packages/core/vitest.config.ts#L1-L9)

核心决策：**所有测试均为集成测试，无 Mock**。每个测试用例直接使用真实 Bluesky API 和 DeepSeek API（或其他 LLM 提供商）。这意味着测试可能受到网络延迟、Bluesky 服务可用性、API 速率限制等因素影响。[来源](docs/TESTING.md#L7-L7)

## 环境凭证

测试从项目根目录加载 `.env` 文件：

```typescript
dotenv.config({ path: path.resolve(__dirname, '..', '..', '..', '.env') });
```

三个必需变量：

| 变量 | 用途 |
|------|------|
| `BLUESKY_HANDLE` | Bluesky 账号 handle |
| `BLUESKY_APP_PASSWORD` | Bluesky 应用密码（非主密码） |
| `LLM_API_KEY` | DeepSeek（或其他兼容 OpenAI 的提供商）API 密钥 |

[来源](docs/TESTING.md#L68-L73)

配置的完整说明请参阅 [环境变量与配置](环境变量与配置.md)。

## 测试模式与常见问题

### 三个典型模式

**API 直连测试**：直接实例化 `BskyClient`，调用 `login` 后操作 Bluesky API。

```typescript
const client = new BskyClient();
await client.login(HANDLE, APP_PASSWORD);
const posts = await client.getTimeline(20);
expect(posts.feed.length).toBeGreaterThan(0);
```

[来源](docs/TESTING.md#L37-L42)

**AI 工具调用测试**：通过 `createTools(client)` 注入真实客户端，`AIAssistant` 执行工具调用循环。

```typescript
const tools = createTools(client);
const assistant = new AIAssistant(AI_CONFIG);
assistant.setTools(tools);
assistant.addSystemMessage('你是一个 Bluesky 助手。');
const result = await assistant.sendMessage('分析帖子 at://...');
expect(result.toolCallsExecuted).toBeGreaterThanOrEqual(1);
```

[来源](docs/TESTING.md#L44-L52)

**E2E 完整流程**：按顺序覆盖认证→时间线→翻译→润色→资料→通知→引导问题，每个步骤依赖前一步的状态，模拟用户真实操作链路。[来源](docs/TESTING.md#L54-L64)

### 常见问题与修复

| 问题 | 原因 | 处理方式 |
|------|------|---------|
| 测试超时（默认 30s） | 真实 API 调用慢 | vitest 配置全局 60s 超时；关键测试使用第 3 参数 `it('...', fn, 60000)` |
| Blob 下载超时 | 图片下载需要鉴权 | 添加 2s 延迟；使用认证后端点；30s 超时 |
| 搜索返回空 | Bluesky 索引延迟 | 搜索前等待 3s 确保索引 |
| AI 响应截断 | 流式模式不稳定 | 强制非流式模式；max_tokens=4096 |
| 测试数据残留 | AT Protocol 不支持简单删除 | 测试帖子使用 `[TRST 测试]` 标记便于识别 |

[来源](docs/TESTING.md#L75-L83)

## 测试数据说明

项目不提供 Mock 或 Sandbox 环境。测试对 Bluesky 的写操作（发帖、建列表等）会真实写入目标账号。代码中的写操作测试大多已被注释，仅在 `list.test.ts` 中保留了列表创建/更新的完整流程（并在 `afterAll` 中尝试清理）。[来源](packages/core/tests/list.test.ts#L122-L129)

如果需要新增测试，请遵循以下原则：在 `packages/core/tests/` 下创建 `.test.ts` 文件，从 `../src/` 导入 `BskyClient`、`createTools`、`AIAssistant` 等核心模块，使用 `.env` 中的真实凭证，不引入 Mock。[来源](docs/TESTING.md#L87-L91)
