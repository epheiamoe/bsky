# 导航状态机

## AppView：17 种视图的联合类型

导航系统的核心是 **`AppView` 联合类型**，它定义了整个应用所有可能屏幕的"形状"。每个视图由 `type` 字面量区分，携带该视图所需的参数载荷：

| type | 载荷字段 | 语义 |
|---|---|---|
| `feed` | `feedUri?: string` | 时间线 feed，缺省时使用默认 feed |
| `thread` | `uri: string` | 单个帖子的线程视图 |
| `detail` | `uri: string` | 通用详情页 |
| `compose` | `replyTo?: string; quoteUri?: string; draftId?: string` | 发帖编辑器，可选回复/引用/草稿 |
| `profile` | `actor: string; profileTab?: string` | 用户主页，含可选 tab |
| `notifications` | 无 | 通知列表 |
| `search` | `query?: string; searchTab?: string` | 搜索，可选预填查询和 tab |
| `aiChat` | `contextUri?: string; sessionId?: string; contextPost?: string; contextProfile?: string` | AI 对话，可附带上下文 |
| `bookmarks` | 无 | 书签列表 |
| `drafts` | 无 | 草稿列表 |
| `components` | 无 | 组件展示页 |
| `dm` | 无 | DM 会话列表 |
| `dmChat` | `conversationId: string` | 单条 DM 对话 |
| `lists` | `actor?: string` | 列表列表，可选限定用户 |
| `listDetail` | `uri: string; tab?: 'posts' \| 'members'` | 单个列表详情 |
| `about` | 无 | 关于页面 |

这种设计将**视图路由**与**视图参数**编码进类型系统，编译器保证路由时不会遗漏必填参数，消费方通过 `currentView.type` 判别分支后安全解包。

## 栈式导航：纯函数实现

导航引擎由 `createNavigation()` 工厂函数生成，维护一个不可变栈（`AppView[]`），栈顶即 `currentView`。

```
┌─────────────┐     goTo(thread)    ┌─────────────┐     goTo(compose)   ┌─────────────┐
│  [feed]      │ ─────────────────→ │  [feed,       │ ────────────────→ │  [feed,      │
│              │                    │   thread]     │                   │   thread,    │
└─────────────┘                    └─────────────┘                   │   compose]   │
                                                                    └─────────────┘
      ←──────────────── goBack() ──────────────── ←──────────────── goBack() ──────
```

三个导航操作的实现都是**纯函数式不可变更新**：

- **`goTo(view)`**：`stack = [...stack, view]`，入栈新视图。
- **`goBack()`**：`stack = stack.slice(0, -1)`，出栈（`length > 1` 守卫）。
- **`goHome()`**：`stack = [{ type: 'feed' }]`，清空栈重置到 feed。

每次操作后调用 `notify()` 遍历 `listeners` 数组通知订阅者。`subscribe(fn)` 返回取消订阅函数，适合 React `useEffect` 的 clean-up。

[来源](packages/app/src/state/navigation.ts#L28-L72)

## useNavigation：React 订阅封装

`useNavigation` hook 将纯状态机嵌入 React 生命周期：

```typescript
const [nav] = useState(() => createNavigation());     // 惰性初始化，仅一次
const [state, setState] = useState(() => nav.getState());
const tick = useCallback(() => setState(nav.getState()), [nav]);
useEffect(() => nav.subscribe(tick), [nav, tick]);
```

关键设计：**`createNavigation()` 放进 `useState(() => ...)` 而非 `useRef`**，保证多次 re-render 不会重建状态机；但注意 `useState` 的 init 函数仅在 mount 时执行一次，这正是预期的单例生命周期。返回的接口剥去了 `subscribe`/`getState` 等内部方法，只暴露 `currentView`、`canGoBack`、`goTo`、`goBack`、`goHome` 给组件消费。

[来源](packages/app/src/hooks/useNavigation.ts#L5-L20)

## 双界面差异：TUI 直接状态 vs PWA Hash 路由

### TUI：直接订阅，无 URL

TUI 在 `packages/tui/src/components/App.tsx` 第 56 行直接调用 `useNavigation()`，将返回值解构到组件顶层。视图切换触发 React 重渲染，Ink 框架重新绘制终端屏幕。无 URL 参与——栈完全在内存中。

```typescript
const { currentView, canGoBack, goBack, goTo, goHome } = useNavigation();
```

键盘快捷键（`n`→通知、`p`→个人资料、`s`→搜索、`a`→AI、`c`→发帖、`b`→书签等）逐一调用 `goTo()`，不再赘述。Escape 键统一调用 `goBack()` 回退。

[来源](packages/tui/src/components/App.tsx#L56-L56)

### PWA：Hash 路由，URL 即状态

PWA 用 `useHashRouter` 替代 `useNavigation`，接口完全兼容（`currentView`/`goTo`/`goBack`/`goHome`），但实现截然不同：

- **状态来源**：`parseHash()` 解析 `window.location.hash` 得出 `AppView`。
- **导航操作**：`goTo(view)` → `encodeView(view)` 生成 hash 字符串 → `history.pushState` 写入 URL。
- **后退**：`goBack()` → `window.history.back()`，依赖浏览器原生栈。
- **监听**：`popstate` 事件驱动视图更新。

**PWA 路由格式一览**（来自 `parseHash`/`encodeView` 的双向映射）：

| AppView | Hash 格式 |
|---|---|
| `feed` | `#/feed` / `#/feed?feed=at://...` / `#/feed?feed=following` |
| `thread` | `#/thread?uri=at://...` |
| `profile` | `#/profile?actor=did:plc:...&tab=posts` |
| `notifications` | `#/notifications` |
| `search` | `#/search?q=...&tab=top` |
| `bookmarks` | `#/bookmarks` |
| `drafts` | `#/drafts` |
| `compose` | `#/compose?replyTo=at://...&quoteUri=at://...&draftId=...` |
| `aiChat` | `#/ai?session=uuid&post=at://...&profile=did:...` / `#/ai?context=at://...` |
| `dm` | `#/dm` / `#/dm?conv=did:plc:...` |
| `lists` | `#/lists?actor=did:...` |
| `listDetail` | `#/list?uri=at://...&tab=posts` |
| `components` | `#/components` |
| `about` | `#/about` |

**关键差异**：PWA 的 `canGoBack` 不是由栈深度决定，而是由 `window.location.hash !== '#/feed'` 判定。首次加载 `/` 或 `/feed` 时会重定向到默认 feed（通过 `getFeedConfig().defaultFeedUri ?? BUILTIN_FEEDS.following`）。

[来源](packages/pwa/src/hooks/useHashRouter.ts#L21-L239)

### 对比总结

| 维度 | TUI | PWA |
|---|---|---|
| 状态容器 | 内存 `AppView[]` | `window.location.hash` |
| 后退机制 | `stack.slice(0, -1)` | `window.history.back()` |
| 可分享 URL | ❌ | ✅（完整 hash 路由） |
| 前进操作 | `push` 入栈 | `history.pushState` |
| 初始化 | `useState(() => createNavigation())` | `parseHash()` 解析当前 URL |
| 生命周期 | React + Ink | React + popstate 事件 |

## 视图切换时的副作用

导航不仅是状态变化，还触发数据层面的副作用。TUI 中通过 `useEffect` 监听 `currentView.type`：

- 进入 `bookmarks` → 调用 `bookmarks.refresh()` 刷新书签列表。
- 进入 `compose` → 重置媒体、草稿列表、润色状态，并根据 `replyTo`/`quoteUri`/`draftId` 载入编辑器上下文。
- `feedUri` 变化 → 通过 `setLastFeedUri()` 记录最近使用的 feed，供 PWA 和 TUI 共享。

[来源](packages/tui/src/components/App.tsx#L165-L192)

## 架构启示

导航系统的分层体现了 @bsky/core → @bsky/app → TUI/PWA 的三层架构（详见[包架构深度解析](包架构深度解析.md)）：

1. **纯状态机**（`createNavigation`）零 UI 依赖，可独立测试。
2. **React 桥接**（`useNavigation`）注入订阅机制，厚度仅 20 行。
3. **界面适配**（TUI 直接消费 vs PWA hash 路由重实现）共享同一 `AppView` 类型，确保跨界面兼容。

这种模式下，**`AppView` 成为整个应用的契约接口**——任何新视图只需在联合类型中追加一个变体，更新 `encodeView`/`parseHash` 双向映射，无需改动路由基础设施。
