# 31 个 AI 工具系统

## 三层接口设计

整个工具系统建立在三个紧密耦合的接口之上，每一层解决一个不同的问题域：

**`ToolDefinition`** — 工具的身份与契约。它声明了工具的名称（`name`）、自然语言描述（`description`）和 JSON Schema 输入格式（`inputSchema`）。这个结构直接映射到 OpenAI 兼容 API 的 `tools` 参数中每个 `function` 对象的定义，供 LLM 理解何时调用哪个工具以及传递什么参数。[来源](packages/core/src/ai/tools.ts#L15-L23)

**`ToolHandler`** — 工具的执行逻辑。签名 `(params: Record<string, unknown>, assistant?: unknown) => Promise<string>` 接受一个扁平参数对象和一个可选的 `assistant` 引用（用于访问 AIAssistant 实例的方法，如 `getUserUpload`、`addPendingImage`）。返回值永远是 JSON 字符串——无论是成功数据还是错误信息，都序列化为字符串传递给 LLM。[来源](packages/core/src/ai/tools.ts#L25)

**`ToolDescriptor`** — 前两者的聚合容器。它将 `definition` 和 `handler` 绑定在一起，并添加一个布尔标记 `requiresWrite`。这个标记是**写操作确认门**（Write Confirmation Gate）的核心开关：当 AIAssistant 的 `sendMessage` 循环检测到即将执行的工具 `requiresWrite === true` 时，它会在执行 handler 前暂停，等待用户通过 `confirmAction(boolean)` 显式批准。[来源](packages/core/src/ai/tools.ts#L27-L31)

这三层的职责分离清晰：`ToolDefinition` 面向 LLM，`ToolHandler` 面向后端实现，`ToolDescriptor` 则是中间编排层（AIAssistant）使用的统一包装。AIAssistant 通过 `setTools()` 接收 `ToolDescriptor[]`，构建 `Map<string, ToolDescriptor>` 以实现 O(1) 查找，并在 `sendMessage` 的工具调用循环中逐个分发。[来源](packages/core/src/ai/assistant.ts#L229-L273)

```
┌─────────────────────────────────────────────┐
│              AIAssistant                     │
│  sendMessage() ──►  tool_call loop          │
│                      │                      │
│                ┌─────┴─────┐                │
│                │ ToolMap   │                │
│                │ (name→    │                │
│                │  Descriptor)│               │
│                └─────┬─────┘                │
│                      │                      │
│          requiresWrite? ──► confirm gate    │
│                      │                      │
│                 handler(params, this)       │
└─────────────────────────────────────────────┘
                         │
                         ▼
              ┌──────────────────────┐
              │     BskyClient       │
              │  (AT Protocol HTTP)  │
              └──────────────────────┘
```

## 工具全景

当前代码中实际注册了 **36 个工具**（29 个读工具 + 7 个写工具），原始设计为 31 个（27 + 4），后经多次迭代扩展了列表管理能力。以下按功能域分组列举。

### 读工具（29 个，无需确认）

#### 身份与解析

| 工具 | 功能 | 核心依赖 |
|------|------|----------|
| `resolve_handle` | handle → DID 解析 | `client.resolveHandle()` |
| `get_profile` | 获取用户完整资料（handle、displayName、描述、计数） | `client.getProfile()` |
| `get_record` | 按 AT URI 获取原始记录 | `client.getRecord()` + `parseAtUri()` |
| `list_records` | 列举仓库集合中的记录（支持分页） | `client.listRecords()` |

`resolve_handle` 使用公共 API 端点（无需认证），`get_profile` 则根据是否已登录自动选择认证/公共端点。[来源](packages/core/src/ai/tools.ts#L62-L112)

#### 帖子与时间线

| 工具 | 功能 |
|------|------|
| `search_posts` | 全文搜索帖子（支持 top/latest 排序） |
| `get_timeline` | 获取已认证用户的主页时间线 |
| `get_author_feed` | 获取指定用户发布的帖子流 |
| `get_feed` | 获取指定 feed generator 的帖子 |
| `get_feed_generator` | 获取 feed generator 的详细信息 |
| `get_popular_feed_generators` | 获取热门 feed generator 列表 |

这些工具的 handler 在返回前都会对数据进行剪裁，提取 `uri`、`author`、`text`（截断至 200 字符）、`likeCount`、`repostCount` 等关键字段，避免将完整的 API 响应原始抛给 LLM（API 响应体通常包含大量无关元数据）。[来源](packages/core/src/ai/tools.ts#L127-L246)

#### 社交图谱

| 工具 | 功能 |
|------|------|
| `get_follows` | 获取用户关注列表 |
| `get_followers` | 获取用户粉丝列表 |
| `get_suggested_follows` | 获取推荐关注 |
| `search_actors` | 搜索用户（返回 DID、handle、displayName、描述） |

`get_follows` 和 `get_followers` 返回的列表包含 `cursor` 字段支持翻页，但工具自身不提供翻页参数——这是有意设计的简化，LLM 若需翻页需自行在后续对话中调用。[来源](packages/core/src/ai/tools.ts#L451-L493)

#### 互动数据

| 工具 | 功能 |
|------|------|
| `get_likes` | 获取点赞用户列表 |
| `get_reposted_by` | 获取转发用户列表 |
| `get_quotes` | 搜索引用某帖子的帖子列表 |

`get_reposted_by` 的返回做了极致的精简：只返回 `handle` 字符串数组，不包含 DID 或 profile 信息。[来源](packages/core/src/ai/tools.ts#L377-L381)

#### 帖子上下文与媒体

| 工具 | 功能 |
|------|------|
| `get_post_thread` | 获取帖子线程的原始树结构 |
| `get_post_thread_flat` | **获取扁平化、人类可读的帖子线程（推荐）** |
| `get_post_subtree` | 展开被折叠的回复子树 |
| `get_post_context` | 获取帖子完整上下文：父链、引用内容、媒体摘要 |
| `extract_images_from_post` | 提取帖子中的图片 blob 引用（did + cid） |
| `download_image` | 将图片下载到用户本地 Downloads 文件夹 |
| `view_image` | 供视觉模型查看/分析图片（支持两种来源） |
| `extract_external_link` | 提取帖子中的外部链接 embed |
| `fetch_web_markdown` | 通过 r.jina.ai 代理将网页转为 Markdown |

`view_image` 是设计最精巧的读工具之一。它接受两种图片来源：
- **`did` + `cid`**：来自 Bluesky 帖子的图片（通过 `extract_images_from_post` 获取）
- **`uploadIndex`**：用户在聊天中上传的图片（通过 `AIAssistant.getUserUpload()` 获取）

在执行时，它将图片转换为 base64 data URL，通过 `assistant.addPendingImage()` 存入 AIAssistant 的待处理图片队列，随后在下一次 API 调用时作为 `image_url` 块注入用户消息。如果视觉模式未开启，handler 会返回提示信息而非实际图片数据。[来源](packages/core/src/ai/tools.ts#L631-L663)

`download_image` 实现了跨平台降级：先尝试 Node.js 的 `fs.writeFileSync` 写入用户 Downloads 目录，若失败（如在 PWA 浏览器环境），则回退到返回 base64 data URL。[来源](packages/core/src/ai/tools.ts#L582-L613)

#### 其他

| 工具 | 功能 |
|------|------|
| `list_notifications` | 获取已认证用户的通知列表 |
| `get_lists` | 获取用户创建的所有列表（策展/ moderation） |
| `get_list_feed` | 获取列表中成员的最近帖子 |

`get_lists` 返回的 `purpose` 字段会从 `app.bsky.graph.defs#modlist` / `#curatelist` 映射为 `'moderation'` / `'curated'` 的人类可读形式。[来源](packages/core/src/ai/tools.ts#L900-L903)

### 写工具（7 个，均需用户确认）

所有写工具的 `requiresWrite` 均为 `true`，AIAssistant 在调用前会暂停并弹出确认对话框。

| 工具 | 功能 | 核心 AT Protocol 操作 |
|------|------|-----------------------|
| `create_post` | 创建帖子/回复/引用帖子（支持图片附件） | `com.atproto.repo.createRecord` |
| `like` | 点赞帖子 | `com.atproto.repo.createRecord`（`app.bsky.feed.like`） |
| `repost` | 转发帖子 | `com.atproto.repo.createRecord`（`app.bsky.feed.repost`） |
| `follow` | 关注用户 | `com.atproto.repo.createRecord`（`app.bsky.graph.follow`） |
| `create_list` | 创建新列表（策展/ moderation） | `client.createList()` |
| `add_to_list` | 将用户添加到列表 | `client.addListItem()` |
| `remove_from_list` | 从列表中移除用户 | `client.listRecords()` + `client.removeListItem()` |

`create_post` 的 handler 是最复杂的写工具，涉及三个子流程：

1. **回复链解析**：如果参数包含 `replyTo`，需要解析目标帖子的父链，遍历 `thread.parent` 直到找到根帖子，构造正确的 `reply.root` 和 `reply.parent` 对象。[来源](packages/core/src/ai/tools.ts#L747-L772)
2. **图片上传管线**：图片可来自 Bluesky CDN（`did` + `cid`）或用户的聊天上传（`pendingImageIndex`）。无论哪种来源，handler 都先下载图片数据，然后调用 `client.uploadBlob()` 上传到用户自己的 PDS，最后构造 `app.bsky.embed.images` 或 `app.bsky.embed.recordWithMedia` embed。[来源](packages/core/src/ai/tools.ts#L776-L820)
3. **引用帖子**：当同时有图片和引用时，使用 `recordWithMedia` 包裹，否则使用纯 `record` embed。[来源](packages/core/src/ai/tools.ts#L804-L819)

`remove_from_list` 的 handler 与其他写工具有显著不同——需要先通过 `listRecords` 查询所有列表项，找到匹配的 `subject` + `list` 条目，再逐一删除。这是为了避免 Bluesky API 没有直接的"按对象移除"端点而必须手动遍历的变通方案。[来源](packages/core/src/ai/tools.ts#L996-L1008)

## get_post_thread_flat：树→扁平格式

`get_post_thread_flat` 是 LLM 理解对话线程的首选工具。它的 description 明确写着 *"Prefer this over get_post_thread for understanding conversations"*，因为它解决了树结构对 LLM 的认知负担问题。

### 内部实现

`flattenThread()` 函数（也用于 `get_post_subtree` 和 `get_post_context`）的工作流程：

```
flattenThread(client, uri, maxDepth, maxReplies)
  │
  ├─ 1. 调用 client.getPostThread(uri, depth+1, 80) 获取原始树
  │
  ├─ 2. 遍历 parent 链，构建祖先列表（负深度）
  │     depth:-3 → depth:-2 → depth:-1 → depth:0 (根)
  │
  ├─ 3. 递归遍历 replies 数组，构建子回复
  │     depth:1 → depth:2 → ... → maxDepth
  │
  └─ 4. 每行通过 formatPostLine() 格式化
```

[来源](packages/core/src/ai/tools.ts#L1018-L1056)

### 行格式规范

每个帖子输出的格式为：

```
{prefix}depth:{n} | {handle} ({displayName}) (post:{rkey})
text: {text}{mediaInfo}
```

**depth 层级**：
- 祖先帖子使用**负深度**（`depth:-3`、`depth:-2`、`depth:-1`），数字越小表示离当前根帖子越远
- 根帖子为 `depth:0`
- 子回复从 `depth:1` 开始递增

**prefix 前缀**：
- 祖先链：` ↳ > ` 表示指向关系
- 回复链：`  ↳` 缩进加分支符号，层级越深缩进越多（每层两个空格）
- 回复箭头：` → (post:{parentRkey})` 显示当前回复指向的父帖子的 rkey

**mediaInfo 标记**：

| embed 类型 | 输出标记 |
|-----------|---------|
| `app.bsky.embed.images` | `[图片: N 张]` |
| `app.bsky.embed.external` | `[链接: hostname]` |
| `app.bsky.embed.record` | `[引用: rkey]` |
| `app.bsky.embed.recordWithMedia` | `[引用+媒体]` |
| `app.bsky.embed.video` | `[视频]` |

[来源](packages/core/src/ai/tools.ts#L1108-L1130)

### 折叠机制

为避免线程过长导致 LLM 上下文溢出或注意力分散，`buildReplyLines()` 实现了两层折叠：

1. **每层最大 sibling 数**：默认 `maxReplies=5`，超过的部分显示折叠提示
2. **最大深度**：默认 `maxDepth=3`，更深层的回复不展平

折叠提示的格式为：

```
  （还有 N 条回复被折叠，可调用 get_post_subtree 展开）
```

LLM 如需展开，可调用 `get_post_subtree`（本质上是同一套展平逻辑，只是从指定帖子开始而非从根开始），传入更大的 `depth` 或 `maxReplies`。[来源](packages/core/src/ai/tools.ts#L1095-L1098)

### 时间排序

同一深度的回复按 `indexedAt` 升序排列（最早的在前），确保 LLM 看到的对话顺序与人类阅读的 chrono 顺序一致。[来源](packages/core/src/ai/tools.ts#L1069-L1073)

## createTools：桥接模式

`createTools(client: BskyClient): ToolDescriptor[]` 是一个工厂函数，它的核心任务是**将 BskyClient 的领域方法适配为 LLM 可消费的工具 handler**。这是典型的**对象适配器模式**：[来源](packages/core/src/ai/tools.ts#L57)

```
BskyClient 方法（领域对象）  ──►  ToolHandler（LLM 接口）
```

适配过程中发生的转换：

| 维度 | BskyClient 侧 | ToolHandler 侧 |
|------|--------------|----------------|
| 参数风格 | 类型化参数（`string`, `number`, 可选参数） | 统一 `Record<string, unknown>`，由 LLM 的 JSON 解析 |
| 返回值 | 完整的 AT Protocol 响应（包含大量元数据） | 精简的 JSON 字符串（只保留 LLM 需要的字段） |
| 错误处理 | 抛出异常 | 捕获异常 → JSON 字符串错误信息 |
| 认证需求 | 部分方法需已登录，部分使用公共 API | handler 内部无认证逻辑，全由 client 封装 |

例如 `search_posts` 的 handler 将原始响应中复杂的 `PostView` 对象映射为仅含 `uri`、`author`、`text`、`likeCount`、`repostCount`、`indexedAt` 的扁平对象，并将 `hitsTotal` 一并返回供 LLM 判断结果数量。[来源](packages/core/src/ai/tools.ts#L133-L141)

`create_post` 的 handler 更进一步，在 BskyClient 方法调用前插入了多层业务逻辑：回复链遍历、图片下载→上传管线、embed 类型决策（纯图片/纯引用/recordWithMedia 混合）。这些逻辑不属于 BskyClient（它只负责 HTTP 通信），也不属于 AIAssistant（它只负责编排），而是作为适配器的一部分嵌入 handler 闭包中。[来源](packages/core/src/ai/tools.ts#L741-L822)

`view_image` 的 handler 则利用了 `assistant` 参数的桥接能力——它通过类型断言访问 AIAssistant 的 `getUserUpload()` 和 `addPendingImage()` 方法，实现了工具 handler 与 AIAssistant 内部状态的间接通信。[来源](packages/core/src/ai/tools.ts#L636-L653)

## 扩展性设计

新增一个工具只需三步：
1. 在 `createTools()` 的 `tools` 数组中新增一个 `ToolDescriptor`
2. 定义 `ToolDefinition` 的 `inputSchema`（供 LLM 理解参数）
3. 编写 handler 闭包（可访问 `client` 和 `assistant`）

如果 BskyClient 尚不支持所需 AT Protocol 端点，需先扩展 BskyClient 类（参见 [BskyClient 深度解析](bskyclient-深度解析.md)），然后再创建适配 handler。工具系统本身没有注册表或扫描机制，所有工具集中在 `tools.ts` 的 `createTools` 工厂函数内显式声明——这是一种"显式优于隐式"的设计哲学，牺牲了自动发现能力，换取了工具清单的可审计性和类型安全性。
