# 系统提示词体系

整个项目的 AI 提示词采用**集中式分片架构**：所有面向 LLM 的文本全部定义在 `packages/core/src/ai/prompts.ts` 中，通过 `useAIChat` hook 的 `buildSystemPrompt` 函数动态拼装为最终的系统提示词。设计目标有三：可审计（所有 AI 指令集中一处）、可组合（按需拼接片段）、可定制（修改该文件即可调整行为）。

[来源](packages/core/src/ai/prompts.ts#L1-L10)

---

## 命名约定

文件顶部明确规定了两种前缀，一眼可辨数据类型：

| 前缀 | 类型 | 示例 |
|------|------|------|
| `P_` | 字符串常量或模板 | `P_ASSISTANT_BASE`、`P_CONCISE` |
| `PF_` | 函数，返回字符串 | `PF_CURRENT_USER(name, handle)`、`PF_ENVIRONMENT(env)` |

翻译和润色相关的提示词也遵循此约定：`PF_TRANSLATE_SIMPLE`、`PF_TRANSLATE_JSON`、`P_POLISH_SYSTEM`、`PF_POLISH_USER`。

[来源](packages/core/src/ai/prompts.ts#L7-L10)

## 核心分片详解

### P_ASSISTANT_BASE：角色定义 + 图片规则 + 安全红线

这是整个提示词体系的锚点，定义了 AI 的**身份**、**能力边界**和**不可逾越的安全规则**。内容分四块：

**① 角色宣言**  
"你是用户的 Bluesky 助手，帮助用户浏览和分析 Bluesky 上的内容。"——定位为分析型助手，而非生成型或社交型代理。

**② 高级搜索语法提示**  
告知 AI 在调用 `search_posts` 时可以使用 Lucene 风格语法：`from:`、`to:`、`mentions:`、`since:`、`until:`、`lang:`、`has:image` 以及精确短语。这是为了提升 AI 检索效率，减少无效调用。

**③ 图片与视频处理规则**  
- **用户上传的图片**：通过 `pendingImageIndex` 参数引用（索引从 0 开始），允许在 `create_post` 中复用。
- **帖子中的视频**：AI 无法分析视频内容，禁止对视频帖子调用 `extract_images_from_post` 或 `view_image`，只能告知用户"帖子包含视频，暂时无法分析"。

**④ 3 条硬安全规则（写操作禁令）**

```
1. 绝对不要主动代表用户发帖、回复、点赞、转发或关注任何人。
   所有写操作（create_post、like、repost、follow）必须由用户明确要求后才执行。
   即使用户让你"查看某人的资料"，你只需要概括和分析，不要自动生成推文或互动。
2. 汇总资料时直接输出分析结果，不要附加"我帮你发条帖子吧"之类的建议。
3. 如果用户要求你发帖，你才通过 create_post 工具执行，否则永远不要。
```

这三条规则在代码层面有两层保障：提示词中的**硬约束**（上述文本）结合 `AIAssistant` 中的**写确认门**（`requiresWrite` 标志触发用户确认弹窗）。提示词约束是防止 AI"好心办坏事"的第一道防线；[确认门](ai-助手引擎.md)是第二道防线，拦截在工具执行前。

[来源](packages/core/src/ai/prompts.ts#L31-L58) | [写确认门实现](packages/core/src/ai/assistant.ts#L247-L268)

### PF_CURRENT_USER / PF_PROFILE_CONTEXT / PF_POST_CONTEXT：动态上下文注入

这三个函数负责将**当前用户界面上下文**注入系统提示词，让 AI 知道"用户当前在看什么"。

| 函数 | 注入时机 | 输出示例 |
|------|----------|----------|
| `PF_CURRENT_USER(name, handle?)` | 任何对话 | `当前用户: 张三 (@zhangsan.bsky.social)。` |
| `PF_PROFILE_CONTEXT(handle, currentUserHandle?)` | 用户从个人主页打开 AI 面板 | 指示 AI 先查近期帖子、再查互动历史、概括至少 3 个要点、末尾引用贴文 |
| `PF_POST_CONTEXT(uri)` | 用户从帖子/线程打开 AI 面板 | `用户正在查看帖子 at://...，如果需要请用工具获取上下文。` |

注意 `PF_PROFILE_CONTEXT` 的行为最为复杂：它不仅告知 AI 用户在查看谁，还指明了**操作顺序**（先 `get_author_feed`、再 `search_posts` 查互动）、**输出要求**（至少 3 个要点 + 引用格式）、以及**兜底逻辑**（无互动则跳过）。这实际上是一个**微型工作流模板**，提示词即逻辑。

[来源](packages/core/src/ai/prompts.ts#L61-L94) | [组装逻辑](packages/app/src/hooks/useAIChat.ts#L68-L86)

### PF_ENVIRONMENT：TUI / PWA 环境感知

```
你运行在终端命令行界面 (TUI/CLI) 中。当前用户通过命令行输入和你交互，
输出是纯文本。保持回复简短，避免复杂格式，每行不要超过80个字符。
可使用OSC 8超链接但不支持图片内嵌。
```

```
你运行在网页浏览器 (PWA) 中。当前用户通过网页界面和你交互，
支持图片、Markdown格式和超链接。
```

**关键设计**：`PF_ENVIRONMENT` 的存在意味着同一套 AI 引擎同时服务于两个完全不同的渲染环境。TUI 下 AI 被要求保持短行、纯文本、无图片，因为终端渲染能力有限；PWA 下则可利用 Markdown 和图片获得更丰富的展示。这是"一个引擎、两个前端"架构在提示词层面的体现。

[来源](packages/core/src/ai/prompts.ts#L96-L102) | 关于 TUI/PWA 差异详见[包架构深度解析](包架构深度解析.md)

### PF_LOCALE_HINT：语言偏好

```
用户界面语言: zh，请优先用该语言回复。
```

一句话指令，让 AI 跟随用户界面语言回复。值来自 i18n 系统记录的语言设置。

[来源](packages/core/src/ai/prompts.ts#L104-L110)

---

### PF_CURRENT_TIME / PF_VISION_HINT：辅助信息

**`PF_CURRENT_TIME()`**：采样系统时钟，输出格式如 `当前时间: 2025/01/15 14:30:00，星期三。`。这使 AI 具备时间感知能力，在分析"近期动态"或"今日趋势"时能正确理解时间范围。使用 `zh-CN` locale 固定格式化，与用户界面语言解耦。

**`PF_VISION_HINT(enabled)`**：根据 `LLM_VISION_ENABLED` 配置输出两种文案之一：
- **开启时**：告知 AI 可使用 `view_image` 查看图片、`download_image` 保存到本地。
- **关闭时**：告知 AI 视觉模式未开启，并有条件地建议用户开启——**但 AI 自身若不支持视觉（如纯文本模型），不应建议用户开启，以免浪费上下文**。这个"自知之明"逻辑完全依靠提示词约束，无后端校验。

[来源](packages/core/src/ai/prompts.ts#L115-L134)

### P_CONCISE：精简回答指令

```
回答简练。
```

全文最短的一个常量，却承担了**质量管控**的功能。作为系统提示词的最后一块拼图，它确保 AI 不会输出冗长的套话。

[来源](packages/core/src/ai/prompts.ts#L112-L113)

---

### PF_AUTO_ANALYSIS：自动分析指令

```
请分析 @{handle} 的主页，概括他们的近期动态。
```

当用户从个人主页打开 AI 面板且对话为空时，`useAIChat` 组件自动发送此消息，触发 AI 执行分析流程。延迟 500ms 发送，给 UI 渲染留出缓冲。

[来源](packages/core/src/ai/prompts.ts#L179-L185) | [自动发送逻辑](packages/app/src/hooks/useAIChat.ts#L369-L379)

### P_GUIDING_QUESTIONS：引导式提问

```typescript
['总结这个讨论', '解释这个讨论', '分析帖子情绪']
```

三个预设问题，在用户刚打开 AI 面板且无历史对话时显示在 UI 中作为快捷入口。它们对应的正是 `PF_POST_CONTEXT` 上下文中最常见的三类需求：**总结**、**解释**、**情绪分析**。这些引导问题并不注入系统提示词，而是作为 UI 状态供组件渲染。

## 提示词组装管线

`buildSystemPrompt` 函数严格按以下顺序拼接各片段：

```
P_ASSISTANT_BASE  →  PF_CURRENT_USER  →  PF_PROFILE/POST_CONTEXT  →  PF_ENVIRONMENT  →  PF_LOCALE_HINT  →  PF_CURRENT_TIME  →  PF_VISION_HINT  →  P_CONCISE
```

顺序有讲究：**身份** → **上下文** → **环境** → **语言偏好** → **时间** → **视觉能力** → **简洁指令**。越通用的越靠前（如角色定义），越具体的越靠后（如简洁指令放在最后，作为全篇收束）。

[来源](packages/app/src/hooks/useAIChat.ts#L68-L86)

## 安全设计全景

| 层级 | 机制 | 拦截点 | 位置 |
|------|------|--------|------|
| **提示词层** | 3 条硬规则（不得主动写操作） | LLM 生成阶段 | `prompts.ts` |
| **工具定义层** | `requiresWrite` 标记 | 工具执行前 | `tools.ts` |
| **运行时层** | 写确认门（`_waitForConfirmation`） | 工具执行前 | `assistant.ts` |

三层防御确保：即使提示词被绕过（如用户自定义 prompt injection），仍会在工具执行前被确认门拦住。这并非过度设计——AI 模型在长对话中可能"遗忘"初始约束，运行时确认是最后的保险。

## 延伸：翻译与润色提示词

`prompts.ts` 还承载了翻译和润色功能所需的提示词，它们通过 `singleTurnAI` 函数独立调用，不参与主对话的系统提示词组装：

- **`PF_TRANSLATE_SIMPLE`**：纯文本翻译，仅输出译文。
- **`PF_TRANSLATE_JSON`**：结构化翻译，要求输出 `{"source_lang": "...", "translated": "..."}` 格式 JSON，支持源语言检测。
- **`P_POLISH_SYSTEM`** / **`PF_POLISH_USER`**：草稿润色双提示词（system + user 模板）。

翻译使用 `thinking: { type: 'disabled' }` 和较低温度（0.3），避免创造性输出干扰翻译准确性。

[来源](packages/core/src/ai/prompts.ts#L136-L173) | [singleTurnAI 实现](packages/core/src/ai/assistant.ts#L661-L707) | 详见[翻译与润色系统](翻译与润色系统.md)
