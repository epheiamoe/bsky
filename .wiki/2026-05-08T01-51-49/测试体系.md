# 测试体系

## 架构哲学：零 Mock，全部真实

项目采用一种极端的集成测试策略：**所有测试直接调用 Bluesky 真实 API 和 DeepSeek 真实 API，不引入任何 Mock 或 Stub**。这意味着测试套件既验证代码逻辑，也验证第三方服务的可用性和契约兼容性。

这种策略的代价是测试耗时较长（全量运行约 103 秒），但换来了极高的信任度——通过即意味着生产环境可用。测试运行在 `@bsky/core` 包内，与被测代码共享相同模块路径，无需任何适配层。

### 环境变量

测试从项目根目录的 `.env` 文件加载凭证，通过 `dotenv` 手动读取：

```typescript
dotenv.config({ path: path.resolve(__dirname, '..', '..', '..', '.env') });
```

| 变量 | 是否必需 | 说明 |
|------|---------|------|
| `BLUESKY_HANDLE` | 是 | Bluesky 用户名（如 `user.bsky.social`） |
| `BLUESKY_APP_PASSWORD` | 是 | Bluesky 应用密码（设置 → App Passwords） |
| `LLM_API_KEY` | 是 | OpenAI 兼容 API 密钥（默认使用 DeepSeek） |
| `LLM_BASE_URL` | 否 | API 基地址，默认 `https://api.deepseek.com` |
| `LLM_MODEL` | 否 | 模型名，默认 `deepseek-v4-flash` |

缺失 `BLUESKY_HANDLE` 或 `BLUESKY_APP_PASSWORD` 时，`beforeAll` 会抛出 `Missing env vars` 错误；缺失 `LLM_API_KEY` 时，AI 相关测试会被 `it.skip` 跳过。三个必需变量的详细说明见 [环境变量与配置](环境变量与配置.md)。

## E2E 完整流程设计

`e2e.test.ts` 原本规划了一个横跨 10 步的完整用户旅程，从登录到 AI 辅助的引导式探索。但出于**避免在真实账户上留下测试数据**的考虑，涉及写操作的步骤（创建帖子、上传图片）已被注释，当前运行的活跃测试如下：

```
[1]  Authentication   ─── login → resolveHandle → getProfile
                               ↓
[2]  Feed             ─── getTimeline (只读)
                               ↓
[5]  Translation      ─── translateToChinese ("Bluesky is a decentralized...")
                               ↓
[6]  Polish           ─── polishDraft ("bluesky is cool i like it" → 更正式)
                               ↓
[7]  Profile & Graph  ─── getProfile('bsky.app') → getFollows
                               ↓
[8]  Timeline         ─── getTimeline(10)
                               ↓
[9]  Notifications    ─── listNotifications(10)
                               ↓
[10] Guiding Qs       ─── singleTurnAI 生成引导性问题
```

**被注释的写操作流程**（设计意图，仅供阅读参考）：

```
login → createPost → getPostThread → get_post_thread_flat → searchPosts (3s 索引等待)
     → uploadBlob → createRecord (含 embed) → extract_images_from_post → download_image (2s 延迟)
     → AIAssistant.sendMessage (分析帖子) → translateToChinese → polishDraft
```

每个步骤的实现在 [BskyClient 深度解析](bskyclient-深度解析.md)（API 端点）和 [AI 助手引擎](ai-助手引擎.md)（AI 调用）中有详细说明。

[来源](packages/core/tests/e2e.test.ts#L37-L186)

## Vitest 配置

```typescript
// packages/core/vitest.config.ts
import { defineConfig } from 'vitest/config';
export default defineConfig({
  test: {
    globals: true,
    testTimeout: 60000,   // 全局超时 60 秒
    hookTimeout: 30000,    // beforeAll/afterAll 超时 30 秒
  },
});
```

- `globals: true`：无需手动导入 `describe`/`it`/`expect`。
- 60 秒超时是默认值（30 秒）的两倍，因为真实 API 调用（尤其是 AI 多轮工具调用）耗时较长。
- 个别用例在 `it` 第三参数进一步覆盖超时：认证相关 15 秒，AI 工具调用 120 秒，翻译/润色 60 秒。

[来源](packages/core/vitest.config.ts#L1-L9)

## 常见问题与修复

测试在持续运行中积累了若干典型问题的解决方案：

| 问题 | 表现 | 修复 |
|------|------|------|
| **默认超时不足** | AI 多轮工具调用或图片下载超时 | 全局超时设 60s；AI 工具调用用例单独设 120s（`it('...', fn, 120000)`）；认证用例设 15s |
| **Blob 下载竞态** | `download_image` 在图片刚上传后调用失败 | 下载前加入 `await new Promise(r => setTimeout(r, 2000))` 2 秒延迟，确保 CDN 完成分发 |
| **搜索索引延迟** | 刚创建的帖子通过 `searchPosts` 查不到 | 搜索前等待 3 秒（但当前搜索测试使用现有关键词 "Bluesky"，该延迟已不必要，注释中保留了 2s 延迟） |
| **JSON 解析错误** | AI 返回内容未严格按 JSON 格式输出 | 写操作确认使用非流式模式，`max_tokens: 4096`；AI 回复由工具结果格式化层保证结构完整性 |
| **测试残留数据** | 多次运行后账户出现大量测试帖子 | 测试默认不创建帖子（代码注释中保留了创建逻辑但已禁用）；写操作前有确认门控（见 [31 个 AI 工具系统](31-个-ai-工具系统.md)） |

[来源](packages/core/tests/e2e.test.ts#L113-L114)，[来源](packages/core/tests/feed.test.ts#L133)，[来源](packages/core/tests/feed.test.ts#L133)

## 测试数据隔离策略

- **无写操作**：当前所有活跃测试均为只读，不创建、修改或删除 Bluesky 上的任何数据。创建帖子/上传图片的代码保留在注释中，供需要时启用。
- **帖子标记**：如果未来启用帖子创建测试，所有测试帖子标题包含 `[TEST 测试]` 或 `[TEST E2E]` 标记，方便识别和过滤。
- **清理策略**：`afterAll` 仅保留清理注释或空实现——AT Protocol 不提供便捷的帖子删除端点，测试帖子会永久留在账户中，因此当前选择不创建。

[来源](packages/core/tests/e2e.test.ts#L32-L35)，[来源](packages/core/tests/feed.test.ts#L28-L30)

## 与包的集成点

| 被测对象 | 引入路径 | 对应文档 |
