# 33 个 AI 工具系统

`createTools` 工厂函数定义了 33 个 **ToolDescriptor**（27 个读工具 + 6 个写工具），构成 AI 助手的全部能力边界。每个工具封装一个 AT Protocol API 调用或复合操作，通过统一接口暴露给 LLM。

## ToolDescriptor 三要素

每个工具由 `packages/core/src/ai/tools.ts` 中的 `ToolDescriptor` 接口定义：

```typescript
export interface ToolDescriptor {
  definition: ToolDefinition;  // 元数据：名称、描述、JSON Schema
  handler: ToolHandler;        // 执行逻辑：(params, assistant?) => Promise<string>
  requiresWrite: boolean;      // 是否为写操作
}
```

- **definition**：LLM 可见的工具签名。`description` 字段直接嵌入系统提示词，引导模型选择正确的工具。`inputSchema` 使用 JSON Schema 描述参数结构。
- **handler**：实际的业务逻辑。接受 `Record<string, unknown>` 参数，通过 `client` 调用 AT Protocol API，返回 JSON 序列化字符串。写工具的 handler 还会额外接收 `assistant` 实例引用，用于获取用户上传的图片数据。
- **requiresWrite**：布尔标记。`true` 的工具在 `AIAssistant.sendMessage` 和 `sendMessageStreaming` 中会触发确认对话框。

## 线程扁平化算法 (`get_post_thread_flat`)

`get_post_thread` 工具在 `format='flat'`（默认）时调用 `flattenThread` 函数（第 1125-1173 行），将 AT Protocol 的树状线程结构转换为人类可读的线性文本。

### 算法步骤

1. **获取原始线程**：调用 `client.getPostThread(uri, Math.max(maxDepth + 1, 6), 80)`。注意内部深度至少设为 6（AT API 的最小深度请求单位），但结果会被后续 `maxDepth` 参数截断。

2. **收集祖先链**：从当前帖子的 `parent` 链向上遍历，将祖先节点倒序排列，使得输出中最早的帖子先显示。

3. **格式化根帖子**：深度标记为 `0`，附带 `threadgate` 信息（如 `[reply restriction: mentioned + followers]`）。

4. **递归展开回复**：`buildReplyLines` 函数对每层回复：
   - 过滤掉 `NotFoundPost`（已删除的帖子）
   - 按 `indexedAt` 时间戳排序（最早的在前）
   - 只显示前 `maxReplies` 条（默认 5，最大 20），多余条目标注折叠提示
   - 每层递归深度受 `maxDepth` 限制（默认 3）

### 深度限制参数

- **flat/subtree** 格式：`depth` 默认 3，`maxReplies` 默认 5（上限 20）
- **tree** 格式：`depth` 默认 6，无 `maxReplies` 限制

### 输出示例

```
depth:-2 | alice.bsky.social (Alice) (post:abc123)
text: 最初的帖子内容
 ↳ > depth:-1 | bob.bsky.social (Bob) (post:def456)
text: 回复 Alice
depth:0 | charlie.bsky.social (Charlie) (post:ghi789)
text: 核心帖子内容 [图片: 2 张]
  ↳depth:1 → (post:ghi789) | dave.bsky.social (Dave) (post:jkl012)
text: 第一级回复
    ↳depth:2 → (post:jkl012) | eve.bsky.social (Eve) (post:mno345)
text: 第二级回复
    （还有 3 条回复被折叠，可调用 get_post_thread(format='subtree') 展开）
```

## 写工具确认对话框触发条件

确认对话框的触发由 AIAssistant 类中的 `requiresWrite` 门机制控制（`assistant.ts` 第 250-271 行和第 597-611 行）。

### 触发流程

```
LLM 调用工具 → 检查 requiresWrite
  ├─ false → 直接执行 handler
  └─ true  → 暂停循环，发出 confirmation_needed 事件
              ├─ 用户确认 (confirmAction(true)) → 执行 handler
              └─ 用户取消 (confirmAction(false)) → 返回 "User cancelled the operation."
```

### 具体触发条件

以下六种工具会触发确认对话框：

| 工具 | 描述文本格式 |
|------|------------|
| `create_post` | `创建帖子: "帖子前100字..."` |
| `like` | `点赞帖子: at://...` |
| `repost` | `转发帖子: at://...` |
| `follow` | `关注用户: handle_or_did` |
| `create_list` | `创建列表: "名称" (精选/管理)` |
| `edit_list_members` | `添加用户 xxx 到列表` 或 `从列表移除用户 xxx` |

描述由 `buildToolDescription` 函数生成（第 647-660 行），通过流式 API 的 `confirmation_needed` 事件发送到 UI 层。

## 零密钥知识查询工具的实现原理

系统提供两个**无需任何 API 密钥**的实时知识查询工具，底层分别依赖 DuckDuckGo（公开搜索引擎）和 Wikipedia（公共知识库）。

### `search_web_ddg`：三级降级架构

该工具不依赖任何 API 密钥，通过三阶段降级策略确保高可用性：

```
Tier 1: jina.ai Reader 代理
  └─ fetch(`https://r.jina.ai/https://html.duckduckgo.com/html?q=QUERY`)
  └─ 返回 Markdown → cleanJinaSearchOutput() 清洗（去广告、去页脚、解 DDG 重定向URL）
  └─ 成功则返回 { heading, content }

Tier 2 (fallback): DDG Lite 直接调用
  └─ TUI: fetch(`https://lite.duckduckgo.com/lite?q=QUERY`)
  └─ PWA: fetch(`/api/search?q=QUERY`) → 经 Cloudflare Pages Function 代理（避免 CORS）
  └─ parseDDGLite() 从 HTML <table> 中解析搜索结果

Tier 3: 返回空结果
```

`cleanJinaSearchOutput` 函数（第 1267-1319 行）执行多项清洗：
- 移除 jina.ai 的 Markdown Content 头部
- 跳过 DDG 页面标题和 logo 链接
- 移除广告标记和隐私声明
- 移除 favicon 图片行
- 将 DDG 重定向 URL (`https://duckduckgo.com/l/?uddg=...`) 还原为真实 URL（通过 `extractRealUrl`）

### `search_wikipedia`：REST API 直接调用

该工具直接调用 Wikipedia 的 REST API，无需任何密钥配置：

```
fetch(`https://{lang}.wikipedia.org/api/rest_v1/page/summary/${encodeURIComponent(query)}`)
  ├─ 200 → formatWikipediaSummary(summary)
  ├─ 404 → 返回 "No Wikipedia article found"
  └─ 其他错误 → 返回错误信息和网络提示
```

`formatWikipediaSummary` 函数（第 1324-1331 行）将 API 返回的 `WikipediaSummary` 格式化为统一的 `{ heading, content }` 结构，包含标题、描述、摘要文本和来源链接。

默认语言为 `en`，支持通过 `lang` 参数切换任意 Wikipedia 语言（`zh`、`ja` 等），参数经过 `[^a-z]` 正则清洗防止注入。

## 设计模式：复合工具

系统中存在多个**复合工具**——它们不直接映射到单一 AT Protocol API，而是组合多个 API 调用完成高层次任务：

- **`get_post_thread`**（flat 格式）：调用 `getPostThread` 获取树状结构，再通过 `flattenThread` 转换为线性文本
- **`get_post_context`**：调用 `getPostThread` + `getRecord`，组合线程、媒体摘要、回复状态
- **`get_quotes`**：通过 `searchPosts` API 实现引用搜索
- **`view_image`**：调用 `downloadBlob` 下载图片 + 通过 `addPendingImage` 注入到下一轮对话
- **`create_post`** 的图片附件：从源 PDS 下载图片 → 上传到用户 PDS → 创建记录

复合工具通过 `requiresWrite: false` 标记只读操作，避免不必要的确认对话框。

## 工具注册与生命周期

工具通过 `createTools(client)` 工厂函数创建，依赖 `BskyClient` 实例。在 `AIAssistant.setTools()` 中注册后，每个工具被放入 `toolMap`（以名称为键），并在每次 API 请求时通过 `body.tools` 数组传递给 LLM（`assistant.ts` 第 378-388 行）。

工具调用的完整生命周期：

```
LLM 返回 tool_calls → 解析 JSON 参数 → 查 toolMap → 写操作? → 确认 → handler → 存入消息历史 → 下一轮
```
