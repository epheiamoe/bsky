# 测试体系

## 测试哲学：不 Mock，全真实

这个项目的测试有一个鲜明的立场——**所有测试都是集成测试，不使用任何 Mock**。每次 `pnpm test` 都会真正调用 Bluesky API 和 DeepSeek API（或其他配置的 LLM 提供商）。

为什么不做 Mock？

- Mock 伪造网络行为，但真实的 Bluesky 网络环境才暴露真正的问题——JWT 过期、PDS 故障、速率限制、搜索结果延迟索引
- 写操作的测试（发帖、上传图片）被注释掉，因为 AT Protocol 不支持便捷删除，项目选择不污染用户账号。所有活跃的测试都是**只读操作**：登录、读取时间线、搜索、查询资料、AI 分析已有帖子

> "如果测试不能到达真实网络，它测量的只是我们对自己的想象。"  
> 来源：`[docs/TESTING.md#L6-L7](docs/TESTING.md#L6-L7)`

## 5 个测试文件

项目实际有 5 个测试文件（比在线文档多一个 `list.test.ts`），全部位于 `packages/core/tests/`。

### 1. `auth.test.ts` — 认证与基础端点（3 个测试）

| 测试 | 验证点 | 超时 |
|------|--------|------|
| 创建会话返回 accessJwt | `session.accessJwt` 非空、handle 匹配、DID 符合 `did:plc:` 格式 | 15s |
| 解析自己的 handle 得到相同 DID | `resolveHandle(HANDLE).did === client.getDID()` | 15s |
| 获取自己的个人资料 | `profile.handle === HANDLE` | 15s |

这三个测试串起了整个应用的入口：**登录 → 解析 → 查资料**。如果 `auth.test.ts` 全红，说明 Bluesky 凭据或网络有问题，后面的测试不用跑了。

来源：`[packages/core/tests/auth.test.ts](packages/core/tests/auth.test.ts)`

### 2. `feed.test.ts` — 帖子读取与序列化（1 个活跃测试）

这个文件很有趣——大部分测试被注释掉了。当前只有一个活跃测试：

- **搜索帖子**：调用 `client.searchPosts({ q: 'Bluesky', limit: 25, sort: 'latest' })`，验证 `searchRes.posts.length >= 0`

注释掉的测试涵盖了**发帖 → 读线程 → 扁平化 → 搜索自己的帖子 → 上传图片 → 提取图片 → 下载图片 → 获取上下文**的完整链路。它们被注释不是因为错误，而是因为**写操作会留下永久数据**。未来如果 AT Protocol 支持删除，这些测试可以解封。

来源：`[packages/core/tests/feed.test.ts](packages/core/tests/feed.test.ts)`

### 3. `ai_integration.test.ts` — AI 集成（8 个测试）

最复杂的测试文件，分为三个 `describe` 块：

**块 1：AI 工具调用流程（3 个测试）**
- 加载所有工具定义：验证 `tools.length > 20`，且包含 `get_post_thread_flat`、`search_posts`、`get_profile`
- 通过 AI 搜索帖子：构造 `AIAssistant` → 设置工具 → 发消息让 AI 搜索 "Bluesky" → 验证 `toolCallsExecuted >= 1`
- 通过 AI 查个人资料：同上，让 AI 查当前用户的资料

**块 2：翻译与润色（3 个测试）**
- `translateToChinese()`：英译中，验证结果包含中文字符
- `polishDraft()`：润色草稿（"更正式"），验证结果非空
- `polishDraft()` 幽默风格：润色（"更幽默"），验证结果非空

**块 3：引导性问题（1 个测试）**
- `singleTurnAI()` 为一个公开帖子生成 3 个引导性问题，验证行数 >= 1

如果缺少 `LLM_API_KEY`，第 2、3 块会用 `it.skip` 跳过，不会导致套件失败。

来源：`[packages/core/tests/ai_integration.test.ts](packages/core/tests/ai_integration.test.ts)`

### 4. `e2e.test.ts` — 端到端全流程（10 个测试）

这是最重的测试，串联了 10 个步骤，编号从 `[1]` 到 `[10]`：

| 步骤 | 测试内容 | 超时 |
|------|----------|------|
| `[1]` | 认证：登录 + resolveHandle + getProfile | 15s |
| `[2]` | 读取时间线 + 验证工具加载（只读） | 15s |
| `[5]` | AI 英译中 | 60s |
| `[6]` | AI 润色草稿（更正式） | 60s |
| `[7]` | 获取 bsky.app 资料 + 自己的关注列表 | 15s |
| `[8]` | 读取时间线 | 15s |
| `[9]` | 列出通知 | 15s |
| `[10]` | AI 生成引导性问题 | 60s |

步骤 `[3]`（图片上传下载）和 `[4]`（AI 分析自己发的帖）被注释掉，原因同上——写操作。

来源：`[packages/core/tests/e2e.test.ts](packages/core/tests/e2e.test.ts)`

### 5. `list.test.ts` — 列表 API 集成（12 个测试）

这是最新的测试文件，覆盖了 [AT Protocol 客户端](AT Protocol 客户端.md) 中新增的列表 API 方法。它是一个**完整的创建-读取-更新-删除循环**：

| 测试 | 操作 |
|------|------|
| 登录 | 创建会话 |
| 创建列表 | `client.createList('测试列表', 'curatelist')` |
| 按 URI 获取列表 | `client.getList(uri)` 验证名称/用途/创建者 |
| 获取自己的所有列表 | `client.getLists(HANDLE)` 验证包含创建的列表 |
| 按用途过滤 | `client.getLists(HANDLE, 50, undefined, ['curatelist'])` |
| 添加成员 | `client.addListItem(uri, 其他用户DID)` |
| 验证成员可见 | `getList` 返回的 items 中包含刚添加的用户 |
| 获取列表时间线 | `client.getListFeed(uri)` 验证是数组 |
| 查询用户所属列表 | `client.getListsWithMembership(did)` |
| 静音/取消静音列表 | `muteActorList → getListMutes → unmute → getListMutes 验证移除` |
| 更新列表描述 | `updateList` 改描述 → `getList` 验证 |
| 移除成员 | `removeListItem` → `getList` 验证成员消失 |
| 清理（afterAll） | `deleteList` 删除创建的列表 |

这是唯一一个**彻底清理所有写操作痕迹**的测试文件——`afterAll` 中会 `deleteList`。

来源：`[packages/core/tests/list.test.ts](packages/core/tests/list.test.ts)`

## 测试环境搭建

### 环境变量

所有测试文件通过 `dotenv` 从项目根目录加载 `.env` 文件：

```typescript
dotenv.config({ path: path.resolve(__dirname, '..', '..', '..', '.env') });
```

路径解析：`tests/` → `src/` → 项目根。来源：`[packages/core/tests/auth.test.ts#L7-L8](packages/core/tests/auth.test.ts#L7-L8)`

**必需变量：**

| 变量 | 用途 | 获取方式 |
|------|------|----------|
| `BLUESKY_HANDLE` | 测试账号的 handle | 在 Bluesky 注册账号 |
| `BLUESKY_APP_PASSWORD` | 应用专用密码 | Bluesky 设置 → App Passwords → 创建 |
| `LLM_API_KEY` | AI 测试的 API Key | DeepSeek / 其他提供商 |

**可选变量：**

| 变量 | 默认值 | 用途 |
|------|--------|------|
| `LLM_BASE_URL` | `https://api.deepseek.com` | 自定义 LLM 端点 |
| `LLM_MODEL` | `deepseek-v4-flash` | 指定模型 |

来源：`[docs/TESTING.md#L68-L73](docs/TESTING.md#L68-L73)`

### 测试账号建议

- 使用**专用测试账号**，不要用个人主账号
- 即使如此，测试中所有写操作（发帖、传图）都被注释掉了，不会产生内容
- `list.test.ts` 会创建和删除列表，但为保险起见，建议测试账号应为一个**干净的新账号**
- 测试账号需关注一些用户（否则时间线为空，`getTimeline` 测试会失败）

## 常见问题与解法

### 超时（最常见）

每个测试都有独立的超时时间，通过 Vitest 的第三个参数设置：

```typescript
it('should do something', async () => {
  // ...
}, 60000); // 60 秒
```

| 场景 | 超时 | 原因 |
|------|------|------|
| 认证测试 | 15s | API 响应通常很快 |
| 搜索/帖子 | 30s | 可能需要等待索引 |
| AI 翻译/润色 | 60s | LLM 推理较慢 |
| AI 工具调用（多轮） | 120s | 2~3 轮工具调用循环 |

全局配置在 `vitest.config.ts` 中设置 `testTimeout: 60000` 和 `hookTimeout: 30000`。

来源：`[packages/core/vitest.config.ts#L5-L7](packages/core/vitest.config.ts#L5-L7)`

### 搜索返回空结果

Bluesky 的搜索引擎有索引延迟。新发的帖子不会立刻被搜到。解法：

- 搜索时添加 2~3 秒 `setTimeout` 等待索引（`e2e.test.ts` 中有 `await new Promise(r => setTimeout(r, 2000))`）
- 搜索关键词使用已有帖子的通用词如 "Bluesky" 而非刚创建的标记
- 测试中 `searchRes.posts.length >= 0` 的断言不要求必须有结果，只要求 API 正常返回

### 图片下载超时

图片 blob 上传后需要时间传播到 CDN。解法：

- 下载前加 2 秒延迟
- 使用带认证的端点（`/xrpc/com.atproto.sync.getBlob` 而非公共端点）
- 下载请求设置 30 秒超时

### AI 响应被截断

- 使用**非流式**模式，确保一次拿到完整响应
- 设置 `max_tokens: 4096`
- 终端内文字用 `Box` 组件包裹以自动换行

## 命令速查表

| 命令 | 作用 | 说明 |
|------|------|------|
| `pnpm test` | 运行所有包的全部测试 | 根级别，递进到各子包 |
| `pnpm --filter @bsky/core test` | 只运行 core 包测试 | 5 个文件全跑 |
| `pnpm --filter @bsky/core test tests/auth.test.ts` | 运行单个测试文件 | 路径相对于 `packages/core/` |
| `pnpm --filter @bsky/core test:e2e` | 仅运行 E2E 套件 | 等价于 `vitest run tests/e2e.test.ts --reporter=verbose` |
| `pnpm --filter @bsky/core test:watch` | 监听模式 | 文件变动自动重跑 |

来源：`[packages/core/package.json#L17-L19](packages/core/package.json#L17-L19)`，`[package.json#L9-L10](package.json#L9-L10)`

> ⚠️ **注意**：`pnpm test`（根级别）会递归执行所有子包的 `test` 脚本。如果 `packages/app/tui` 或 `packages/app/pwa` 没有测试脚本，可能要配置 `"test": "echo 'no tests'"` 避免报错。当前只有 `packages/core` 有实际测试。

## 添加新测试的模板

```typescript
import { describe, it, expect, beforeAll } from 'vitest';
import { BskyClient } from '../src/at/client.js';
import dotenv from 'dotenv';
import path from 'path';
import { fileURLToPath } from 'url';

const __dirname = path.dirname(fileURLToPath(import.meta.url));
dotenv.config({ path: path.resolve(__dirname, '..', '..', '..', '.env') });

const HANDLE = process.env.BLUESKY_HANDLE!;
const APP_PASSWORD = process.env.BLUESKY_APP_PASSWORD!;

describe('My New Tests', () => {
  let client: BskyClient;

  beforeAll(async () => {
    if (!HANDLE || !APP_PASSWORD) throw new Error('Missing env vars');
    client = new BskyClient();
    await client.login(HANDLE, APP_PASSWORD);
  });

  it('should do something real', async () => {
    const result = await client.someMethod();
    expect(result).toBeTruthy();
  }, 30000); // 根据 API 响应速度调整
});
```

规则：
- 放在 `packages/core/tests/` 下，`.test.ts` 后缀
- 从 `../src/` 导入 `BskyClient`、`createTools`、`AIAssistant`
- 真实环境 + 真实凭据 —— **不 Mock**
- 测试帖子会永久留在 Bluesky 上（AT Protocol 没有便捷删除 API），所以写操作测试一律注释

来源：`[docs/TESTING.md#L87-L91](docs/TESTING.md#L87-L91)`
