# 时间线、帖子与线程

三个 Hook 覆盖了 Bluesky 社交浏览的核心路径：**时间线流式加载**、**线程树浏览**、**单帖详情查看**。它们分别对应 `useTimeline`、`useThread`、`usePostDetail`，三者共享相同的数据来源（`BskyClient`），但在数据形态、交互模型和渲染准备上各有分工。

---

## useTimeline：cursor 分页与刷新

`useTimeline` 是三个 Hook 中最"平"的一个——它把时间线（或自定义 Feed）的帖子列表以 **cursor 分页** 方式递交给 UI。

```typescript
// packages/app/src/hooks/useTimeline.ts#L6-L10
export function useTimeline(client: BskyClient | null, feedUri?: string) {
  const [store] = useState(() => createTimelineStore());
  const [, force] = useState(0);
  const tick = useCallback(() => force(n => n + 1), []);
```

其核心模型是一个 **纯 Store**：`createTimelineStore()` 返回一个包含 `posts[]`、`cursor`、`loading`、`error` 的对象，所有状态变更通过发布订阅通知 React 重新渲染。这正是 [React Hooks 架构与 Store 模式](react-hooks-架构与-store-模式.md) 中描述的"Store + Hook 桥接模式"的典型实例。

### 分页加载

`loadMore` 方法在每次调用时将 `cursor` 传给 API，并把新返回的帖子追加到 `posts` 末尾：

```
[来源](packages/app/src/stores/timeline.ts#L61-L78)

const res = shouldUseTimeline(feedUri)
  ? await client.getTimeline(20, store.cursor)
  : await client.getFeed(feedUri!, 20, store.cursor);
store.posts = [...store.posts, ...res.feed.map(f => f.post)];
store.cursor = res.cursor;
```

### Refresh 机制

`refresh` 是"暴力重置"：清空 `cursor` 和 `posts`，再重新调用 `load`。这在首次进入时间线或用户手动刷新时触发。

```
[来源](packages/app/src/stores/timeline.ts#L80-L84)

async refresh(client, feedUri) {
  store.cursor = undefined;
  store.posts = [];
  await store.load(client, feedUri);
}
```

### Feed URI 切换与重载

`useTimeline` 内部通过 `lastFeed` ref 检测 `feedUri` 变化。当消费方传入新的 `feedUri`（非 undefined），它会自动清空旧数据并触发重载。导航离开再返回时，`lastGoodFeed` 兜住了上次有效的 Feed URI，避免不必要的重置。

```
[来源](packages/app/src/hooks/useTimeline.ts#L12-L28)
```

### 双路由判断

`shouldUseTimeline()` 函数区分"主页时间线"与"自定义 Feed 生成器"：前者调用 `client.getTimeline()`，后者调用 `client.getFeed()`。两者 API 结构相同，区别在于后者需传 `feedUri` 参数。


Store 的 `load` 方法内置了 2 次重试机制，首次失败后等待 1.5 秒重试——这是为了容忍 **JWT 刷新** 与首次请求之间的竞态条件。

## useThread：线程树的扁平化与焦点模型

线程浏览是 Bluesky 社交中最复杂的交互场景。`useThread` 的核心设计思路是：**把 API 返回的嵌套树（`ThreadViewPost`）拍平为一维数组，以深度标记层级关系，并分离"逻辑焦点"与"视觉光标"**。

### FlatLine 接口：Depth 即层级

`FlatLine` 是扁平化后的"通用帖子行"，通过 `depth` 字段标记其在树中的位置：

| depth 值 | 含义 |
|----------|------|
| `< 0` | **父链**（-1 为直接父帖，-2 为祖父帖，依此类推） |
| `0` | **根帖子**（`isRoot = true`） |
| `> 0` | **回复链**（+1 为一级回复，+2 为二级回复） |

```
[来源](packages/app/src/hooks/useThread.ts#L140-L214)
```

扁平化的核心是 `flattenThreadTree()` 函数。它从根节点出发，先递归遍历 `parent` 链（depth 递减），再遍历 `replies` 列表（depth 递增），最终生成一个按深度排序的线性数组。

```typescript
function walk(node: ThreadViewPost | NFP, d: number) {
  // … 递归向上走 parent 链（d-1），再向下走 replies（d+1）
  if (node.parent && ...) walk(node.parent, d - 1);
  lines.push({ depth: d, /* ... */ });
  if (node.replies && d >= 0) {
    // 按时间排序，截断到 maxSiblings
    walk(reply, d + 1);
  }
}
```

```
[来源](packages/app/src/hooks/useThread.ts#L140-L214)
```

### 媒体元数据前置解析

`FlatLine` 不止保存文本和作者信息，还提前解析了丰富的 **媒体元数据**，让 UI 层无需再次遍历 API 原数据：

| 字段 | 类型 | 来源 |
|------|------|------|
| `mediaTags` | `string[]` | `getMediaTags()` — 判别图片/动图/视频/链接/引用帖，生成人类可读标签（如 "🖼 3张图片"） |
| `imageDetails` | `Array<{url, alt}>` | `getImageDetails()` — CDN 图片 URL 及 alt 文本 |
| `externalLink` | `{uri, title, description} \| null` | `getExternalLink()` — 外部链接卡片 |
| `hasVideo` / `videoThumbnailUrl` / `videoPlaylistUrl` | 布尔 + URL | `getVideoInfo()` — HLS 视频播放 |
| `quotedPost` | 嵌套 FlatLine | `getQuotedPost()` — 被引用的嵌入帖子，递归解析其媒体 |

```
[来源](packages/app/src/hooks/useThread.ts#L216-L368)
```

其中 `getMediaTags` 对嵌入类型做了精细分类——单纯的 `app.bsky.embed.record` 显示"📌 引用"，`recordWithMedia` 则显示"📌🖼 引用+图片"或"📌🎞 引用+动图"。

### 回复截断与展开

API 返回的 `getPostThread(uri, depth=5, parentHeight=80)` 能获取完整线程，但 `useThread` 通过 `maxSiblings` 控制同一层级展示的回复数量（初始 `INITIAL_SIBLINGS = 5`）。当回复数超过阈值时，生成一条 `isTruncation = true` 的占位行：

```typescript

  depth: d + 1, isTruncation: true,
  text: `（还有 ${remaining} 条回复未显示）`,
  // …
});
```

用户点击"展开"时调用 `expandReplies()`，将 `maxSiblings` 增加 10，然后重新请求 API 并重新扁平化。

### 光标/焦点分离模式

这是 `useThread` 设计中最精巧的一点——**逻辑焦点（focusedIndex）与视觉光标（cursorIndex）是分离的**。

```
[来源](packages/app/src/hooks/useThread.ts#L53)

const [focusedIndex, setFocusedIndex] = useState(0);
```

- **focusedIndex**（来自 useThread）：用户"当前正在看的帖子"。初始值为 `isRoot` 帖子的索引。它不随方向键移动而改变。
- **cursorIndex**（在 TUI 消费方 `UnifiedThreadView` 中）：箭头键移动的**高亮位置**，用户可以上下浏览所有扁平行。

```typescript
// TUI 消费方中的分离模式
// 来源：packages/tui/src/components/UnifiedThreadView.tsx#L25-L26
const [cursorIndex, setCursorIndex] = useState(0);
useEffect(() => { setCursorIndex(focusedIndex); }, [focusedIndex]);
```

箭头键仅影响 `cursorIndex`，按 **Enter** 时才把 cursorIndex 对应的帖子设为新的 focusedIndex：

```typescript
// 来源：packages/tui/src/components/UnifiedThreadView.tsx#L94-L101
if (key.upArrow || input === 'k') { setCursorIndex(i => Math.max(0, i - 1)); return; }
if (key.downArrow || input === 'j') { setCursorIndex(i => Math.min(..., i + 1)); return; }
if (key.return && cursorLine?.uri) { refreshThread(cursorLine.uri); return; }
```

按 **h** 键则跳回 `themeUri`（初始进入线程时的那条帖子），实现"回到上级"。

```
[来源](packages/tui/src/components/UnifiedThreadView.tsx#L104)
```

## usePostDetail：单帖详情 + 翻译

`usePostDetail` 的目的是**查看单条帖子的完整信息并翻译其内容**。它同样会拉取整个线程作为上下文，但聚焦在目标帖子上。

### 数据加载

跟 `useTimeline` 一样，`usePostDetail` 使用纯 Store 模式：

```

```

它调用 `client.getPostThread(uri, 3, 80)`，但只提取 `thread.thread.post`（目标帖子）作为 `store.post`。

除了原始的 `PostView`，`createPostDetailStore` 还生成一个纯文本的 `flatThread`——递归遍历线程树，把每条帖子的 `handle + text` 以缩进格式拼接成字符串。这主要用于 AI 工具的上下文传递，而非用户界面渲染。


## 使用场景对比

| 维度 | `useTimeline` | `useThread` | `usePostDetail` |
|------|--------------|-------------|-----------------|
| **数据形态** | 平铺的 `PostView[]` | 扁平化 `FlatLine[]`（含 depth） | 单条 `PostView` + 纯文本线程 |
| **分页** | cursor 无限滚动 | 按 sibling 截断（可展开） | 无分页 |
| **交互模型** | 列表浏览 + 点击进入 | 光标/焦点分离的键盘导航 | 单条帖子详情 + 动作按钮 |
| **媒体解析** | 原始 PostView | 预解析的 mediaTags/imageDetails 等 | 原始 PostView |
| **翻译** | 无 | 外部（TUI 中通过 translateText） | 内建 translate() 方法 |
| **典型场景** | 主页时间线、自定义 Feed 列表 | 完整的线程浏览（含父链+回复） | 点击帖子后的详情弹窗/页 |
| **导航出口** | 点击帖子 → `thread` 视图 | Enter 切换焦点 / h 回主题 | `viewThread` → `thread` 视图 |

三个 Hook 构成了从"信息流发现"到"深入阅读"的递进路径：用户在 `useTimeline` 提供的时间线中浏览帖子列表，点击进入 `usePostDetail` 查看详情（可翻译），再通过 `viewThread` 进入 `useThread` 的完整线程视图进行深度浏览。

也见 [导航与状态管理](导航与状态管理.md) 了解 AppView 如何在三种视图间路由跳转，以及 [AT Protocol 类型系统与工具](at-protocol-类型系统与工具.md) 了解 `PostView`、`ThreadViewPost` 等底层类型定义。
