# 36 个 AI 工具：从定义到执行

## 工具注册架构

整个工具系统建立在三个核心抽象之上，全部定义在 `packages/core/src/ai/tools.ts` 中：

```typescript
export interface ToolDefinition {
  name: string;
  description: string;
  inputSchema: {
    type: 'object';
    properties: Record<string, Record<string, unknown>>;
    required: string[];
  };
}

export type ToolHandler = (params: Record<string, unknown>, assistant?: unknown) => Promise<string>;

export interface ToolDescriptor {
  definition: ToolDefinition;
  handler: ToolHandler;
  requiresWrite: boolean;
}
```

**`ToolDescriptor` 是工具注册的最小单元**，它将三个维度的信息绑定在一起：声明（definition）、执行（handler）、权限（requiresWrite）。`createTools(client: BskyClient)` 工厂函数接收一个 [BskyClient](bskyclient-at-protocol-客户端实现.md) 实例，返回包含 36 个 `ToolDescriptor` 的数组。

`requiresWrite` 是读写分离策略的核心开关。`false` 表示只读操作——工具 handler 被调用后立即执行并返回结果；`true` 表示写入操作——handler 的执行会被挂起，等待用户通过 [AIAssistant](aiassistant-多提供者-llm-引擎.md) 的确认门批准后才真正执行。

### 确认门执行流程

确认门的实现位于 `assistant.ts`，在**非流式**（`sendMessage`）和**流式**（`sendMessageStream`）两种模式下复用同一逻辑：

```
LLM 返回 tool_call
    │
    ├─ requiresWrite === false → 立即执行 handler
    │
    └─ requiresWrite === true
         ├─ 挂起: 创建 Promise<boolean>，存入 _confirmPromise
         ├─ 等待: _waitForConfirmation() 阻塞直到用户响应
         ├─ 用户批准 → 执行 handler
         └─ 用户取消 → 返回 "User cancelled the operation."
```

核心机制是一个**可外部解析的 Promise**：

```typescript
private _confirmPromise: Promise<boolean> | null = null;
private _confirmResolve: ((v: boolean) => void) | null = null;

private async _waitForConfirmation(): Promise<boolean> {
  this._confirmPromise = new Promise<boolean>((resolve) => {
    this._confirmResolve = resolve;
  });
  return this._confirmPromise;
}

confirmAction(approved: boolean): void {
  if (this._confirmResolve) {
    this._confirmResolve(approved);
    this._confirmPromise = null;
    this._confirmResolve = null;
  }
}
```

UI 层通过 `hasPendingConfirmation` 属性检测是否有待处理的确认，调用 `confirmAction(true/false)` 完成批准或拒绝。在流式模式下，确认请求以 `{ type: 'confirmation_needed', content: desc, toolName }` 事件的形态从 generator 中 yield 出来，UI 收到后弹出确认对话框。

`buildToolDescription` 函数为每个写工具生成人类可读的描述：`create_post` 显示截断到 100 字符的帖子正文，`like`/`repost` 显示目标 URI，`follow` 显示被关注者 DID，`create_list` 显示列表名称和用途。

## 深度剖析：`create_post` 的多层复合逻辑

`create_post` 是写入工具中实现最复杂的。其 handler（第 741-823 行）包含三个可选的附加处理路径：

1. **回复（replyTo）**：首先解析目标帖子的 AT URI，获取其 `cid`。然后通过 `getPostThread` 回溯根帖（root post）——遍历 parent 链直到最顶层，确保 `reply.root` 指向原始根帖而非直接父帖。

2. **图片（images）**：接受一个图片数组，每个条目可以是 `did + cid`（来自 `extract_images_from_post`）或 `pendingImageIndex`（来自用户聊天上传）。处理流程：从 Bluesky CDN 下载原图（或从 `_userUploads` 读取）→ 上传到当前用户的 PDS（`uploadBlob`）→ 构造 `app.bsky.embed.images` 嵌入。若同时有引用帖子，则自动包装为 `app.bsky.embed.recordWithMedia`。

3. **引用（quoteUri）**：解析被引用帖子的 AT URI，获取其 `cid`，构造 `app.bsky.embed.record`。若已有图片嵌入，则两者合并为 `recordWithMedia`。

## 深度剖析：`view_image` 的双模式

`view_image` 的定位是**视觉模型的图像接口**。其 handler 接受第二个参数 `assistant`（类型为 `unknown`，运行时为 `AiAssistant` 实例），通过它访问 AI 引擎的内部状态。

### 模式一：did + cid（Bluesky 帖子图片）

```typescript
if (p.did && p.cid) {
  data = await client.downloadBlob(p.did as string, p.cid as string);
  mimeType = detectMimeType(data);
}
```

`did` 和 `cid` 来自 `extract_images_from_post` 的输出。handler 通过 `BskyClient.downloadBlob` 从 Bluesky CDN 拉取原始二进制数据，然后 `detectMimeType` 通过文件魔数（magic bytes）推断 MIME 类型——检查前 4 字节匹配 PNG、JPG 或 GIF 的特征字节序列。

### 模式二：uploadIndex（聊天上传图片）

```typescript
if (p.uploadIndex !== undefined && p.uploadIndex !== null) {
  const ai = assistant as unknown as { getUserUpload?: ... };
  const upload = ai.getUserUpload?.(p.uploadIndex as number);
  // ...
}
```

`uploadIndex` 指向 `AiAssistant._userUploads` 数组中的某一项。这个数组由用户在聊天界面直接上传的图片填充（通过 `addUserUpload` 方法），**不经过 Bluesky PDS**，纯本地缓存，用户关闭对话后会被 `clearUserUploads` 清除。

### 数据传递通道

两种模式下，最终二进制数据都被转为 base64 data URL，通过 `assistant.addPendingImage` 存入 `_pendingImages` 队列：

```typescript
const base64 = toBase64(data);
const dataUrl = `data:${mimeType};base64,${base64}`;
ai.addPendingImage?.(dataUrl, alt);
```

`toBase64` 函数（第 50-55 行）实现了跨平台 base64 编码：Node.js 环境下使用 `Buffer.from(data).toString('base64')`，浏览器环境下使用 `btoa`。

`_pendingImages` 队列随后被注入到下一轮 LLM 请求的消息列表中（assistant.ts 第 337-357 行），前提是 `visionEnabled` 配置为 `true`。若 Vision 模式未开启，工具仍然可以正常调用返回元数据（mimeType、size、alt），但返回的 `note` 字段会提示"Vision mode is OFF"。

### 为什么 `required: []`

`view_image` 的 `inputSchema` 中 `required: []` —— 两个参数都可选。这要求 LLM 必须根据上下文自主决定使用哪种模式。工具描述字符串承担了引导作用：
> "Use uploadIndex to view images uploaded by the user in the chat. Use did+cid to view images from Bluesky posts."

## 深度剖析：`download_image` 的文件系统回退策略

`download_image` 与前者的关键区别在于：**它不传递图片给 LLM，而是持久化到用户文件系统**。

### 一级策略：Node.js `fs.writeFileSync`

```typescript
try {
  const { writeFileSync, mkdirSync } = await import('fs');
  const { homedir } = await import('os');
  const { join } = require('path');  // 实际代码使用 import
  const downloadsDir = join(homedir(), 'Downloads');
  mkdirSync(downloadsDir, { recursive: true });
  const filepath = join(downloadsDir, filename);
  writeFileSync(filepath, data);
  return JSON.stringify({ saved: filepath, size, mimeType });
} catch {
  // fallback
}
```

使用**动态 `import()`**（而非顶层 `import`），使得该工具在浏览器/PWA 环境中不会因 `fs` 模块不存在而崩溃。`catch` 块中的回退在浏览器中必然触发——这正是设计意图。

### 二级策略：base64 data URL

```typescript
catch {
  const base64 = toBase64(data);
  const dataUrl = `data:${detectMimeType(data)};base64,${base64}`;
  return JSON.stringify({
    saved: false,
    dataUrl,
    filename,
    size,
    mimeType,
    note: 'Image data returned as data URL. In PWA/browser, the frontend can render this as a download link.',
  });
}
```

在这个分支中，图片通过 JSON 响应中的 `dataUrl` 字段返回给 AI，由 AI 在最终回复中渲染为一个可点击的下载链接。PWA 前端利用 `download` 属性触发浏览器原生下载对话框。

双策略使得同一个工具在 TUI（终端）和 PWA（浏览器）两种环境下都能提供一致的"保存图片"体验，无需针对运行环境编写两套工具。

## 辅助函数与基础设施

`createTools` 之外，`tools.ts` 还定义了三个关键辅助函数，它们被多个工具的 handler 共享：

- **`detectMimeType`**：通过魔数识别 PNG/JPG/GIF，确保不依赖文件扩展名或外部库
- **`toBase64`**：跨平台 Node.js/Browser base64 编码，在 node 环境使用 `Buffer`，浏览器使用 `btoa`
- **`extractRkey`**：从 AT URI 中提取最后一个路径段作为 record key
- **`flattenThread`**：将嵌套的 `ThreadViewPost` 树压平为带 `depth:N` 前缀的可读文本，支持父链回溯（负深度）、同层排序、折叠提示
- **`extractTitle`**：从 Markdown 文本中提取标题（h1 > h2 > title 标签 > 首行）
