# 流式输出与思考模式

## 问题背景

非流式 (`sendMessage`) 与流式 (`sendMessageStreaming`) 的核心差异在于**交互时序**。前者等待完整响应后一次性返回，用户全程无感知；后者将 LLM 响应拆解为多个事件逐帧推送，使 UI 层可以实现**打字机效果**、**实时推理过程展示**和**工具调用中间状态反馈**。

整个流式系统的设计围绕一个核心抽象：**`AsyncGenerator` yield 类型系统**。

---

## Yield 类型系统

`sendMessageStreaming` 的签名定义了一个 5 元事件枚举：

```typescript
async *sendMessageStreaming(content: string, signal?: AbortSignal): AsyncGenerator<{
  type: 'tool_call' | 'tool_result' | 'token' | 'done' | 'thinking';
  content: string;
  toolName?: string;
}>
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L382-L386)

这五种事件类型覆盖了一次 AI 对话的完整生命周期：

| 类型 | 触发时机 | content 含义 | 消费方行为 |
|---|---|---|---|
| `thinking` | 收到 `delta.reasoning_content` | 推理文本片段 | 逐帧追加到"思考中"消息块 |
| `token` | 收到 `delta.content` | 输出文本片段 | 逐帧追加到助手回复消息块 |
| `tool_call` | 工具调用累积完成后 | `函数名(参数JSON)` | 插入工具调用气泡 |
| `tool_result` | 工具执行完成 | 执行结果摘要 | 插入工具结果气泡 |
| `done` | 全部响应结束或中断 | 完整文本内容 | 触发状态切换 |

其中 `toolName` 是可选字段，仅在 `tool_call` 和 `tool_result` 事件中携带，供消费方做类型化渲染。

## SSE 数据流解析

流式请求通过 HTTP POST 将 `stream: true` 发送到 LLM API，返回一个 `ReadableStream`。解析逻辑位于方法内部的 `while (true)` 循环中：

```
原始字节流 → TextDecoder(stream:true) → 按 \n 分行 → data: 前缀过滤 → JSON.parse → delta 提取
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L444-L479)

关键解析细节：

1. **逐帧解码**：`TextDecoder.decode(value, { stream: true })` 确保多字节字符在跨 chunk 边界时不被截断。
2. **行过滤**：只处理以 `data: ` 开头的行，跳过所有 SSE 注释行和空行。
3. **终止标记**：遇到 `data: [DONE]` 时直接跳过，不触发任何 yield。
4. **容错**：`try { JSON.parse } catch { /* skip */ }` 静默跳过任何无法解析的 chunk，保证不因单帧异常而中断整个流。

消费方（`useAIChat` 中的 `for await...of`）同步消费这些事件。对于 `token` 事件，它维护一个 `streamingContent` 累加器，每收到新 token 就更新最后一条 assistant 消息：

```typescript
// 来自 useAIChat.ts
if (event.type === 'token') {
  streamingContent += event.content;
  setMessages(prev => {
    const last = prev[prev.length - 1];
    if (last?.role === 'assistant') {
      const updated = [...prev];
      updated[updated.length - 1] = { ...last, content: streamingContent };
      return updated;
    }
    return [...prev, { role: 'assistant', content: streamingContent }];
  });
}
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/app/src/hooks/useAIChat.ts#L249-L261)

## Tool Calls 流式累积

工具调用在流式模式下以**分片**方式到达：LLM 的每个 delta chunk 可能只携带 `tool_calls[0].function.arguments` 中的一个片段，也可能在不同 chunk 中逐步填充 `id`、`name` 和 `arguments`。

核心算法使用 `Map<index, accumulator>` 按工具调用索引进行合并：

```typescript
let toolCallAccum: Map<number, { id: string; name: string; arguments: string }> = new Map();

if (delta.tool_calls) {
  for (const tc of delta.tool_calls) {
    const idx = tc.index;
    if (!toolCallAccum.has(idx)) {
      toolCallAccum.set(idx, { id: tc.id || '', name: tc.function?.name || '', arguments: '' });
    }
    const acc = toolCallAccum.get(idx)!;
    if (tc.id) acc.id = tc.id;
    if (tc.function?.name) acc.name = tc.function.name;
    if (tc.function?.arguments) acc.arguments += tc.function.arguments;
  }
}
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L467-L476)

### 合并规则

| 字段 | 策略 |
|---|---|
| `id` | 首次出现即锁定，后续 delta 中的空串不覆盖 |
| `name` | 同 id，首次非空值锁定 |
| `arguments` | **追加累加**（`+=`），因为参数 JSON 是逐字符流式到达的 |

流结束后，将 `toolCallAccum` 按 index 排序转换为标准 `ToolCall[]` 数组，然后推入消息历史，进入工具执行循环。这与非流式的 `sendMessage` 共享同一套 `_waitForConfirmation` 写操作门控逻辑。

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L499-L507)

## Thinking 模式：推理内容分离

当 `config.thinkingEnabled` 为 `true`（默认）时，请求体包含 `thinking: { type: 'enabled' }`。支持此能力的 LLM（如 DeepSeek 的 R1 系列）会在 SSE 流中额外发送 `delta.reasoning_content` 字段。

系统的处理策略是**双轨累加 + 独立事件**：

1. `fullContent` 只累积 `delta.content`（可见输出）
2. `reasoningContent` 独立累积 `delta.reasoning_content`（推理过程）
3. 每一帧的 `reasoning_content` 立即 yield 为 `type: 'thinking'` 事件

```typescript
if (delta.reasoning_content) {
  reasoningContent += delta.reasoning_content;
  yield { type: 'thinking', content: delta.reasoning_content as string };
}
if (delta.content) {
  fullContent += delta.content;
  yield { type: 'token', content: delta.content };
}
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L460-L465)

消费方 UI 层（`useAIChat`）将 thinking 事件渲染为独立的消息块。关键逻辑是**增量合并**：如果最后一条消息已经是 `thinking` 类型，则在其 `content` 上追加；否则新建一条：

```typescript
if (event.type === 'thinking') {
  setMessages(prev => {
    const last = prev[prev.length - 1];
    if (last?.role === 'thinking') {
      const updated = [...prev];
      updated[updated.length - 1] = { role: 'thinking', content: last.content + event.content };
      return updated;
    }
    return [...prev, { role: 'thinking', content: event.content }];
  });
}
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/app/src/hooks/useAIChat.ts#L237-L245)

这意味着 UI 可以做到**边推理边展示**，而不需要等待完整推理结束。`reasoning_content` 最终也会写入 `ChatMessage.reasoning_content` 字段，随消息历史持久化。

## AbortSignal 中断机制

中断机制贯穿整个流式请求的生命周期，处理三个不同阶段的取消：

### 阶段 1：HTTP 请求发起前/中

```typescript
res = await fetch(url, {
  ...,
  signal,  // 传入 AbortSignal
});
```

如果用户在请求尚未完成时点击"停止"，`ctrl.abort()` 会导致 `fetch` 抛出 `AbortError`。捕获后，系统不是简单抛异常，而是 yield 一个 `done` 事件携带 `[已暂停]` 标记：

```typescript
catch (e) {
  if (signal?.aborted) {
    yield { type: 'done', content: '\n\n[已暂停]' };
    return;
  }
  throw e; // 网络错误等非中断异常继续向上传播
}
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L431-L436)

### 阶段 2：SSE 流读取中

在读流循环的每一轮迭代开始时检查 `signal?.aborted`：

```typescript
while (true) {
  if (signal?.aborted) {
    yield { type: 'done', content: '\n\n[已暂停]' };
    break;
  }
  const { done, value } = await reader.read();
  ...
}
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L445-L449)

### 阶段 3：reader.read() 挂起中

`reader.read()` 本身是异步的，可能在等待网络数据时被中断。这种情况通过外层 `try/catch` 处理：

```typescript
try {
  while (true) { ... }
} catch (_err) {
  if (signal?.aborted) {
    yield { type: 'done', content: fullContent };  // 保留已收到的内容
    return;
  }
  throw _err;
}
```

[来源](https://github.com/your-repo/bsky/blob/main/packages/core/src/ai/assistant.ts#L480-L487)

注意这里 `content` 是 `fullContent`（已累积的完整内容），而不是硬编码的 `[已暂停]`，因为此时可能已经有部分 token 被成功消费。

### 三个阶段的中断语义对比

| 阶段 | 中断时 `content` | 处理方式 |
|---|---|---|
| HTTP 请求 | `\n\n[已暂停]` | 无任何内容已到达 |
| 流循环检测 | `\n\n[已暂停]` | 本轮尚未调用 `reader.read()` |
| 流读取挂起中 | `fullContent` | 已有部分内容成功累积 |

## 设计要点总结

1. **AsyncGenerator 作为双向通信通道**：yield 类型系统定义了 5 种事件的契约，消费方通过 `for await...of` 获得完全的类型安全。

2. **SSE 按行解析 + 容错**：只处理 `data: ` 前缀行，JSON 解析失败静默跳过，保证单帧异常不破坏整个流。

3. **工具调用按索引合并**：使用 `Map<index, accumulator>` 解决流式 `tool_calls` 的碎片化问题，`arguments` 字段采用追加策略。

4. **推理流分离展示**：`reasoning_content` 与 `content` 双轨独立累积，yield 为独立事件类型，UI 层通过角色状态机实现增量渲染。

5. **三段式中断防护**：在 HTTP 请求、流循环、reader 读取三处设防，确保用户在任何时刻停止都能获得恰当的反馈。
