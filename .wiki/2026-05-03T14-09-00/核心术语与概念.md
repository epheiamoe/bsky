# 核心术语与概念

这个页面帮你建立两套词汇表：左边是 **Bluesky / AT Protocol 协议层术语**，右边是 **本项目代码中的内部概念**。每个词都配一句人话解释。

---

## AT Protocol 生态术语

这些是 Bluesky 去中心化社交协议的核心概念。理解它们，就理解了数据怎么来、往哪去。

### DID（分散式标识符）

**人话：** 每个用户在 Bluesky 上的永久身份证号，换服务器也不变。

格式为 `did:plc:xxxxxxxxxxxx`。你注册 Bluesky 时，系统会自动分配一个 DID，它绑定你的所有数据。DID 是协议层面的**真实主键**，Handle（用户名）只是便于人类记忆的别名。

> 在 `types.ts` 中，`ProfileViewBasic` 和 `PostView` 等所有核心类型的第一个字段都是 `did: string`。
>
> [来源](../packages/core/src/at/types.ts#L65-L70)

### Handle（句柄）

**人话：** 你在 Bluesky 上显示的用户名，比如 `@yourname.bsky.social`。

Handle 可以变更（比如绑定自己的域名），而 DID 永远不变。API 接受 Handle 作为参数，通过 `com.atproto.identity.resolveHandle` 将其解析为 DID。`ProfileViewBasic.handle` 就是存储它的字段。

> [来源](../packages/core/src/at/types.ts#L66)

### AT URI

**人话：** AT Protocol 里每个资源（帖子、图片、用户等）的统一地址，类似 `https://` 在 Web 中的地位。

格式为 `at://<DID>/<Collection>/<RKey>`，例如：

```
at://did:plc:abc123/app.bsky.feed.post/3lkxyz789
```

`parseAtUri()` 函数按正则 `^at:\/\/(did:plc:[^/]+)\/([^/]+)\/([^/]+)$` 将其拆解为 **DID**、**Collection**、**RKey** 三部分。

```
at://did:plc:abc123/app.bsky.feed.post/3lkxyz789
 └── DID ──┘└──── Collection ────┘└── RKey ──┘
```

> [来源](../packages/core/src/at/types.ts#L1-L16)

### Collection（集合）

**人话：** 资源类型的命名空间，类似数据库中的"表名"。

每个 AT URI 的第二段就是 Collection，例如：

| Collection | 含义 |
|---|---|
| `app.bsky.feed.post` | 帖子 |
| `app.bsky.feed.like` | 点赞 |
| `app.bsky.graph.follow` | 关注关系 |
| `app.bsky.actor.profile` | 用户资料 |

### Record（记录）

**人话：** AT Protocol 里的一行数据，比如你写的一条帖子正文。

`PostRecord` 接口定义了帖子的核心字段：`text`（正文）、`createdAt`（创建时间）、`embed`（嵌入内容，如图片/视频/引用）、`facets`（富文本标记）、`reply`（回复指向）。

```typescript
export interface PostRecord {
  text: string;
  createdAt: string;
  embed?: ImageEmbed | ExternalEmbed | RecordEmbed | RecordWithMediaEmbed | VideoEmbed;
  facets?: Facet[];
  reply?: { root: {...}; parent: {...} };
}
```

> [来源](../packages/core/src/at/types.ts#L18-L24)

### PDS（个人数据服务器）

**人话：** 存放你所有 Bluesky 数据（帖子、点赞、关注列表）的服务器。

PDS 是 AT Protocol 去中心化的核心——你可以选择由官方托管，也可以自建。所有 CRUD 操作（发帖、删帖等）都发送到你的 PDS。

### PD（Personal Data，个人数据）

**人话：** 你的 PDS 上存储的所有你产生的数据。

PD 是账户所有权的证明，确保你换 PDS 时可以带走全部数据。

### Feed Generator（数据源生成器）

**人话：** 按某种算法筛选帖子的"推荐流服务器"，比如发现热门帖子、关注的话题等。

项目内置的 `BUILTIN_FEEDS` 包含 `following`（关注流）和 `discover`（发现流）。用户也可以在设置中配置自定义 Feed Generator。

> [来源](../packages/pwa/src/hooks/useHashRouter.ts#L11-L12) — 路由中对 Feed 的处理可见 `getFeedConfig().defaultFeedUri`

### ThreadViewPost（讨论串视图帖子节点）

**人话：** 一个帖子在讨论串树形结构中的节点，包含其父帖子和子回复。

```typescript
export interface ThreadViewPost {
  post: PostView;
  parent?: ThreadViewPost | NotFoundPost;   // 指向父帖子
  replies?: Array<ThreadViewPost | NotFoundPost>; // 子回复列表
}
```

它形成了递归树结构：每个节点知道自己的父帖子和子回复，`flattenThreadTree()` 函数将其展平为线性列表供 UI 渲染。

> [来源](../packages/core/src/at/types.ts#L87-L91)

## 本项目内部概念

这些术语不来自 Bluesky 协议，而是本项目代码中为解决问题的能力。

### FlatLine（扁平帖子行）

**人话：** 把树形讨论串"拍扁"成一行行的数据，方便 UI 逐行渲染。

`flattenThreadTree()` 从 `ThreadViewPost` 树中递归遍历，生成 `FlatLine[]` 数组。每行包含：帖子正文、作者 Handle/昵称、深度标记（`depth`）、媒体标签、图片 URL、引用帖子等。

深度规则：主题帖 `depth=0`，回复 `depth>0`（越大越深），讨论源 `depth<0`（主题帖上方的上下文）。

```typescript
export interface FlatLine {
  depth: number;             // 0=主题帖, 正=回复, 负=讨论源
  uri: string;               // 帖子 AT URI
  text: string;              // 帖子正文
  handle: string;            // 作者用户名
  displayName: string;       // 作者显示名
  imageUrls: string[];       // 图片列表
  quotedPost?: { ... };      // 引用的帖子
  isRoot: boolean;           // 是否主题帖
  isTruncation: boolean;     // 是否"还有N条未显示"
  // ... 更多字段
}
```

> [来源](../packages/app/src/hooks/useThread.ts#L7-L40)

### AIChatMessage（AI 聊天消息）

**人话：** AI 对话中的一条消息——用户说的、AI 回的、工具的调用记录都算。

```typescript
export interface AIChatMessage {
  role: 'user' | 'assistant' | 'tool_call' | 'tool_result' | 'thinking';
  content: string;
  toolName?: string;   // 如果是工具调用，记录工具名
  isError?: boolean;   // 工具执行是否出错
}
```

五种角色对应 AI 对话的不同片段：`user`（用户输入）、`assistant`（AI 回复）、`tool_call`（AI 决定调用的工具）、`tool_result`（工具执行结果）、`thinking`（AI 的推理过程）。

> [来源](../packages/app/src/services/chatStorage.ts#L5-L9)

### ChatStorage 接口（聊天存储抽象）

**人话：** 存聊天记录的"接口合同"——不管存在文件里还是浏览器里，都按这个规矩来。

```typescript
export interface ChatStorage {
  saveChat(chat: ChatRecord): Promise<void>;
  loadChat(id: string): Promise<ChatRecord | null>;
  listChats(): Promise<ChatSummary[]>;
  deleteChat(id: string): Promise<void>;
}
```

TUI 环境有 `FileChatStorage`（存在 `~/.bsky-tui/chats/` 目录下的 JSON 文件），PWA 环境有 `IndexedDBChatStorage`（存在浏览器的 IndexedDB 中）。双实现靠同一个接口保证了上层代码的一致性。

> [来源](../packages/app/src/services/chatStorage.ts#L29-L34)
> 详细讨论见[聊天记录存储方案](聊天记录存储方案.md)

### Viewport 渲染（视口渲染）

**人话：** 终端的屏幕有限，只渲染能看到的那几行，别一次性画几千行。

在 TUI 的 `AIChatView.tsx` 中，通过 `rows`（终端行数）和 `cols`（终端列数）动态计算可见区域：

```typescript
const maxCols = Math.max(20, cols - 6);
const baseMaxVisible = Math.max(10, rows - 6);
```

这种"只渲染视口内内容"的策略避免了 Ink 在长对话场景下的性能瓶颈。

> [来源](../packages/tui/src/components/AIChatView.tsx#L62-L64)

### useHashRouter（Hash 路由钩子）

**人话：** PWA 版的"导航控制器"——通过 URL 里的 `#/feed`、`#/thread?uri=...` 等 Hash 切换页面。

PWA 部署在静态托管上，没有服务端路由，只能靠 URL Hash 做导航。`useHashRouter()` 返回四个核心导航方法：

| 返回值 | 用途 |
|---|---|
| `currentView` | 当前视图对象（如 `{type:'feed', feedUri:'...'}`） |
| `goTo(view)` | 跳转到新视图（推入历史栈） |
| `goBack()` | 回退（调用 `history.back()`） |
| `goHome()` | 回到主页（`#/feed`） |

支持的 Hash 路径包括：`/feed`、`/thread`、`/profile`、`/notifications`、`/search`、`/bookmarks`、`/compose`、`/ai`。所有参数都通过 query string 传递并编解码。

> [来源](../packages/pwa/src/hooks/useHashRouter.ts#L20-L48)
> 详细讨论见[状态管理与路由系统](状态管理与路由系统.md)
