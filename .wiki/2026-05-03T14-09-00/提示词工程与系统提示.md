# 提示词工程与系统提示

所有与大语言模型交互的文本—从助手角色定义到翻译指令、从上下文注入到润色约束—集中在 `packages/core/src/ai/prompts.ts` 一个文件中。这是整个项目中唯一维护提示词的地方。TUI 与 PWA 两端共享同一份提示定义，任何行为变更只需修改此文件并重新构建。

## 设计哲学：单点维护

`prompts.ts` 顶部注释明确了此约定：

> Centralized AI prompts — single source of truth for all LLM-facing text.
> To customize AI behavior, edit this file and rebuild.

[来源](prompts.ts#L1-L5)

两条命名规则贯彻始终：

| 前缀 | 语义 | 示例 |
|------|------|------|
| `P_` | 纯字符串常量或模板 | `P_ASSISTANT_BASE`, `P_CONCISE`, `P_POLISH_SYSTEM` |
| `PF_` | 参数化函数，运行时注入上下文 | `PF_CURRENT_USER(name, handle)`, `PF_ENVIRONMENT(env)` |

`buildSystemPrompt()` 函数（位于 `packages/app/src/hooks/useAIChat.ts`）将各片段按固定顺序拼接，构成最终发送给 LLM 的 `system` 消息：

```
P_ASSISTANT_BASE
  → PF_CURRENT_USER (若有)
    → PF_PROFILE_CONTEXT 或 PF_POST_CONTEXT (场景相关)
      → PF_ENVIRONMENT
        → PF_LOCALE_HINT (若有)
          → PF_CURRENT_TIME
            → PF_VISION_HINT
              → P_CONCISE
```

[来源](useAIChat.ts#L55-L81)

## `P_ASSISTANT_BASE` — 核心角色与行为约束

这是系统提示的基座，定义了 AI 助手的**身份、能力边界与禁止行为**。内容分为三个逻辑层：

**角色定义（前 4 行）**：助手被定位为"Bluesky 助手"，职责是"浏览和分析 Bluesky 上的内容"。同时内嵌了 `search_posts` 高级搜索语法的文档（`from:`、`to:`、`since:`、`lang:` 等 Lucene 运算符），让 LLM 理解工具参数之外的能力。

**媒体处理规则（后 2 段）**：
- **图片**：用户上传的图片以 `pendingImageIndex` 索引引用，支持多图。`download_image` 工具可将图片保存到用户本地。
- **视频**：Bluesky 帖子的视频以 `[视频]` 标记显示。AI **不能**分析视频内容，收到视频帖子时必须直接告知能力限制，不得调用 `extract_images_from_post` 或 `view_image`。

**三条不可违反的规则（末尾）**：
1. 禁止主动代表用户执行写操作（发帖、点赞、转发、关注）。所有写操作必须由用户明确要求。
2. 汇总资料时直接输出分析，不得附加"帮你发条帖子"之类的建议。
3. 除非用户明确要求发帖，否则永远不通过 `create_post` 执行。

[来源](prompts.ts#L31-L62)

## 上下文注入函数（`PF_` 前缀）

### `PF_CURRENT_USER` — 身份锚点

注入当前用户的显示名（或 handle），让助手知道在与谁对话。可选 handle 参数追加 `@handle` 后缀：

```
当前用户: Alice (@alice.bsky.social)。
```

[来源](prompts.ts#L66-L69)

### `PF_PROFILE_CONTEXT` — 主页场景

当用户从某个 Bluesky 主页打开 AI Chat 时触发。此提示指导执行三步流程：
1. 调用 `get_author_feed` 查看该用户的近期帖子
2. 使用 `search_posts from:{currentUser} to:{handle}` 查找互动历史
3. 概括至少 3 个要点，引用至少一则帖子

末尾强调「仅分析，不要代表用户发帖或互动」。[来源](prompts.ts#L76-L89)

当存在 `contextProfile` 时，`buildSystemPrompt` 优先使用 `PF_PROFILE_CONTEXT`，跳过 `PF_POST_CONTEXT`。[来源](useAIChat.ts#L67-L73)

### `PF_POST_CONTEXT` — 帖子场景

当用户从某个帖子或讨论串打开 AI Chat 时触发。仅注入帖子的 AT URI，具体上下文获取由助手通过工具调用自行完成：

```
用户正在查看帖子 at://did:plc:xxx/app.bsky.feed.post/yyy，如果需要请用工具获取上下文。
```

[来源](prompts.ts#L92-L95)

### `PF_ENVIRONMENT` — 运行时适配

关键的分支提示。根据 `env` 参数生成两套截然不同的指令：

- **TUI 终端**：纯文本输出，每行不超过 80 字符，不支持图片内嵌，但支持 OSC 8 超链接。
- **PWA 浏览器**：支持图片、Markdown 格式和超链接。

这让同一份 AI 引擎可以适配完全不同的渲染介质。[来源](prompts.ts#L100-L105)

### `PF_LOCALE_HINT` — 语言偏好

将 UI 语言代码（`zh`/`en`/`ja` 等）传递给 LLM，要求优先以此语言回复。与项目 [国际化 (i18n) 系统设计](国际化-i18n-系统设计.md) 一致，实现语言感知的对话。

[来源](prompts.ts#L108-L110)

### `PF_CURRENT_TIME` — 时间感知

读取系统时钟生成当前 UTC 时间和星期几，格式为 `2025-01-15 10:30:00 (UTC+0)，星期三。`。注意：时间只在调用 `buildSystemPrompt()` 时定格，不会流式更新。

[来源](prompts.ts#L119-L123)

### `PF_VISION_HINT` — 视觉模式开关

根据用户设置（`LLM_VISION_ENABLED`）生成两条完全不同长度的提示：

- **开启**：简短授权—允许使用 `view_image` 和 `download_image`。
- **关闭**：详细引导—告知如何开启视觉模式，同时附加警告：如果模型本身不支持视觉（如纯文本模型），不应建议用户开启以免浪费上下文。视觉模式不提供外置 OCR。

[来源](prompts.ts#L128-L140)

## 翻译双模式

翻译系统使用两个函数实现两种输出格式：

| 函数 | 模式 | 输出格式 | 用途 |
|------|------|----------|------|
| `PF_TRANSLATE_SIMPLE` | 简单模式 | 纯翻译文本，无解释 | 快速翻译 |
| `PF_TRANSLATE_JSON` | JSON 模式 | `{"source_lang":"en","translated":"..."}` | 需要源语言检测的场景 |

`assistant.ts` 中的翻译分支据此二选一：

```typescript
const systemPrompt = mode === 'json'
  ? PF_TRANSLATE_JSON(langLabel)
  : PF_TRANSLATE_SIMPLE(langLabel);
```

[来源](assistant.ts#L667-L669), [prompts.ts#L143-L159]

JSON 模式额外要求输出必须是合法的 JSON 对象，且包含 `"json"` 关键词—这是应对某些 LLM 拒绝输出纯 JSON 的防御性设计。

## `P_POLISH_SYSTEM` — 润色专用提示

润色功能拥有独立的系统提示，与主助手完全解耦：

```
你是一个文字润色助手，根据用户要求调整以下帖子草稿，只返回润色后的文本。
```

配合 `PF_POLISH_USER(requirement, draft)` 构成完整的润色请求模板。润色提示不继承任何主助手的工具调用能力或上下文信息，是纯文本到文本的转换。[来源](prompts.ts#L165-L170)

关于润色流程的完整描述，参见 [翻译与润色功能](翻译与润色功能.md)。

## 辅助片段

- **`P_CONCISE`** — 仅 4 字："回答简练。" 作为所有助手提示的收尾，对抗 LLM 的冗长倾向。[来源](prompts.ts#L113)
- **`PF_AUTO_ANALYSIS`** — 从主页打开 AI Chat 时自动发送的首条用户消息，内容为 `请分析 @{handle} 的主页，概括他们的近期动态。`。[来源](prompts.ts#L180-L183)
- **`P_GUIDING_QUESTIONS`** — 三条 UI 引导问题（"总结这个讨论""解释这个讨论""分析帖子情绪"），由前端渲染为快捷按钮，仅影响用户体验，不注入 LLM。[来源](prompts.ts#L187-L191)
