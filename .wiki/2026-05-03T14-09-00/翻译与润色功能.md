# 翻译与润色功能

作为一款深度集成 AI 的双界面客户端，项目将翻译和草稿润色抽象为两个**无工具调用（tool-less）的单轮 AI 调用**。它们不走 `AIAssistant.sendMessage` 的多轮工具循环，而是直连 LLM API，在纯文本输出与结构化 JSON 输出之间提供切换能力。

## 翻译：双模式设计

翻译功能的核心入口是 `translateText` 函数，位于 `packages/core/src/ai/assistant.ts`。它接受源文本、目标语言和模式参数，返回统一的 `TranslationResult` 结构。

```typescript
// TranslationResult 接口
interface TranslationResult {
  translated: string;
  sourceLang?: string; // ISO 639-1 代码，仅 JSON 模式返回
}
```

[来源](core/src/ai/assistant.ts#L669-L673)

### Simple 模式 vs JSON 模式

两种模式的核心差异在于**输出格式**和**额外信息**。

| 维度 | Simple 模式 | JSON 模式 |
|---|---|---|
| 输出格式 | 纯文本，只返回翻译结果 | 结构化 JSON `{"translated": "...", "source_lang": "..."}` |
| 源语言检测 | 不检测 | 自动检测并返回 ISO 639-1 代码 |
| 适用场景 | 快速翻译、向后兼容 | 需要知道原文语言的分析场景 |
| `response_format` | 不设置 | 设置 `response_format: { type: 'json_object' }` |
| 提示词 | `PF_TRANSLATE_SIMPLE` | `PF_TRANSLATE_JSON` |

**Simple 模式的提示词**简洁直接：

> Translate the following text to {targetLang}. Keep the original meaning, output only the translation, no explanations.

**JSON 模式的提示词**要求 LLM 输出可解析的 JSON：

> You are a translator. Translate the user's text to {targetLang}. Output valid JSON with these keys: {"source_lang": "...", "translated": "..."}. Output ONLY the JSON object.

[来源](core/src/ai/prompts.ts#L140-L152)

在 JSON 模式下，`translateText` 会尝试解析 LLM 返回的 JSON。如果解析失败或缺少 `translated` 字段，会触发重试；重试耗尽后回退为将原始内容作为 `translated` 返回，`sourceLang` 标记为 `'und'`。

[来源](core/src/ai/assistant.ts#L704-L733)

### 指数退避重试机制

`translateText` 内置最多 3 次重试（通过 `maxRetries` 参数控制），覆盖以下失败场景：

1. **网络错误** — fetch 抛出异常
2. **HTTP 错误** — API 返回非 2xx 状态码
3. **空内容** — 返回的 `content` 为空或仅空白字符
4. **JSON 解析失败** — 仅 JSON 模式，LLM 输出非标准 JSON

退避策略采用**递增等待时间**：每次重试前等待 `800ms × (attempt + 1)`（普通错误为 `1000ms × (attempt + 1)`）。代码通过 `continue` 跳过当前循环进入下一轮。

[来源](core/src/ai/assistant.ts#L686-L748)

### 便捷函数

`translateToChinese` 是对 `translateText` 的语法糖，固定目标语言为 `zh`、模式为 `simple`：

```typescript
export async function translateToChinese(config: AIConfig, text: string): Promise<string> {
  const result = await translateText(config, text, 'zh', 'simple');
  return result.translated;
}
```

## 润色：基于 `singleTurnAI`

润色功能 `polishDraft` 是对 `singleTurnAI` 的简单封装。`singleTurnAI` 是一个**无工具、无流式、单次请求**的 AI 调用函数，与 `AIAssistant.sendMessage` 不同，它不维护消息历史，不加载工具定义。

```typescript
export async function polishDraft(
  config: AIConfig,
  draft: string,
  requirement: string
): Promise<string> {
  return singleTurnAI(
    config,
    P_POLISH_SYSTEM,           // 系统提示
    PF_POLISH_USER(requirement, draft),  // 用户提示
    0.7,                       // temperature（比翻译更高，鼓励创造性）
    2000,                      // max_tokens
  );
}
```

[来源](core/src/ai/assistant.ts#L763-L768)

### 提示词设计

系统提示只做一件事：

> 你是一个文字润色助手，根据用户要求调整以下帖子草稿，只返回润色后的文本。

用户提示将「用户要求」和「草稿」拼接在一起：

> 用户要求：更正式
>
> 草稿：
> bluesky is cool i like it

[来源](core/src/ai/prompts.ts#L165-L171)

`temperature = 0.7` 高于翻译的 `0.3`，因为润色需要一定程度的创造性改写，而非忠实还原。

### singleTurnAI 的实现细节

`singleTurnAI` 函数与 `AIAssistant.makeRequest` 共享相同的 HTTP 调用逻辑，但有几点关键区别：

- **无消息历史** — 每次都从头构造 `system + user` 两条消息
- **无工具** — 不传递 `tools` 参数，`tool_choice` 不设置
- **非流式** — `stream: false`
- **思考模式** — 默认继承配置中的 `thinkingEnabled`

[来源](core/src/ai/assistant.ts#L625-L665)

## UI 层的触发入口

### TUI 终端界面

在 TUI 中，翻译功能通过键盘快捷键 `f` / `F` 触发，出现在两个视图中：

**帖子讨论串** (`UnifiedThreadView`)：按下 `f` 后，动态导入 `translateText`，对当前光标所在的帖子文本进行翻译。结果以 `[sourceLang→targetLang] 翻译内容` 的格式显示。

[来源](tui/src/components/UnifiedThreadView.tsx#L145-L153)

**个人主页** (`ProfileView`)：按下 `f` 后，翻译用户的简介（`profile.description`）。结果以 `[→targetLang] 翻译内容` 的格式显示，无源语言检测（固定使用 simple 模式）。

[来源](tui/src/components/ProfileView.tsx#L80-L88)

TUI 中的翻译模式与目标语言由全局配置传入，无运行时切换 UI。

### PWA 网页应用

PWA 的翻译入口以**图标按钮**的形式嵌入在帖子操作栏和简介区域。

**帖子视图** (`ThreadView`)：每个帖子右侧的 🌐 语言图标按钮。点击后调用 `useTranslation` Hook，显示翻译结果及源语言检测信息（JSON 模式下）。再次点击切换回原文。翻译结果在切换焦点帖子时自动清除。

[来源](pwa/src/components/ThreadView.tsx#L80-L90)

**个人主页** (`ProfilePage`)：简介区域旁的「翻译」按钮。点击后翻译个人简介，按钮文字变为「原文」可切回。

[来源](pwa/src/components/ProfilePage.tsx#L300-L306)

**设置面板** (`SettingsModal`)：用户可以通过下拉菜单在 `simple` / `json` 模式间切换，并选择目标语言。这些配置通过 `useAppConfig` 持久化。

## `useTranslation` Hook

`packages/app/src/hooks/useTranslation.ts` 提供了一个 React Hook，封装了翻译功能的调用逻辑：

- **缓存** — 使用 `Map<string, TranslationResult>` 缓存翻译结果，缓存键为 `${mode}::${lang}::${text}`，避免重复请求
- **惰性加载** — 运行时动态 `import('@bsky/core')`，不增加初始包体积
- **状态管理** — 暴露 `loading`、`lang`、`mode` 状态及其 setter

## 草稿系统（Draft）的完整生命周期

翻译和润色的结果最终可能进入发帖流程，而草稿系统（Draft）是发帖前的暂存机制。

### Draft 数据结构

```typescript
interface Draft {
  id: string;
  text: string;
  replyTo?: string;   // 回复目标 URI
  quoteUri?: string;   // 引用帖子 URI
  createdAt: string;   // ISO 8601
  updatedAt: string;   // ISO 8601
}
```

[来源](app/src/hooks/useCompose.ts#L13-L21)

### 存储实现：`createDraftsStore`

草稿存储是一个**内存数组**实现，不涉及 IndexedDB 或文件系统：

- **`saveDraft`** — 根据 `id` 查找已有草稿。存在则更新 `text`/`replyTo`/`quoteUri` 并刷新 `updatedAt`；不存在则追加，设置 `createdAt` 和 `updatedAt`
- **`loadDraft`** — 通过 `id` 查找
- **`deleteDraft`** — 按 `id` 过滤移除

[来源](app/src/hooks/useDrafts.ts#L11-L32)

### TUI 中的草稿交互

在 TUI 的 compose 模式下：

1. **保存** — 按 `Esc` 退出编辑时，如果内容非空，弹出 `y/n/esc` 确认提示。确认后调用 `saveDraft`
2. **载入** — 在编辑模式按 `D` 键打开草稿列表，使用 `j/k` 上下移动，`Enter` 载入选中草稿
3. **删除** — 在草稿列表中按 `d` 删除当前选中草稿
4. **新建** — 在草稿列表中按 `n` 返回空白编辑

[来源](tui/src/components/App.tsx#L148-L230)

### PWA 中的草稿交互

在 PWA 的 `ComposePage` 中：

1. **保存** — 点击「取消」按钮时，如果内容非空，弹出 `confirm` 对话框询问是否保存
2. **载入** — 点击「Drafts」按钮展开草稿面板，每个草稿旁有「Load」按钮
3. **删除** — 每个草稿旁的「Delete」按钮
4. **显示** — 侧边栏（Sidebar）显示草稿计数角标，`draftCount` 从 `useDrafts` 获取

[来源](pwa/src/components/ComposePage.tsx#L179-L191)
