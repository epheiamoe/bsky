# 导航与状态管理

整个应用只有一个导航问题需要回答：**用户当前在哪个视图，以及如何切换**。答案由一个 **纯状态机** 和两个 **UI 层适配器** 组成——TUI 将状态保存在内存中，PWA 将其编码到 URL 哈希里。

---

## AppView：一个联合类型，16 种视图

导航系统的核心是 `AppView` 联合类型，它枚举了应用中所有可能的页面：

```typescript
export type AppView =
  | { type: 'feed'; feedUri?: string }
  | { type: 'detail'; uri: string }
  | { type: 'thread'; uri: string }
  | { type: 'compose'; replyTo?: string; quoteUri?: string; draftId?: string }
  | { type: 'profile'; actor: string; profileTab?: string }
  | { type: 'notifications' }
  | { type: 'search'; query?: string; searchTab?: string }
  | { type: 'aiChat'; contextUri?: string; sessionId?: string;
        contextPost?: string; contextProfile?: string }
  | { type: 'bookmarks' }
  | { type: 'drafts' }
  | { type: 'components' }
  | { type: 'dm' }
  | { type: 'dmChat'; conversationId: string }
  | { type: 'lists'; actor?: string }
  | { type: 'listDetail'; uri: string; tab?: 'posts' | 'members' }
  | { type: 'about' };
```

每种视图携带自己的参数——`thread` 需要 AT URI，`compose` 可能需要 `replyTo` 或 `quoteUri`，`aiChat` 可以携带上下文帖子或会话 ID。没有参数的视图（如 `notifications`、`bookmarks`）仅靠 `type` 区分。

## createNavigation()：纯状态机

`createNavigation()` 返回一个基于堆栈的导航控制器，不依赖任何 UI 框架。

```
┌─────────────────────────────────────┐
│          NavigationController       │
│                                     │
│  stack: AppView[]                   │
│         ↑ 初始化为 [{ type: 'feed' }] │
│                                     │
│  goTo(view)  → stack.push(view)     │
│  goBack()    → stack.pop()          │
│  goHome()    → stack = [{feed}]     │
│  canGoBack   → stack.length > 1     │
│                                     │
│  subscribe(fn) → pub/sub 通知       │
└─────────────────────────────────────┘
```

三个核心方法的行为非常直观：

- **`goTo(view)`** — 将新视图推入堆栈。每次导航都产生新数组（不可变风格）。
- **`goBack()`** — 弹出栈顶元素。如果堆栈只剩一个元素（即已在首页），则不操作。
- **`goHome()`** — 将堆栈重置为 `[{ type: 'feed' }]`，清空所有导航历史。

控制器内置发布-订阅模式：`subscribe(fn)` 注册监听器，每次堆栈变更后调用 `notify()` 遍历所有监听器。取消订阅通过返回值函数实现。

## useNavigation()：React Hook 桥接（TUI 路径）

`useNavigation()` 是连接纯状态机与 React 渲染层的桥梁：

```typescript
export function useNavigation() {
  const [nav] = useState(() => createNavigation());
  const [state, setState] = useState(() => nav.getState());
  const tick = useCallback(() => setState(nav.getState()), [nav]);
  useEffect(() => nav.subscribe(tick), [nav, tick]);
  return { currentView, canGoBack, goTo, goBack, goHome };
}
```

关键设计决策：**每个组件实例拥有自己的导航状态机**。`useState(() => createNavigation())` 确保 `nav` 只在组件挂载时创建一次；`subscribe(tick)` 在状态变更时触发 React 重渲染。返回的解构 API 与 `NavigationState` 接口完全一致。

TUI 的 `App.tsx` 正是通过这个 Hook 获取导航能力：

```typescript
const { currentView, canGoBack, goTo, goBack, goHome } = useNavigation();
```

然后在 `switch (currentView.type)` 中分发到对应的视图组件（`PostList`, `ThreadView`, `ComposeView` 等）。键盘快捷键（`n` → 通知、`a` → AI 聊天、`c` → 发帖）通过 `goTo()` 触发导航。

## useHashRouter()：PWA 哈希路由

PWA 不直接使用 `useNavigation()`——它用 `useHashRouter()` 替代，后者将相同的接口映射到浏览器 URL 哈希。

### 双向编解码

`useHashRouter()` 内部维护两个映射函数：

**`encodeView(view)`** 将 `AppView` → 哈希 URL：

| AppView                  | 哈希 URL                                  |
|--------------------------|-------------------------------------------|
| `{ type: 'feed' }`       | `#/feed`                                  |
| `{ type: 'thread', uri }`| `#/thread?uri=at://...`                   |
| `{ type: 'profile', actor, tab }`| `#/profile?actor=did:...&tab=posts` |
| `{ type: 'compose', replyTo }`| `#/compose?replyTo=at://...`         |
| `{ type: 'aiChat', sessionId }`| `#/ai?session=uuid`                |

**`parseHash()`** 将哈希 URL → `AppView`，解析 `#/path` 和 `?key=value` 参数。对于 `#/feed` 路径，如果未指定 `feed` 参数，会自动解析为默认 feed URI。

### 浏览器历史集成

三个导航方法直接操作浏览器历史：

- **`goTo(view)`** — 调用 `history.pushState(null, '', hash)`，更新当前视图，设置 `canGoBack=true`。空 feedUri 的 feed 导航会被自动解析为最后活跃的 feed 或默认 feed。
- **`goBack()`** — 调用 `window.history.back()`，依赖浏览器的 `popstate` 事件恢复状态。
- **`goHome()`** — 调用 `history.pushState` 跳转到 `#/feed?feed=<default>`，重置视图并清空返回状态。

`popstate` 事件监听器负责同步浏览器前进/后退按钮与 React 状态。

## TUI vs PWA：导航对比

| 维度 | TUI（终端） | PWA（浏览器） |
|------|-------------|---------------|
| **Hook** | `useNavigation()` 来自 `@bsky/app` | `useHashRouter()` 来自 `packages/pwa` |
| **状态位置** | 内存堆栈（`AppView[]`） | URL 哈希（`#/path?params`） |
| **后退机制** | `stack.pop()` 编程式 | `window.history.back()` 浏览器原生 |
| **页面刷新** | 状态丢失 | 从哈希恢复完整视图 |
| **静态托管** | 不适用 | `#` 路径无需服务端路由配置 |
| **初始化** | `{ type: 'feed' }` | 解析 `window.location.hash`，若无则默认 feed |

PWA 选择哈希路由而非 History API 路径的原因在架构文档中有明确说明：

> **Hash-based routing (useHashRouter)** — PWA uses `history.pushState` + `popstate` with `#/path` format for static hosting compatibility.

由于 PWA 部署在 Cloudflare Pages 等静态托管上，没有服务端路由重写支持。哈希路由确保所有页面路径在刷新时由前端解析，无需服务端参与。

## 分发模式

两个 UI 层采用相同的分发模式：通过 `switch (currentView.type)` 将当前视图映射到实际渲染组件。

PWA 的 `App.tsx` 中的核心分发：

```
currentView.type
     │
     ├── feed         → <FeedTimeline />
     ├── thread       → <ThreadView />
     ├── compose      → <ComposePage />
     ├── profile      → <ProfilePage />
     ├── notifications→ <NotifsPage />
     ├── search       → <SearchPage />
     ├── aiChat       → <AIChatPage />
     ├── bookmarks    → <BookmarkPage />
     ├── lists        → <ListsPage />
     ├── listDetail   → <ListDetailPage />
     ├── drafts       → <DraftsPage />
     ├── dm           → <ConvoListPage />
     ├── dmChat       → <DMChatPage />
     ├── components   → <ComponentsPage />
     └── about        → <AboutPage />
```

每个页面组件接收 `goTo`、`goBack` 等导航回调，以便在内部触发子导航（例如点击帖子 → `goTo({ type: 'thread', uri })`）。
