# 31 个 AI 工具详解

`createTools(client: BskyClient)` 在 `@bsky/core` 中一次性注册了 31 个工具描述符（`ToolDescriptor`），构成 AI 助手操作 Bluesky 的完整能力边界。这 31 个工具在概念上分为 **27 个只读工具**（`requiresWrite: false`）和 **4 个写入工具**（`requiresWrite: true`），后者的执行受[AI 助手系统](ai-助手与工具调用系统.md)中 **写操作确认门控** 的约束。本文从实现代码出发，逐一解析每个工具的定义、输入参数，并对 `flattenThread` 线程展平算法、`create_post` 多场景处理、`download_image` 双通道降级等关键实现做深度剖析。

[来源](packages/core/src/ai/tools.ts#L32-L72)

## 1. 工具类型体系

每个工具由三层结构描述：

```
ToolDefinition  →  声明层：name + description + inputSchema（JSON Schema）
ToolHandler     →  执行层：(params, assistant?) => Promise<string>
ToolDescriptor  →  聚合层：definition + handler + requiresWrite
```

`ToolHandler` 的第二个参数 `assistant` 是可选的 `AIAssistant` 实例引用，仅在 `view_image` 和 `create_post` 中有实际用途——前者需要调用 `addPendingImage` 注入图片数据，后者需要调用 `getUserUpload` 读取用户上传。这种设计避免了将所有工具 handler 绑定到 assistant 实例，同时保留了反向控制流的通道。

`requiresWrite` 是一个布尔标志位，在[AI 助手系统](ai-助手与工具调用系统.md)的 `sendMessage` 循环中触发确认门控：当 `toolDesc.requiresWrite === true` 时，执行器会在调用 handler 之前 `await this._waitForConfirmation()`，阻塞整个异步循环直到用户通过 `confirmAction(boolean)` 决策。拒绝后 handler 返回 `"User cancelled the operation."` 字符串，LLM 可以感知到用户的否定意图。

## 3. 线程展平算法：flattenThread

`flattenThread` 是 AI 工具集中最精巧的算法之一，服务于 `get_post_thread_flat`、`get_post_subtree`、`get_post_context` 三个工具。它递归遍历 Bluesky API 返回的 `ThreadViewPost` 树结构，输出带 **深度标记**（`depth:N`）的纯文本行。

### 3.1 算法流程

```
flattenThread(client, uri, maxDepth=3, maxReplies=5)
    ↓
1. maxReplies = Math.min(maxReplies, 20)       // 硬上限保护
2. getPostThread(uri, depth=maxDepth+1, 80)    // 一次API调用拉取整棵树
3. 检查 thread.$type 是否为 threadViewPost
    ↓
4. 收集祖先链（ancestors）
   祖先链：thread.parent → 递归 parent → 直到根
   while (cursor && cursor.$type === 'threadViewPost')
     ancestors.unshift(cursor.post)            // 用 unshift 保持正向顺序
     cursor = cursor.parent
    ↓
5. 输出祖先行（负深度 depth:-N）
   for (let i = 0; i < ancestors.length; i++)
     depth = -(ancestors.length - i)           // 最远的祖先 depth 最小
    ↓
6. 输出根帖（depth:0）
7. 递归输出回复（从 depth:1 开始）
   buildReplyLines(replies, 1, maxDepth, capped, lines)
```

### 3.2 祖先链收集

这是一个 **逆向爬升然后正向输出** 的过程：

```
ThreadViewPost {
  post: PostView           ← 目标帖子
  parent?: ThreadViewPost  ← 父帖子（递归）
  replies?: [...]          ← 子回复列表
}
```

算法从 `thread.parent` 开始，通过 `while` 循环向上追溯到最顶层的祖先，用 `ancestors.unshift(post)` 将祖先按 **从老到新** 的顺序存入数组。输出时，最远的祖先获得 `depth:-N`（N 为祖先总数），根帖父节点获得 `depth:-1`。

```
祖先链输出格式：
depth:-2 | ancient_handle (post:rkey111)
 ↳ > depth:-1 | parent_handle (post:rkey222)
depth:0 | root_handle (post:rkey333)
text: 这是根帖子
```

` ↳ > ` 前缀仅出现在有多个祖先时，`ancestors[i-1]` 存在时添加，形成关系链的视觉连接。

### 3.3 回复树的递归展平

`buildReplyLines` 是递归函数，每层执行以下步骤：

1. **过滤与排序**：只保留 `$type === 'threadViewPost'` 的有效回复，按 `indexedAt` 升序排列（时间顺序）
2. **截断**：取前 `maxSiblings` 条（上限 20）
3. **输出**：每行格式为 `{indent}{branch}depth:{d} → (post:parentRkey) | author (post:rkey)\ntext: ...`
4. **递归**：对每条显示的回复，在其子回复上调用 `buildReplyLines(reply.replies, currentDepth + 1, ...)`
5. **折叠提示**：如果有效回复数超过 `maxSiblings`，在末尾追加折叠行

```
输出示例：
  ↳ depth:1 → (post:rkey333) | replier (post:rkey444)
  text: 这是回复
    ↳ depth:2 → (post:rkey444) | sub_replier (post:rkey555)
    text: 深度回复
  （还有 3 条回复被折叠，可调用 get_post_subtree 展开）
```

折叠提示中的 `可调用 get_post_subtree 展开` 是一个 **自指引提示**——指引 LLM 在需要时调用 `get_post_subtree` 工具来展开被折叠的线程部分，形成工具间的协作闭环。

### 3.4 媒体信息注入

`formatPostLine` 函数在每行帖子文本后附加媒体摘要：

- `[图片: N 张]` — `app.bsky.embed.images`
- `[链接: hostname]` — `app.bsky.embed.external`，提取 URL hostname
- `[引用: rkey]` — `app.bsky.embed.record`，提取被引用帖的 rkey
- `[引用+媒体]` — `app.bsky.embed.recordWithMedia`
- `[视频]` — `app.bsky.embed.video`

## 4. download_image：Node.js / PWA 双通道降级策略

`download_image` 的设计体现了项目 **跨平台架构** 的本质约束。它的目标是将 Bluesky 上的图片保存到用户本地，但实现方式因运行环境而异：

### 4.1 核心逻辑

```typescript
handler: async (p) => {
  const data = await client.downloadBlob(did, cid);  // 1. 从CDN下载二进制
  const ext = detectMimeType(data).split('/')[1];    // 2. 嗅探扩展名
  const filename = ...;                                // 3. 生成文件名

  // ── 双通道降级 ──
  try {
    // 通道 A：Node.js（TUI）
    const { writeFileSync, mkdirSync } = await import('fs');
    const { homedir } = await import('os');
    const { join } = await import('path');
    const downloadsDir = join(homedir(), 'Downloads');
    mkdirSync(downloadsDir, { recursive: true });
    writeFileSync(join(downloadsDir, filename), data);
    return { saved: filepath, size, mimeType };
  } catch {
    // 通道 B：浏览器/PWA（降级）
    const base64 = toBase64(data);
    const dataUrl = `data:${mimeType};base64,${base64}`;
    return { saved: false, dataUrl, filename, size, mimeType,
             note: 'Image data returned as data URL...' };
  }
}
```

### 4.2 通道选择逻辑

| 条件 | 通道 | 文件系统要求 | 结果 |
|------|------|-------------|------|
| `import('fs')` 成功 | Node.js 原生 | 有文件系统 | 写入 `~/Downloads/`，返回 `{ saved: filepath }` |
| `import('fs')` 失败 | 浏览器 base64 | 无文件系统 | 返回 `data:` URL，`saved: false` |

关键设计要点在于 **动态 import 的位置**：`fs`、`os`、`path` 三个 Node.js 内置模块通过 `await import()` 按需加载。在浏览器环境下，这些 import 会抛出异常，立即进入 catch 分支——这正是双通道的"降级"触发点。

### 4.3 跨平台 Base64 编码

`toBase64` 函数也体现了同样的跨平台考量：

```typescript
function toBase64(data: Uint8Array): string {
  if (typeof Buffer !== 'undefined')
    return Buffer.from(data).toString('base64');   // Node.js 通道
  let binary = '';
  for (let i = 0; i < data.length; i++)
    binary += String.fromCharCode(data[i]!);
  return btoa(binary);                              // 浏览器通道
}
```

先检测 `Buffer` 是否存在（Node.js 全局对象），存在则用原生 `Buffer.toString('base64')`，否则退化到 `btoa()` 的 polyfill 实现。MIME 类型嗅探同样兼顾两端——`detectMimeType` 通过检查文件魔数（Magic Bytes）识别 PNG/JPEG/GIF，不依赖文件扩展名。

## 5. view_image：视觉模型的图片注入协议

`view_image` 的设计意图是让 LLM **在响应中描述图片内容**，而不是返回图片数据。它通过三个步骤实现：

### 5.1 三参数互斥

```typescript
if (p.uploadIndex !== undefined) {
  // 来源 A：用户通过 UI 上传的图片
  const upload = assistant.getUserUpload(uploadIndex);
  data = upload.data; mimeType = upload.mimeType;
} else if (p.did && p.cid) {
  // 来源 B：Bluesky 帖子中的图片
  data = await client.downloadBlob(did, cid);
} else {
  return { error: 'Provide either did+cid or uploadIndex' };
}
```

两个参数组互斥、但都不强制——因为 Schema 中 `required: []`。handler 内部通过条件分支判断实际使用了哪一组。

### 5.2 图片注入机制

`view_image` 不直接返回图片内容给 LLM，而是通过 `assistant.addPendingImage(dataUrl, alt)` 将图片注册到 `_pendingImages` 队列。当下一次 LLM 请求发生时，`_buildMessages()` 会将该队列中的图片注入到最后一个 `role: 'user'` 消息中，格式为 `ContentBlock[]`（包含 ALT 文本 + `image_url` 块）。注入后立即清空队列，避免跨轮重复。

这意味着 LLM 必须 **在调用 `view_image` 之后紧接着调用另一个思考步骤**（或发送最终响应）才能"看到"图片——这是一个先获取、后注入的两步协议。

## 6. fetch_web_markdown：r.jina.ai 代理架构

`fetch_web_markdown` 通过 [r.jina.ai](https://r.jina.ai) 代理服务将任意 URL 转换为干净 Markdown。不需要 LLM 直接访问原始网页，而是经过 Jina AI 的解析层预处理。

### 6.1 请求组装

```typescript
const proxyUrl = `https://r.jina.ai/${url}`;
const res = await fetch(proxyUrl, {
  headers: { 'Accept': 'text/markdown' },
});
```

代理 URL 的构造方式是将原始 URL 拼接在 `https://r.jina.ai/` 之后。`Accept: text/markdown` 头部要求代理返回 Markdown 格式，而非默认的 HTML。

### 6.2 长度守卫

```typescript
const trimmed = md.length > 10000
  ? md.slice(0, 10000) + '\n\n... (truncated)'
  : md;
```

10,000 字符的硬截断防止响应体过大。截断后的 `... (truncated)` 标记让 LLM 知道内容不完整，可根据需要请求特定片段。

### 6.3 标题提取

`extractTitle` 函数采用三级降级策略：

1. 从 Markdown `# 标题`（H1）提取
2. 若无 H1，从 `## 标题`（H2）提取
3. 若均无，从 `<title>` 标签提取
4. 最终保底：取第一行的前 120 字符

## 7. create_post：多场景发帖引擎

`create_post` 是写入工具中复杂度最高的一个，其 handler 约 90 行代码，需要处理 **四种发帖模式** 及其组合：

### 7.1 场景矩阵

| 场景 | `replyTo` | `quoteUri` | `images` | 输出 Record 结构 |
|------|:---------:|:----------:|:--------:|-----------------|
| 纯文字帖 | — | — | — | `{ text, createdAt }` |
| 回复帖 | ✅ | — | — | `{ text, createdAt, reply: { root, parent } }` |
| 引用帖 | — | ✅ | — | `{ text, createdAt, embed: { $type: 'record', record: { uri, cid } } }` |
| 带图帖 | — | — | ✅ | `{ text, createdAt, embed: { $type: 'images', images: [...] } }` |
| 回复+引用 | ✅ | ✅ | — | `{ text, createdAt, reply, embed: { $type: 'record', ... } }` |
| 引用+带图 | — | ✅ | ✅ | `{ text, createdAt, embed: { $type: 'recordWithMedia', record, media } }` |

### 7.2 回复链：root/parent 解析

当 `replyTo` 参数存在时，handler 需要回复链的 `root` 和 `parent` 两个值：

```typescript
// Step 1: 获取被回复帖的 Record，取其 CID 作为 parent
const replyRec = await client.getRecord(replyParsed.did, ...);
const parentCid = replyRec.cid;

// Step 2: 向上追溯 root
let rootUri = replyTo, rootCid = parentCid;
const threadRes = await client.getPostThread(replyTo, 0, 10);
if (threadRes.thread.$type === 'threadViewPost') {
  let current = threadRes.thread;
  while (current?.parent?.$type === 'threadViewPost') {
    current = current.parent;    // 沿 parent 链上行
  }
  rootUri = current.post.uri;    // 最顶层即为 root
  rootCid = current.post.cid;
}
record.reply = { root: { uri: rootUri, cid: rootCid },
                 parent: { uri: replyTo, cid: parentCid } };
```

算法通过 `getPostThread`（depth=0，上限 10 条回复）获取目标帖子的父链，沿 `parent` 指针上行直到遇见 `$type !== 'threadViewPost'` 或无 `parent`，此时 `current` 指向线程的根帖子。这与 AT Protocol 的 `app.bsky.feed.post` Record 中 `reply.root` 和 `reply.parent` 的定义完全一致。

### 7.3 图片嵌入：跨 PDS 搬运 + pendingImageIndex

图片嵌入分三阶段：

```
1. 下载阶段
   for each image:
     if (img.pendingImageIndex !== undefined)
       data = assistant.getUserUpload(index)  // 用户上传图（来自 UI）
     else if (img.did && img.cid)
       data = client.downloadBlob(did, cid)    // Bluesky 帖图（来自 CDN）

2. 上传阶段
   const blob = await client.uploadBlob(data, mimeType);  // 上传到用户 PDS

3. 嵌入阶段
   record.embed = {
     $type: 'app.bsky.embed.images',
     images: [{ image: { ref: { $link: blob.blob.ref.$link } }, mimeType, size, alt }]
   };
```

`pendingImageIndex` 机制的意义在于：用户通过 UI 上传的图片存储在 assistant 的 `_userUploads` 数组中，AI 无法直接访问文件系统或浏览器 DOM，只能通过这个整数索引来引用。`getUserUpload` 返回 `Uint8Array` 二进制数据，给 handler 带来了完全一致的 blob 处理路径和跨 PDS 上传能力。

### 7.4 引用帖 + 图片：recordWithMedia 嵌套

当 `quoteUri` 与 `images` 同时存在时，AT Protocol 要求使用 `app.bsky.embed.recordWithMedia` 这种复合嵌入类型：

```typescript
if (p.quoteUri) {
  const quoteRec = await client.getRecord(quoteParsed.did, ...);
  if (record.embed) {
    // 已有图片嵌入 → 包装成 recordWithMedia
    record.embed = {
      $type: 'app.bsky.embed.recordWithMedia',
      record: { uri: p.quoteUri, cid: quoteRec.cid },
      media: record.embed,                         // 原来的 images embed 作为 media
    };
  } else {
    // 无图片 → 纯引用 embed
    record.embed = {
      $type: 'app.bsky.embed.record',
      record: { uri: p.quoteUri, cid: quoteRec.cid },
    };
  }
}
```

这里的核心逻辑是 **条件包装**：`images` 先被嵌入到 `record.embed` 的 `app.bsky.embed.images` 中，当检测到需要引用帖子时，将其包裹在 `recordWithMedia` 的外壳下，原 `images` 成为 `media` 字段。这种实现遵循了 Bluesky 的 `app.bsky.embed.recordWithMedia` Lexicon 定义。

## 8. 辅助函数与工具函数

### 8.1 MIME 类型嗅探

```typescript
function detectMimeType(data: Uint8Array): string {
  if (data.length < 4) return 'application/octet-stream';
  const pngMagic = [0x89, 0x50, 0x4e, 0x47];
  const jpgMagic = [0xff, 0xd8, 0xff];
  const gifMagic = [0x47, 0x49, 0x46];
  if (pngMagic.every((b, i) => data[i] === b)) return 'image/png';
  if (jpgMagic.every((b, i) => data[i] === b)) return 'image/jpeg';
  if (gifMagic.every((b, i) => data[i] === b)) return 'image/gif';
  return 'application/octet-stream';
}
```

通过检查文件前 3-4 字节的魔数识别格式，不依赖文件扩展名或 `Content-Type` 头部。

### 8.2 extractRkey

```typescript
function extractRkey(uri: string): string {
  const parts = uri.split('/');
  return parts[parts.length - 1] ?? uri;
}
```

从 AT URI 中提取最后的记录键（Record Key），用于在展平线程的 `(post:rkey)` 标记中标识帖子。

## 9. 与项目其他系统的关系

- 所有工具通过 `createTools(client: BskyClient)` 注册，`BskyClient` 实例来自 [AT Protocol 客户端封装](at-protocol-客户端封装.md)，其 `downloadBlob` 和 `uploadBlob` 方法支撑了图片工具的跨 PDS 搬运逻辑
- 工具的 `requiresWrite` 标志位在 [AI 助手系统](ai-助手与工具调用系统.md) 中被 `sendMessage` 和 `sendMessageStreaming` 消费，触发写操作确认门控
- 系统提示词（System Prompt）在[提示词工程与系统提示](提示词工程与系统提示.md)中约束 AI 不能主动发起写操作，需等待用户指令
- `get_post_thread_flat` 的线程展平算法在前端 `useThread` Hook 中有对应的实现 `flattenThreadTree`，两者结构相似但输出格式不同——前者输出纯文本供 LLM 理解，后者输出 `FlatLine[]` 供 UI 渲染
- `contracts/tools.json` 记录了相同的工具清单，但存在两处差异：代码中的 `view_image` 在 JSON 中不存在；JSON 中的 `upload_blob` 在代码中尚未实现
