## 设计总览

翻译系统是 `@bsky/core` 中一个**独立于 AI 对话引擎**的轻量功能模块。它不走 `AIAssistant` 的完整工具循环和流式管线，而是直接使用裸 `fetch` 调用 LLM 的 `/v1/chat/completions` 端点，专为"翻译帖文/简介"这一高频低延迟场景优化。

核心代码集中在两个位置：

1. **`assistant.ts`** — `translateText()` 函数 + `TranslationResult` 接口 + `translateToChinese()` 便捷封装
2. **`prompts.ts`** — `PF_TRANSLATE_SIMPLE()` 和 `PF_TRANSLATE_JSON()` 两个 prompt 工厂函数 + `LANG_LABELS` 语言标签映射

[来源](packages/core/src/ai/assistant.ts#L710-L810) | [来源](packages/core/src/ai/prompts.ts#L1-L13)

## 双模式：Simple vs JSON

`translateText()` 接受一个 `mode: 'simple' | 'json'` 参数，区分两种输出格式。两者的核心差异体现为**提示词（prompt）**与**请求体（request body）**的双重分化。

### Prompt 差异

| 维度 | `simple` 模式 | `json` 模式 |
|------|--------------|-------------|
| 函数 | `PF_TRANSLATE_SIMPLE(langLabel)` | `PF_TRANSLATE_JSON(langLabel)` |
| 输出指令 | 纯文本，只输出翻译结果，无解释 | 强制输出含 `source_lang` + `translated` 字段的 JSON |
| 源语言检测 | ❌ 无 | ✅ 通过 `source_lang` 输出 ISO 639-1 代码 |
| 容错指令 | 无 | 指示 LLM "不确定时用 'und'" |

**Simple 模式的 prompt 原文：**

```
Translate the following text to ${targetLang}. Keep the original meaning, output only the translation, no explanations.
```

**JSON 模式的 prompt 原文：**

```
You are a translator. Translate the user's text to ${targetLang}. Output valid JSON with these keys: {"source_lang": "<ISO 639-1 code, use 'und' if unsure>", "translated": "<the translation>"}. Output ONLY the JSON object. The response must be a valid JSON object containing the word json.
```

注意 JSON 模式 prompt 末尾的 `containing the word json` 是一道针对性护栏——某些模型在 `response_format: json_object` 模式下仍然会输出纯文本，加上这段措辞能显著提高 JSON 输出的遵从率。

[来源](packages/core/src/ai/prompts.ts#L77-L97)

### 请求体差异

两种模式共享相同的消息结构（system + user）和超参数（`temperature: 0.3`、`max_tokens: 2000`、`thinking: disabled`），唯一的请求体差异是 JSON 模式**追加 `response_format` 字段**：

```typescript
// JSON 模式独有的请求体字段
if (mode === 'json') {
  body.response_format = { type: 'json_object' };
}
```

对于 `thinking: { type: 'disabled' }`，这一设置确保了翻译调用不会触发 LLM 的推理过程（受限于较短的 `max_tokens: 2000`，翻译场景不需要推理），避免 token 浪费在推理内容的输出上。

[来源](packages/core/src/ai/assistant.ts#L752-L757)

### 响应解析差异

返回类型统一为 `TranslationResult` 接口：

```typescript
export interface TranslationResult {
  translated: string;
  sourceLang?: string;  // ISO 639-1 code, 或 'und' (undefined); 仅 JSON 模式有值
}
```

- **Simple 模式**：直接将 LLM 响应内容赋值给 `translated`，`sourceLang` 不出现。
- **JSON 模式**：对 LLM 响应执行 `JSON.parse()`，提取 `translated` 和 `source_lang` 两个字段。若 `source_lang` 不存在则回退为 `'und'`（unknown/undefined）。

[来源](packages/core/src/ai/assistant.ts#L710-L723) | [来源](packages/core/src/ai/assistant.ts#L776-L800)

## 重试逻辑：指数退避 + 三阶段判定

`translateText()` 的内核是一个 `for (let attempt = 0; attempt < maxRetries; attempt++)` 循环，默认 `maxRetries = 3`。每次重试的等待时间遵循**阶梯式线性增长**（不完全是指数，但严格递增）：

| 尝试次数 | 重试延迟（空内容/解析失败） | 重试延迟（网络/HTTP 错误） |
|---------|--------------------------|--------------------------|
| 第 0 次 | 直接执行，不平移 | 直接执行，不平移 |
| 第 1 次 | 800ms × 1 = 800ms | 1000ms × 1 = 1000ms |
| 第 2 次 | 800ms × 2 = 1600ms | 1000ms × 2 = 2000ms |

### 触发重试的三个条件

1. **空内容（empty content）** — API 正常返回但 `content.trim()` 为空字符串。这是最有价值的重试场景，通常由 LLM 短暂"失语"引起。
   ```typescript
   if (!content.trim()) {
     if (attempt < maxRetries - 1) {
       await new Promise(r => setTimeout(r, 800 * (attempt + 1)));
       continue;
     }
     throw new Error('Translation returned empty content after retries');
   }
   ```

2. **JSON 解析失败（parse error）** — 仅在 `json` 模式下触发。`JSON.parse(content)` 抛出异常（非法 JSON 格式）。
   ```typescript
   try {
     const parsed = JSON.parse(content) as { translated?: string; source_lang?: string };
     // ...
   } catch {
     if (attempt < maxRetries - 1) {
       await new Promise(r => setTimeout(r, 800 * (attempt + 1)));
       continue;
     }
     return { translated: content, sourceLang: 'und' };
   }
   ```
   注意末次重试的降级策略：不抛出异常，而是**直接返回原始内容**并将 `sourceLang` 标记为 `'und'`。这是比空内容更宽松的行为——至少用户能看到 LLM 说了什么。

3. **缺少 `translated` 字段** — 仅在 `json` 模式下触发。`JSON.parse` 成功但解析结果没有 `translated` 键。
   ```typescript
   if (!parsed.translated) {
     if (attempt < maxRetries - 1) {
       await new Promise(r => setTimeout(r, 800 * (attempt + 1)));
       continue;
     }
     return { translated: content, sourceLang: 'und' };
   }
   ```
   末次降级逻辑与条件 2 一致。

4. **网络/HTTP 错误** — `fetch` 抛出的异常或非 2xx 状态码。由外层的 `catch (e)` 捕获，重试间隔为 1000ms × attempt（比数据错误略大，因为网络通常需要更久恢复）。
   ```typescript
   } catch (e) {
     if (attempt < maxRetries - 1) {
       await new Promise(r => setTimeout(r, 1000 * (attempt + 1)));
       continue;
     }
     throw e;
   }
   ```

### 重试流程状态图

```mermaid
flowchart TD
    A[发起请求] --> B{HTTP 200?}
    B -- 否 --> C[catch 块捕获]
    C --> D{还有重试次数?}
    D -- 是 --> E[等待 1000ms×(attempt+1)]
    E --> A
    D -- 否 --> F[抛出异常]
    B -- 是 --> G{content 为空?}
    G -- 是 --> H{还有重试次数?}
    H -- 是 --> I[等待 800ms×(attempt+1)]
    I --> A
    H -- 否 --> J[抛出异常]
    G -- 否 --> K{mode === json?}
    K -- 否 --> L[返回 {translated: content}]
    K -- 是 --> M[JSON.parse]
    M --> N{解析成功?}
    N -- 否 --> O{还有重试次数?}
    O -- 是 --> I
    O -- 否 --> P[返回 {translated: content, sourceLang: 'und'}]
    N -- 是 --> Q{有 translated 字段?}
    Q -- 否 --> R{还有重试次数?}
    R -- 是 --> I
    R -- 否 --> S[返回 {translated: content, sourceLang: 'und'}]
    Q -- 是 --> T[返回 {translated: parsed.translated, sourceLang: parsed.source_lang || 'und'}]
```

## 语言支持：LANG_LABELS

翻译目标语言由 `LANG_LABELS` 常量表定义，支持 **7 种语言**，覆盖东亚 + 欧洲主要语种：

| 代码 | 标签 | 语系 |
|------|------|------|
| `zh` | 中文 | 汉藏语系 |
| `en` | English | 印欧语系（日耳曼语族） |
| `ja` | 日本語 | 孤立语言（日琉语系） |
| `ko` | 한국어 | 孤立语言（朝鲜语系） |
| `fr` | Français | 印欧语系（罗曼语族） |
| `de` | Deutsch | 印欧语系（日耳曼语族） |
| `es` | Español | 印欧语系（罗曼语族） |

`langLabel = LANG_LABELS[targetLang] ?? targetLang` 的 fallback 机制确保了即使传入未注册的代码（如 `'it'`），系统仍能以原始代码作为标签发送给 LLM，不会崩溃。

`LANG_LABELS` 同时在三个包中被引用：
- **`@bsky/core`** — 在 `translateText()` 中用于将代码转换为自然语言标签
- **`@bsky/app`** — 在 `useTranslation` hook 中重新导出为 `LANG_LABELS` 常量
- **`@bsky/tui` / PWA** — 在设置界面用作语言选择器的选项标签（如 TUI 配置中的 `targetLang` 字段）

[来源](packages/core/src/ai/prompts.ts#L7-L16) | [来源](packages/app/src/hooks/useTranslation.ts#L8-L17)

## 消费端集成

### `useTranslation` Hook（`@bsky/app`）

React 侧的翻译入口，封装了与 `@bsky/core` 的异步导入 + 内存缓存：

- **`cache`** — `Map<string, TranslationResult>`，key 为 `${mode}::${lang}::${text}`。**对同一文本的重复翻译请求直接走缓存**，避免相同内容反复调用 API。
- **`translate(text, overrideLang?)`** — 异步函数，动态 `import('@bsky/core')` 后调用 `translateText()`。
- **`setMode`** — 允许在运行时切换 `'simple'` / `'json'` 模式。

[来源](packages/app/src/hooks/useTranslation.ts#L1-L56)

### TUI 端调用

TUI 中翻译功能通过快捷键 `f`（`ProfileView`）或 `f`（`UnifiedThreadView`）触发，直接 `import('@bsky/core').then(({ translateText: tt }) => ...)` 动态加载：

```typescript
// UnifiedThreadView.tsx — 翻译帖文
import('@bsky/core').then(({ translateText: tt }) => {
  const cfg = { apiKey, baseUrl, model };
  tt(cfg, cursorLine.text, targetLang || 'zh', translateMode || 'simple')
    .then(r => setTranslatedText(`[${r.sourceLang ?? '?'}→${targetLang}] ${r.translated}`));
});

// ProfileView.tsx — 翻译个人简介
tt(cfg, profile.description!, targetLang || 'zh', 'simple')
  .then(r => setTranslatedBio(`[→${targetLang}] ${r.translated}`));
```

注意 ProfileView 下翻译简介**固定使用 `simple` 模式**，而帖文翻译会读取用户的 `translateMode` 配置。

[来源](packages/tui/src/components/UnifiedThreadView.tsx#L145-L156) | [来源](packages/tui/src/components/ProfileView.tsx#L80-L87)

PWA 的设置页面（`SettingsModal`）提供了一个 `<select>` 下拉菜单，支持在 `simple` 和 `json` 两种模式间切换，配置存储在 `useAppConfig` 中，持久化为 `translateMode` 字段。

## 设计决策要点

1. **独立于 AI 对话系统**：翻译不走 `AIAssistant` 的 `sendMessage()`，而是通过 `singleTurnAI()` 风格的独立 `fetch` 调用。原因：翻译是单轮、确定性任务，不需要工具调用、流式输出、对话历史管理、写确认门控等复杂机制。将两者分离降低了翻译路径的延迟和失败概率。

2. **`thinking: disabled`**：翻译调用显式禁用推理过程，因为翻译不需要"思考"——更短的 token 消耗意味着更低成本和更快响应。

3. **`modelOverride` 参数**：`translateText()` 接受可选的 `modelOverride`，允许调用方为翻译指定与主对话不同的模型。这在 [配置指南](配置指南.md) 中提到的"场景模型覆盖"场景下尤为有用——例如用更便宜的模型做翻译，用更强的模型做对话。

4. **JSON 模式的源语言检测**：`source_lang` 字段为需要元数据的场景（如 UI 中显示"从日语翻译为中文"）提供了可能性，且 backward compatible——已有 `simple` 模式的消费者无需改动。
