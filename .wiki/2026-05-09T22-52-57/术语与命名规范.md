## 讨论串相关术语

Bluesky 的帖子不是孤立存在的——一条帖子可以回复另一条，形成树状结构。以下是项目中对这个结构每个节点的命名：

| 中文术语 | 英文对照 | 含义 |
|---------|---------|------|
| **主题帖** | Theme Post | 一条讨论串的起点，记录中没有 `reply` 字段，不指向任何其他帖子 |
| **回复** | Reply | 指向另一条帖子的帖子，记录中有 `reply.parent` 字段 |
| **当前帖子** | Current Post / Focused Post | 用户当前正在查看或交互的那条帖子，UI 中高亮显示 |
| **讨论串** | Discussion Chain | 从当前帖子回溯到主题帖的完整链条，包含当前帖子本身 |
| **讨论源** | Discussion Source | 讨论串中**排除**当前帖子后的部分，用于展示当前帖子之上的上下文 |

[来源](docs/TERMINOLOGY.md#L7-L11)

### 为什么不用"父帖/子帖"？

很多论坛系统用"父帖/子帖"来描述回复关系，但本项目**刻意避免**这类术语，原因有三：

- ❌ **"父帖/母帖"（parent post）**——带有父权/母权色彩的语言暗示
- ❌ **"子帖"（child post）**——有居高临下的意味
- ✅ **"主题帖""回复""讨论串/讨论源"**——完全中立、描述性的词汇，与 Bluesky 官方术语一致

> 这不是文字洁癖，而是设计选择：用"主题帖"替代"父帖"，用"回复"替代"子帖"，使得每条帖子的地位平等，讨论结构由内容关联驱动，而非层级权威驱动。

[来源](docs/TERMINOLOGY.md#L13-L18)

## 代码命名规范

代码中的变量名和类型名直接反映了上述术语体系：

| 概念 | 代码中的名字 | 说明 |
|-----|-------------|------|
| 当前帖子在列表中的位置 | `focusedIndex` | 数字索引，指向 `flatLines` 数组中的某个元素 |
| 主题帖的 AT URI | `themeUri` | 字符串，形如 `at://did:plc:xxx/app.bsky.feed.post/yyy` |
| 当前帖子对象 | `focused` / `focusedLine` | 类型为 `FlatLine`，是 `flatLines[focusedIndex]` |
| 展开后的帖子列表 | `flatLines: FlatLine[]` | 将树状讨论结构"拍平"为一维数组 |
| 帖子在树中的深度 | `FlatLine.depth` | 数字：`0` 表示主题帖，负值表示讨论源，正值表示回复 |
| 回复指示符 | `↳` | 箭头符号，指向被回复的帖子 |

[来源](docs/TERMINOLOGY.md#L34-L41)

### `FlatLine` 的 depth 设计

这是理解讨论串渲染的关键。`FlatLine` 接口定义在 `useThread.ts` 中：

```typescript
export interface FlatLine {
  depth: number;    // 0 = 主题帖, 负数 = 讨论源, 正数 = 回复
  uri: string;
  rkey: string;
  text: string;
  isRoot: boolean;  // depth === 0 时 true
  isTruncation: boolean;
  // ... 更多字段
}
```

[来源](packages/app/src/hooks/useThread.ts#L7-L42)

`flattenThreadTree` 递归遍历时，`depth` 的计算方式：
- 从主题帖（`depth = 0`）开始
- 向上遍历**父帖**（讨论源方向）：每向上走一层，`depth` 减 **1**（变成 -1, -2, ...）
- 向下遍历**回复**方向：每向下走一层，`depth` 加 **1**（变成 1, 2, ...）

[来源](packages/app/src/hooks/useThread.ts#L140-L210)

这样，UI 组件只需根据 `depth` 的正负号就能区分"上下文"和"回复"，无需手动判断方向。
