# 哈希路由、存储与离线支持

PWA 端与 TUI 端最大的差异在于运行环境：浏览器没有文件系统、没有进程级别的状态栈，但提供了 `history.pushState`、`IndexedDB` 和 `Service Worker` 等 Web 平台能力。本文分析 PWA 如何利用这些能力，在纯静态托管下实现 **导航路由**、**聊天持久化** 和 **离线缓存**。

---

## 哈希路由：将 AppView 编入 URL

PWA 部署为静态站点（Cloudflare Pages / Netlify 等），无法依赖服务端路由。解决方案是 **URL hash 路由**：将应用视图状态编码到 `#` 之后的部分，借助 `popstate` 事件和 `history.pushState` 实现浏览器原生前进/后退集成。

### 双向编解码

核心是 `useHashRouter` 中的两对函数：

**`parseHash()`** — 从 URL hash 解析出 `AppView`：

```
#/feed?feed=at://...          → { type: 'feed', feedUri: 'at://...' }
#/thread?uri=at://...         → { type: 'thread', uri: 'at://...' }
#/profile?actor=did:plc:...   → { type: 'profile', actor: 'did:plc:...' }
#/notifications               → { type: 'notifications' }
#/search?q=...                → { type: 'search', query: '...' }
#/bookmarks                   → { type: 'bookmarks' }
#/compose?replyTo=at://...    → { type: 'compose', replyTo: 'at://...' }
#/ai?session=xxx              → { type: 'aiChat', sessionId: 'xxx' }
```

[来源](哈希路由-存储与离线支持.md#L1-L107)

**`encodeView()`** — 反向操作，将 `AppView` 序列化为 hash 字符串。两者配合实现了 **双向映射**，确保浏览器地址栏与应用状态严格同步。

### 浏览器导航集成

`useHashRouter` 使用 `history.pushState` 代替直接修改 `location.hash`。这么做有两个意图：
- **不触发 `hashchange` 事件**，统一由 `popstate` 控制状态流转；
- **保留历史记录条目**，使浏览器的前进/后退按钮正常工作。

`goBack()` 调用 `window.history.back()`，由浏览器自身决定回退到哪个历史条目。`goHome()` 则直接 `pushState` 重置为默认 Feed 视图。

[来源](哈希路由-存储与离线支持.md#L1-L107)

> 关于 `AppView` 类型系统的完整定义，参见 [](导航路由与视图管理.md)。

### 与 TUI 导航的差异

| 方面 | TUI (useNavigation) | PWA (useHashRouter) |
|------|--------------------|-------------------|
| 存储位置 | 内存栈 | URL hash + History API |
| 前进/后退 | 程序化栈操作 | 浏览器原生 popstate |
| 刷新保持 | 丢失 | 保留（URL 可恢复） |
| 分享能力 | 无 | 可复制 URL 分享 |

TUI 端的导航栈在内存中维护，详见 [](状态管理模式.md)。

## IndexedDB 聊天存储

聊天记录需要跨会话持久化，浏览器环境下 `IndexedDB` 是唯一适合存储结构化大对象（完整消息列表）的 API。

### ChatStorage 接口

`@bsky/app` 包定义了统一的 `ChatStorage` 接口：

```typescript
interface ChatStorage {
  saveChat(chat: ChatRecord): Promise<void>;
  loadChat(id: string): Promise<ChatRecord | null>;
  listChats(): Promise<ChatSummary[]>;
  deleteChat(id: string): Promise<void>;
}
```

[来源](哈希路由-存储与离线支持.md#L1-L36)

TUI 端由 `FileChatStorage` 实现（读写 JSON 文件），PWA 端则由 **`IndexedDBChatStorage`** 实现。

### IndexedDBChatStorage 实现

核心结构：

```
数据库: bsky-chats (v1)
  对象存储: chats (keyPath: 'id')
```

每个 `ChatRecord` 以 `id` 为主键直接存取。实现要点：

| 方法 | 操作 | 事务模式 |
|------|------|---------|
| `saveChat` | `store.put()` — 覆盖式写入 | `readwrite` |
| `loadChat` | `store.get(id)` | `readonly` |
| `listChats` | `store.getAll()` → 过滤 → 按 `updatedAt` 降序 | `readonly` |
| `deleteChat` | `store.delete(id)` | `readwrite` |

`listChats()` 的特殊之处：它读出完整 `ChatRecord`，计算 `user`/`assistant` 角色消息数量作为 `messageCount`，再按更新时间排序返回 `ChatSummary[]`。这意味着每次获取列表都要读取全部数据，适合消息量不大的场景。

[来源](哈希路由-存储与离线支持.md#L1-L63)

在 `AIChatPage` 中，`IndexedDBChatStorage` 通过 `useMemo` 实例化一次，注入到 `useChatHistory` hook 中：

```typescript
const storage = useMemo(() => new IndexedDBChatStorage(), []);
```

[来源](哈希路由-存储与离线支持.md#L1-L7)

> 关于 `useChatHistory` 如何消费该存储，详见 [](useaichat-深度解析.md)。

## Service Worker：离线缓存策略

`sw.js` 位于 `public/` 目录，由 Vite 构建时原样复制到 `dist/`。注册入口在 `main.tsx`：

```typescript
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => {
    navigator.serviceWorker.register('./sw.js', { scope: './' })
  });
}
```

[来源](哈希路由-存储与离线支持.md#L1-L13)

### 缓存策略矩阵

Service Worker 为不同类型的资源采用差异化策略：

| 资源类型 | 域名/路径 | 策略 | 缓存名称 |
|---------|----------|------|---------|
| Bluesky CDN 图片 | `cdn.bsky.app` | Cache-first | `bsky-img-v1` |
| Google Fonts 字体文件 | `fonts.gstatic.com` | Cache-first | `bsky-font-v1` |
| Google Fonts CSS | `fonts.googleapis.com` | Stale-while-revalidate | `bsky-font-v1` |
| API 请求 | `bsky.social` / `*.api.*` | Network-first | `bsky-v3` |
| 构建产物的 JS/CSS 图标 | `/assets/` `/icons/` | Cache-first | `bsky-v3` |
| HTML / 根路径 | 所有其他 | Stale-while-revalidate | `bsky-v3` |

[来源](哈希路由-存储与离线支持.md#L1-L91)

**Cache-first**：直接返回缓存副本，无缓存时回退网络并缓存结果（适合内容寻址的不可变资源，如 CDN 图片和带 hash 的构建产物）。

**Network-first**：优先请求网络，成功则更新缓存；网络失败时返回缓存副本（适合 API 数据，始终需要最新值）。

**Stale-while-revalidate**：立即返回缓存（不管是否过期），同时在后台发起网络请求更新缓存（适合偶尔变化的资源，如 HTML 和 CSS 配置文件）。

### 生命周期

- **install**：缓存静态 shell（`index.html`、`manifest.json`），调用 `skipWaiting()` 立即激活。
- **activate**：清除旧版本缓存（`bsky-v1`、`bsky-v2` 等），调用 `clients.claim()` 接管所有客户端。
- **fetch**：根据 URL 路由到不同缓存策略。

[来源](哈希路由-存储与离线支持.md#L30-L91)

## Node 模块桩（Stubs）

`@bsky/app` 包的 `FileChatStorage` 依赖 Node.js 内置模块 `fs`、`path`、`os`。PWA 在浏览器中运行，这些模块不存在，于是 `packages/pwa/src/stubs/` 下提供了 **桩模块（stubs）**，在 Vite 构建时通过 `resolve.alias` 替换掉真实的 Node 模块。

### fs.ts

```typescript
export const existsSync = () => false;
export const mkdirSync = () => {};
export const writeFileSync = () => {};
export const readFileSync = () => '';
export const readdirSync = () => [];
export const unlinkSync = () => {};
```

所有方法均为空操作或返回空值。这保证了 `FileChatStorage` 在浏览器中引入时不会崩溃——虽然它永远不会被 PWA 实际使用（PWA 使用 `IndexedDBChatStorage`），但模块导入解析必须通过。

[来源](哈希路由-存储与离线支持.md#L1-L7)

### path.ts

```typescript
export const join = (...args: string[]) => args.join('/');
```

用简单的斜杠拼接模拟 `path.join`，足以应付 `FileChatStorage` 中的路径组装需求。

[来源](哈希路由-存储与离线支持.md#L1-L2)

### os.ts

```typescript
export const homedir = () => '/';
```

返回根路径作为用户目录，防止 `path.join(homedir(), '.bsky-tui', 'chats')` 出错。

[来源](哈希路由-存储与离线支持.md#L1-L2)

这三个桩文件共同构成了一个 **兼容层**，使得引用 `@bsky/app` 的代码在浏览器中能正常编译。Vite 的配置通过 `resolve.alias` 将 `fs`/`path`/`os` 映射到这些桩文件。

> 关于 `@bsky/app` 包的导出入口和依赖管理，参见 [](包管理与依赖关系.md)。

## 用户会话持久化：localStorage

除聊天记录外，**认证会话** 也需要持久化。PWA 使用 `localStorage` 存储 JWT 凭据：

```typescript
const SESSION_KEY = 'bsky_session';
// saveSession / getSession / clearSession
```

[来源](哈希路由-存储与离线支持.md#L1-L23)

`App.tsx` 在挂载时调用 `getSession()` 恢复会话，调用 `client.restoreSession()` 重建认证状态。登出时调用 `clearSession()`。认证过期时（`authError` 触发），也会自动清除会话并回到登录页。

[来源](哈希路由-存储与离线支持.md#L48-L77)

> 关于 JWT 自动刷新的机制，参见 [](认证与会话自动刷新.md)。
