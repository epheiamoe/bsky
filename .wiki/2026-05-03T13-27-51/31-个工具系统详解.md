# 31 个工具系统详解

`createTools` 是连接 **AI 大模型**与 **Bluesky 网络**的桥梁。它生成 31 个工具描述符，注入 `AIAssistant` 的对话循环中，让 LLM 能够读取 Bluesky 数据并在必要时执行写操作。本文聚焦工具的**分类、接口契约**与**核心算法**。

## ToolDescriptor：工具的三元组契约

每个工具由三个字段构成：

```typescript
interface ToolDescriptor {
  definition: ToolDefinition;  // 向 LLM 暴露的名称、描述、JSON Schema
  handler: ToolHandler;        // 实际执行函数，返回 Promise<string>
  requiresWrite: boolean;      // 写操作门控标记
}
```

`definition` 直接映射到 OpenAI 兼容的 function calling schema——`AIAssistant.makeRequest` 会将其转换为 `tools` 数组发送给 LLM API。`handler` 接受 `(params, assistant?)` 参数，第二个参数允许工具访问 AIAssistant 实例上的 `addPendingImage` 或 `getUserUpload` 等方法，用于多模态图片传递。`requiresWrite` 是安全门控，下文详述。

## 写入工具（4 个）

所有写工具均标记 `requiresWrite: true`。它们都需要用户身份认证，通过 `client.createRecord` 向用户 PDS 写入 AT Protocol 记录。

### create_post：最复杂的工具

`create_post` 的 handler 内部包含**三个分支逻辑**：

1. **回复（replyTo）**：解析父帖 AT URI → 获取 `cid` → 通过 `getPostThread` 向上遍历到**根帖子**以填充 `reply.root`。
2. **图片附件（images）**：支持从 Bluesky CDN 下载现有图片（did/cid）或引用用户上传的图片（`pendingImageIndex`）→ 通过 `client.uploadBlob` 上传到用户 PDS → 构造 `app.bsky.embed.images` 嵌入。
3. **引用帖子（quoteUri）**：若同时有图片，则包装为 `app.bsky.embed.recordWithMedia`。

```typescript
// 回复根帖子的追溯逻辑
let current: ThreadViewPost | undefined = threadRes.thread as ThreadViewPost;
while (current?.parent && (current.parent as ThreadViewPost).$type === 'app.bsky.feed.defs#threadViewPost') {
  current = current.parent as ThreadViewPost;
}
```

## flattenThread 递归格式化算法

`get_post_thread_flat` 是 `get_post_thread` 的**文本化包装**——它将 `ThreadViewPost` 的嵌套树结构展平为带深度标记的文本行，更适合 LLM 理解对话脉络。

### 调用链

```
get_post_thread_flat handler
  → flattenThread(client, uri, maxDepth=3, maxReplies=5)
    → client.getPostThread(uri, depth=max(maxDepth+1, 6), parentHeight=80)
    → 祖先链收集 → 根帖子 → buildReplyLines 递归
```

注意底层 `getPostThread` 请求的 `depth` 传的是 `Math.max(maxDepth + 1, 6)`——这意味着 API 层总能获取至少 6 层完整数据，但 flattenThread 只渲染到 `maxDepth`。这样上层截断不浪费已请求的数据。

[来源](packages/core/src/ai/tools.ts#L697-L701)

### 祖先链收集

```typescript
const ancestors: PostView[] = [];
let cursor = thread.parent;
while (cursor && cursor.$type === 'app.bsky.feed.defs#threadViewPost') {
  ancestors.unshift(cursor.post);   // 从头插入得到正序
  cursor = cursor.parent;
}
```

祖先节点以**负深度**标记：若祖先链有 3 条，则深度依次为 `-3`、`-2`、`-1`，根帖子为 `0`。非首个祖先前加 `↳ >` 前缀。

[来源](packages/core/src/ai/tools.ts#L703-L715)

### buildReplyLines：递归截断与分页

这是算法的精粹：

```
buildReplyLines(replies, currentDepth, maxDepth, maxSiblings, lines)
```

1. **深度截止**：`currentDepth > maxDepth` 时直接返回。
2. **有效性过滤**：只保留 `$type === 'app.bsky.feed.defs#threadViewPost'` 的回复，过滤 `NotFoundPost`。
3. **时间排序**：按 `indexedAt` 升序排列——从最早到最新。
4. **兄弟截断**：`valid.slice(0, maxSiblings)` 只取前 N 条，其余用折叠提示替代：

```
（还有 X 条回复被折叠，可调用 get_post_subtree 展开）
```

5. **递归**：每条显示的回复再递归调用 `buildReplyLines(reply.replies, depth + 1, ...)`。

[来源](packages/core/src/ai/tools.ts#L721-L756)

### formatPostLine：depth:0 格式与媒体指示器

每条帖子的输出格式：

```
depth:-3 | alice.bsky.social (Alice) (post:3lkabc)
depth:0 | bob.bsky.social (Bob) (post:3lkxyz)
text: 这是帖子内容 [图片: 2 张]
  ↳ → (post:3lkdef)depth:1 | carol.bsky.social (Carol) (post:3lkdef)
text: 回复内容 [链接: example.com]
```

关键细节：

- **depth:0 格式**：根帖子加了 `prefix: ''`（空字符串）；祖先链的 `prefix` 为 `↳ >`；回复链的 `prefix` 为 `  ↳` + 可选的 ` → (post:parentRkey)` 箭头。
- **maxDepth 截断**：`buildReplyLines` 入口处直接判断 `currentDepth > maxDepth` 返回。
- **maxSiblings 截断**：`.slice(0, maxSiblings)` 后隐藏超出的回复。
- **媒体指示器**：`formatPostLine` 根据 `embed.$type` 添加后缀：

| embed 类型 | 指示器 |
|-----------|--------|
| `app.bsky.embed.images` | `[图片: N 张]` |
| `app.bsky.embed.external` | `[链接: hostname]` |
| `app.bsky.embed.record` | `[引用: rkey]` |
| `app.bsky.embed.recordWithMedia` | `[引用+媒体]` |
| `app.bsky.embed.video` | `[视频]` |

## requiresWrite 确认门控

`requiresWrite: true` 的工具在执行前会被 `AIAssistant.sendMessage`（或 `sendMessageStreaming`）挂起：

```typescript
if (toolDesc.requiresWrite) {
  const approved = await this._waitForConfirmation();
  if (!approved) {
    toolResult = 'User cancelled the operation.';
    // ... 跳过执行
  }
}
```

`_waitForConfirmation` 返回一个**永不自动 resolve 的 Promise**——外部 UI 必须调用 `assistant.confirmAction(true/false)` 才能解锁。这构成了一个**同步风格**的异步确认门控，让 LLM 的自动工具循环不会在未获用户批准的情况下执行任何写操作。

`sendMessageStreaming` 版本还会 `yield { type: 'confirmation_needed', content: desc }`，让前端渲染确认对话框。`buildToolDescription` 函数为各写工具生成人类可读的描述文本，例如 `"创建帖子: \"Hello world\""`、`"点赞帖子: at://..."`。

## 架构位置与下一步

`createTools` 的输出是 `createTools(client) → ToolDescriptor[]`，在 `useAIChat` Hook 中实例化后通过 `assistant.setTools(tools)` 注入 `AIAssistant`。整套工具系统依赖底层的 [BskyClient](bskyclient-bluesky-api-封装.md) 和 [ATS Protocol 类型系统](bskyclient-bluesky-api-封装.md)。工具调用结果通过 [SSE 流式输出](流式输出-sse-实时渲染.md) 实时渲染给用户。所有工具名称和描述在[系统提示词工程](系统提示词工程.md)中有对应的约束规则（规则 1：禁止自动写操作）。
