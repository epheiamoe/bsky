# 键盘快捷键完整系统

TUI 客户端的快捷键系统由 **5 个 `useInput` 处理器**分层注册构成，覆盖 7 个视图和 4 个对话框模式。本文从架构原则入手，逐步展开全局键、视图本地键、冲突表和新键添加流程。

---

## 架构总览：5 个 useInput + 1 个 raw stdin

Ink 框架在每次按键时，会**按注册顺序触发所有 `useInput` 回调**。本系统共有 5 个 `useInput` 处理器，外加 1 个 `process.stdin.on('data')` 原始监听器（仅用于鼠标滚轮）。

| 处理器 | 文件位置 | 作用域 |
|--------|----------|--------|
| `App.tsx` | `packages/tui/src/components/App.tsx` | **始终激活**。处理 Tab、Esc、Ctrl+G、全局导航、Feed、Bookmarks、Compose |
| `UnifiedThreadView` | `packages/tui/src/components/UnifiedThreadView.tsx` | 仅 `currentView.type === 'thread'` 时激活 |
| `AIChatView`（主聊天） | `packages/tui/src/components/AIChatView.tsx` | 仅 `!showHistory` 时激活 |
| `AIChatView`（历史记录） | `packages/tui/src/components/AIChatView.tsx` | 仅 `showHistory` 时激活 |
| `NotifView` | `packages/tui/src/components/NotifView.tsx` | 仅 `currentView.type === 'notifications'` 时激活 |
| `SetupWizard` | `packages/tui/src/components/SetupWizard.tsx` | 仅首次运行 |

此外，`App.tsx` 中的 `process.stdin.on('data', onData)` 监听器专门解析 **鼠标滚轮事件**（Xterm 追踪格式），仅作用于 Feed 视图。[来源](../packages/tui/src/components/App.tsx#L87-L88) [来源](../packages/tui/src/utils/mouse.ts#L1-L42)

### 注册顺序与条件守卫

每个 `useInput` 回调必须在入口处使用**条件守卫**（condition guard）判断当前是否属于自己的作用域。`App.tsx` 的处理器排在首位，这意味着它在其他视图处理器**之前**先拿到按键事件。关键在于：

- `App.tsx` 中遇到 `currentView.type === 'aiChat' && focusedPanel === 'ai'` 时会立即 `return`，将控制权让给 AIChatView
- 类似地，`currentView.type === 'search'` 时也会提前 return，把键盘交给 SearchView
- Thread 视图中的 `c` 键被全局处理器跳过：`if (k === 'c') { if (currentView.type !== 'thread') goTo(...); return; }`——这是为了防止全局 `c`（去 compose）和线程本地 `c`（回复）冲突

**核心原则**：高优先级处理器（App.tsx）处理全局键后，必须显式将控制权让给视图本地处理器。[来源](../packages/tui/src/components/App.tsx#L111-L119)

## 全局键（所有视图通用）

### 控制键

| 键 | 行为 | 上下文 |
|----|------|--------|
| `Tab` | 切换 `focusedPanel` 在 `'main'` 和 `'ai'` 之间 | 仅在 aiChat 视图中生效 |
| `Esc` | 见下方表格 | 因视图而异 |
| `,`（逗号） | 打开设置视图（.env 编辑器） | 所有视图 |

### Esc 行为矩阵

| 视图 | 第一次 Esc | 第二次 Esc |
|------|-----------|-----------|
| aiChat + `focusedPanel === 'ai'` | 取消 AI 聚焦 → `focusedPanel = 'main'` | `goBack()` |
| compose + `imagePathInput !== null` | 取消图片输入 | `goBack()`（若草稿非空则弹出保存提示） |
| compose + 无图片输入 | 若草稿非空则弹出保存提示 / 直接 `goBack()` | — |
| feed | 无操作 | — |
| thread / profile / notifications / search / bookmarks | `goBack()` | — |

### Ctrl+G

| 键 | 行为 |
|----|------|
| `Ctrl+G` | 启动 AI 聊天，附带当前线程 URI 作为上下文（仅 thread 视图有 threadUri） |

### 全局键保留规则

以下键在**所有视图中永久保留**，视图本地处理器不得重复使用：

| 键 | 原因 |
|----|------|
| `t`, `n`, `p`, `s`, `a`, `c`, `b` | 全局导航 |
| `Esc` | 万能返回键 |
| `Tab` | AI 面板聚焦切换 |
| `Ctrl+G` | AI 聊天启动器 |
| `,` | 设置 |

可用的视图本地键池：`f`, `z`, `x`, `w`, `u`, `o`, `g`, `q`, `e`, `d`（书签中已使用）、`l`（线程/AI 历史中已使用）、`h`（线程中已使用）、`y`（线程中已使用）、`i`（compose 中已使用）。

### Compose 视图

键盘焦点委托给 `TextInput` 组件（通过 `onSubmit`）。全局快捷键在 compose 中被**屏蔽**。

| 键 | 行为 |
|----|------|
| `Enter` | 发送帖子（通过 TextInput `onSubmit`） |
| `Esc` | 返回（若草稿非空则弹出保存提示） |

## 添加新快捷键的 5 步规范流程

此流程是所有开发者添加新快捷键时必须遵守的规范，也作为 AI Agent 审查清单的依据（详见 `AGENTS.md`）。

### 第一步：检查全局键保留表

确认新键不在 `t`, `n`, `p`, `s`, `a`, `c`, `b`, `Esc`, `Tab`, `Ctrl+G`, `,` 之列。若冲突，**必须**选择其他键。

### 第二步：检查冲突表

确认新键在所有视图中没有冲突含义。例如，`f` 在 Feed 中是切换 Feed，在 Thread 中是翻译——如果要在第三个视图中添加新含义，需确认是否产生歧义。

### 第三步：添加 useInput / switch case

在对应的组件中添加按键处理逻辑：

- 全局键 → `App.tsx` 的 `useInput` 回调
- Feed/Bookmarks/Compose 的视图键 → 同样在 `App.tsx` 中处理（这些视图无独立 `useInput`）
- Thread 键 → `UnifiedThreadView.tsx` 的 `useInput`
- AI Chat 键 → `AIChatView.tsx` 的对应 `useInput`
- Notifications 键 → `NotifView.tsx` 的 `useInput`

每个处理函数必须使用条件守卫确保不会越界处理。若新键涉及对话框（如转发确认、删除确认），需在对话框打开时**屏蔽所有其他键**。

### 第四步：更新 i18n 底部提示

在 `packages/app/src/i18n/locales/` 目录的三个语言文件中更新对应的 `keys.*` 字符串：

- `en.ts` — `'keys.feed'`, `'keys.thread'`, `'keys.aiChat'` 等
- `zh.ts` — 对应中文翻译
- `ja.ts` — 对应日文翻译

底部提示位于每个视图的标题栏右侧，格式示例：`↑↓/jk:导航 Enter:查看 m:更多 r:刷新 f:切换Feed v:收藏 q:引用`。

### 第五步：更新本文档

在 `docs/KEYBOARD.md` 中添加新键的条目，同步更新全局键保留表、冲突表和对应的视图键说明表。若引入新的跨视图冲突，还必须更新冲突表。

附加要求见 `AGENTS.md` 中的强制审查步骤。[来源](../AGENTS.md#L65-L71)

## 技术要点

### Ink 的 useInput 注册顺序

Ink 使用**先注册先触发**策略。`App.tsx` 的 `useInput` 在 `renderView()` 之前注册，因此它始终是最早处理按键的处理器。这意味着：

1. `App.tsx` 的全局键守卫最先执行
2. 只有 `App.tsx` 显式 `return` 后，子视图（`UnifiedThreadView`、`AIChatView` 等）的 `useInput` 才会收到按键

### 条件守卫的最佳实践

每个 `useInput` 回调的典型结构：

```
if (对话框模式) { 处理对话框键; return; }
if (不属于本视图) { return; }  // 让其他处理器处理
if (屏蔽条件) { return; }       // 如 compose/search 中的全局键
处理本地键
```

违反此模式的后果：一个按键可能被多个处理器意外消费，导致功能冲突或静默失效。

### ink-text-input 的键盘接管

在 Compose 和 AI Chat（聚焦时）中，`TextInput` 组件接管了键盘输入。全局 `useInput` 处理器**不会收到**字符键事件，但仍能收到控制键（如 `Esc`）。这是 Ink 的设计行为——`TextInput` 在内部调用 `useInput` 时设置了更高的优先级（通过 `isActive` 属性控制）。
