# 导航路由与视图管理

整个应用的导航系统围绕一个纯 TypeScript 的 **栈式状态机** 构建：一个 `AppView` 联合类型定义了所有可能的视图，一个 `stack: AppView[]` 数组管理浏览历史。TUI（终端）和 PWA（网页）共享这套核心逻辑，仅在"如何持久化并同步导航状态"上分岔。

> 核心代码位于 `packages/app/src/state/navigation.ts`，双端通过 `packages/app` 包的导出入口导入。参见[三层架构设计](三层架构设计.md)了解包层级关系。

---

## AppView：视图的联合类型

`AppView` 是一个 **受歧视联合类型**（discriminated union），`type` 字段作为判别器。当前共定义 9 种变体：

```typescript
export type AppView =
  | { type: 'feed';               feedUri?: string }       // 时间线/自定义 Feed
  | { type: 'detail';             uri: string }             // 未使用，保留类型
  | { type: 'thread';             uri: string }             // 帖子线程详情
  | { type: 'compose';            replyTo?: string; quoteUri?: string }  // 发帖/回复/引用
  | { type: 'profile';            actor: string; profileTab?: string }   // 用户主页
  | { type: 'notifications' }                                // 通知列表
  | { type: 'search';             query?: string; searchTab?: string }   // 搜索
  | { type: 'aiChat';             sessionId?: string; contextPost?: string;
                                   contextProfile?: string; contextUri?: string }  // AI 对话
  | { type: 'bookmarks' };                                   // 书签
```

[来源](navigation.ts#L1-L11)

**设计要点**：
- 每个变体的负载数据（uri、actor、query 等）直接放在类型中，无需路由参数解析——这是 TypeScript 类型的天然优势。
- `detail` 变体已定义但未在任何客户端渲染分支中匹配，属于遗留类型。
- `profileTab` 和 `searchTab` 是可选字段，用于记住用户在子标签页间的选择（如"帖子"、"媒体"、"赞"）。

## createNavigation：纯状态管理

`createNavigation()` 是一个零依赖的工厂函数，返回一个拥有 `subscribe/notify` 模式的微型 Store。

```typescript
export function createNavigation() {
  let stack: AppView[] = [{ type: 'feed' }];   // 初始视图
  let listeners: Array<() => void> = [];

  function getState(): NavigationState {
    return {
      currentView: stack[stack.length - 1]!,   // 栈顶 = 当前视图
      canGoBack: stack.length > 1,
      stack,
      goTo, goBack, goHome,
    };
  }

  function goTo(view: AppView) {
    stack = [...stack, view];   // 推入新视图，不可变更新
    notify();
  }

  function goBack() {
    if (stack.length > 1) {
      stack = stack.slice(0, -1);  // 弹出栈顶
      notify();
    }
  }

  function goHome() {
    stack = [{ type: 'feed' }];   // 重置到 Feed
    notify();
  }

  return { getState, subscribe, goTo, goBack, goHome };
}
```

[来源](navigation.ts#L21-L62)

**核心行为**：
- 导航即压栈：`goTo({ type: 'thread', uri })` 将新视图推入 `stack`，栈顶始终是当前视图。
- 返回即弹栈：`goBack()` 删除栈顶元素，但不保留"前进"历史——这是一个简化设计。
- `canGoBack` 依赖 `stack.length > 1`，栈中始终至少保留 Feed 视图作为根节点。

这与浏览器的 History API 不同。它是为 TUI 设计的：终端没有浏览器的前进/后退按钮，`canGoBack` 直接决定是否显示返回提示。

## useNavigation：React 封装

`useNavigation()` Hook 用 `useState` 包裹 `createNavigation` 实例，并通过 `subscribe` 订阅状态变更：

```typescript
export function useNavigation() {
  const [nav] = useState(() => createNavigation());  // 惰性初始化
  const [state, setState] = useState(() => nav.getState());

  const tick = useCallback(() => setState(nav.getState()), [nav]);

  useEffect(() => nav.subscribe(tick), [nav, tick]);

  return {
    currentView: state.currentView,
    canGoBack: state.canGoBack,
    goTo: (v: AppView) => nav.goTo(v),
    goBack: () => nav.goBack(),
    goHome: () => nav.goHome(),
  };
}
```

[来源](useNavigation.ts#L1-L20)

这个 Hook 在 `packages/app` 包中定义，TUI 直接使用。PWA **不使用它**，而是使用 `useHashRouter`——下文详述。

## TUI 端：直接调用 goTo/goBack

TUI 的导航流程最为直接：

```
useNavigation() ──→ currentView ──→ App.tsx: renderView()
                         │                  switch(currentView.type)
                    goTo/goBack             case 'feed' → <PostList />
                          ↑                  case 'thread' → <UnifiedThreadView />
                    Keyboard shortcuts       case 'profile' → <ProfileView />
                    Sidebar clicks           ...
```

[来源](tui/App.tsx#L46-L46)

键盘快捷键和 Sidebar 的点击事件直接调用 `goTo`：

```typescript
// 键盘快捷键
if (k === 'n') { goTo({ type: 'notifications' }); return; }
if (k === 'p') { goTo({ type: 'profile', actor: config.blueskyHandle }); return; }
if (k === 'a') { goTo({ type: 'aiChat', sessionId: crypto.randomUUID(), ... }); return; }
```

[来源](tui/App.tsx#L309-L314)

TUI 的 Sidebar 用硬编码的 TAB 列表枚举所有可导航视图，包括 feed、notifications、search、profile、bookmarks、aiChat、compose。[来源](tui/Sidebar.tsx#L24-L31)

TUI 也不处理浏览器的前进/后退——终端没有这个概念。`stack` 数组仅在内存中维护，刷新终端即丢失状态。

## PWA 端：useHashRouter 与哈希 URL 编解码

PWA 不能使用内存栈：用户期望浏览器地址栏可分享、可刷新、可前进/后退。解决方案是 **`useHashRouter`**——一个将 `AppView` 与 URL `#/path?params` 双向编解码的 Hook。

### 编码方向

```typescript
function encodeView(view: AppView): string {
  switch (view.type) {
    case 'feed':     return view.feedUri ? `#/feed?feed=${encodeURIComponent(view.feedUri)}` : '#/feed';
    case 'thread':   return `#/thread?uri=${encodeURIComponent(view.uri)}`;
    case 'profile':  return `#/profile?actor=${encodeURIComponent(view.actor)}`;  // 可选 &tab=...
    case 'search':   return view.query ? `#/search?q=${encodeURIComponent(view.query)}` : '#/search';
    case 'compose':  return `#/compose?replyTo=...&quoteUri=...`;
    case 'aiChat':   return `#/ai?session=${encodeURIComponent(view.sessionId)}`;  // 可选 &post= &profile=
    case 'notifications': return '#/notifications';
    case 'bookmarks':     return '#/bookmarks';
    default:          return '#/feed';
  }
}
```

[来源](useHashRouter.ts#L141-L167)

### 解码方向

`parseHash()` 读取 `window.location.hash`，按路径分发到各 `AppView` 变体。例如 `#/profile?actor=did:plc:abc` 解码为 `{ type: 'profile', actor: 'did:plc:abc' }`。[来源](useHashRouter.ts#L75-L139)

### 与浏览器历史同步

`useHashRouter` 使用 `history.pushState` 和 `popstate` 事件：

```typescript
const goTo = useCallback((view: AppView) => {
  const hash = encodeView(view);
  window.history.pushState(null, '', hash);   // 更新地址栏
  setCurrentView(view);
  setCanGoBack(true);
}, []);

const goBack = useCallback(() => {
  if (window.location.hash && window.location.hash !== '#/feed') {
    window.history.back();   // 触发 popstate
  }
}, []);

// 监听浏览器前进/后退按钮
useEffect(() => {
  const handler = () => {
    const view = parseHash();
    setCurrentView(view);
    // ...
  };
  window.addEventListener('popstate', handler);
  return () => window.removeEventListener('popstate', handler);
}, []);
```

[来源](useHashRouter.ts#L20-L72)

关键差异点：

| 特性 | TUI (useNavigation) | PWA (useHashRouter) |
|------|-------------------|---------------------|
| 状态存储 | 内存栈 `AppView[]` | 浏览器地址栏 `#hash` |
| 持久化 | 无（刷新即失） | 可刷新、可分享链接 |
| 前进/后退 | 仅 `goBack()` 弹栈，无前进 | 浏览器原生前进/后退按钮 |
| 初始视图 | `stack = [{ type: 'feed' }]` | 解析当前 `location.hash`，若为空则重定向到默认 Feed |
| 依赖 | 仅 TypeScript | 浏览器 API (`history`, `popstate`) |
| `canGoBack` 判定 | `stack.length > 1` | `hash !== '#/feed'` |

PWA 在首次挂载时还会做一项额外工作：如果 `hash` 为空或只有 `#/feed`，则根据用户配置的默认 Feed URI（来自 `getFeedConfig()`）重定向，确保新用户首屏看到正确的 Feed。[来源](useHashRouter.ts#L32-L37)

## 导航与状态管理的协同

导航系统与其他状态模块的交互遵循清晰的模式：

1. **Feed → Thread**：帖子列表项点击后调用 `goTo({ type: 'thread', uri })`，线程视图通过 `usePostDetail` Hook 加载数据。
2. **Thread → Compose**：回复按钮调用 `goTo({ type: 'compose', replyTo: uri })`，`useCompose` Hook 接收 `goBack` 回调，发帖成功后自动返回。[来源](useCompose.ts#L22-L22)
3. **AI Chat 上下文注入**：从帖子或用户主页发起 AI 对话时，传递 `contextPost` 或 `contextProfile` 给 `aiChat` 视图，`useAIChat` 利用这些上下文初始化提示。[来源](usePostDetail.ts#L55-L58)

参见 [状态管理模式](状态管理模式.md) 了解 Store 层的 subscribe/notify 模式，以及 [useAIChat: 深度解析](useAIChat-深度解析.md) 了解上下文如何从导航流入对话。
