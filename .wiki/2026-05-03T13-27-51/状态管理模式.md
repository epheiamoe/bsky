# 状态管理模式

这个项目没有使用 Redux、Zustand、Zustand、Jotai 或任何第三方状态管理库。状态管理只有两种原材料：**纯 TypeScript 工厂函数**和 **React Hooks**。这种选择并非偷懒，而是对依赖层级和端间复用的刻意设计。

---

## 核心模式：纯 Store + Hook 订阅

整个状态管理建立在一条简洁的契约之上：

1. **Store** 是纯 TS 工厂函数返回的普通对象，零 React 依赖。
2. **Store** 暴露一个 `subscribe(fn)` 方法和一个 `_notify()` 内部触发方法。
3. **Hook** 通过 `useState` 创建 Store 实例，用 `useEffect` 注册订阅，收到通知后 `forceUpdate` 触发重渲染。

[来源](docs/HOOKS.md#L26-L42)

### 两种订阅变体

虽然概念上统一，但代码中存在**单监听器**和**监听器数组**两种实现，对应不同的使用场景。

**变体 A：单监听器（`createAuthStore`、`createTimelineStore`、`createPostDetailStore`）**

```typescript
// stores/auth.ts
export function createAuthStore(): AuthStore {
  const store: AuthStore = {
    client: null,
    session: null,
    profile: null,
    loading: false,
    error: null,
    listener: null,                     // 单个插槽

    _notify() { if (store.listener) store.listener(); },
    subscribe(fn) {
      store.listener = fn;              // 直接覆盖
      return () => { store.listener = null; };  // 取消订阅即清空
    },
    // ... login, restoreSession ...
  };
  return store;
}
```

[来源](packages/app/src/stores/auth.ts#L18-L66)

**变体 B：监听器数组（`createNavigation`）**

```typescript
// state/navigation.ts
export function createNavigation() {
  let stack: AppView[] = [{ type: 'feed' }];
  let listeners: Array<() => void> = [];     // 数组，支持多个

  function subscribe(fn: () => void) {
    listeners.push(fn);
    return () => { listeners = listeners.filter(l => l !== fn); };
  }

  function notify() {
    listeners.forEach(fn => fn());           // 广播所有
  }
  // ... goTo, goBack, goHome ...
}
```

[来源](packages/app/src/state/navigation.ts#L21-L63)

为什么会有这种差异？因为 `createNavigation` 是一个**全局单例**（整个应用只有一个导航栈），理论上可以被多个 Hook 或组件同时订阅。而 `createAuthStore`、`createTimelineStore` 等是**实例化 store**——每个 `useAuth()` 调用都创建独立的 `createAuthStore()` 实例，实例与组件一一对应，单监听器绰绰有余。

## Hook 层的两种订阅风格

### 风格 1：快照式（`useNavigation`）

```typescript
export function useNavigation() {
  const [nav] = useState(() => createNavigation());
  const [state, setState] = useState(() => nav.getState());  // 全量快照

  const tick = useCallback(() => setState(nav.getState()), [nav]);

  useEffect(() => nav.subscribe(tick), [nav, tick]);

  return {
    currentView: state.currentView,
    canGoBack: state.canGoBack,
    goTo: (v: AppView) => nav.goTo(v),
    goBack: () => nav.goBack(),
    goHome: () => nav.goHome(),
  };
}
```

[来源](packages/app/src/hooks/useNavigation.ts#L1-L21)

`setState(nav.getState())` 每次通知都取一个完整的状态对象。由于 `getState()` 返回一个新对象（`{ currentView, canGoBack, stack, goTo, goBack, goHome }`），每次 `setState` 都会触发 React 重渲染。这里 **不需要 `forceUpdate`**，因为 `setState` 传入一个新对象自然触发 re-render。

### 风格 2：forceUpdate 式（`useAuth`、`useTimeline`）

```typescript
export function useAuth() {
  const [store] = useState(() => createAuthStore());
  const [, force] = useState(0);                      // 仅用计数器
  const tick = useCallback(() => force(n => n + 1), []);

  useEffect(() => store.subscribe(tick), [store, tick]);

  return {
    client: store.client,          // 直接读取 store 属性
    session: store.session,
    profile: store.profile,
    loading: store.loading,
    error: store.error,
    login: (h, p) => store.login(h, p),
    restoreSession: (s) => store.restoreSession(s),
  };
}
```

[来源](packages/app/src/hooks/useAuth.ts#L1-L22)

这里的 `force(n => n + 1)` 是经典的 **forceUpdate** 模式：通过改变 `useState` 的计数器值强制 React 重渲染，然后从 store 对象上直接读取最新属性。因为 store 对象本身在 `useState` 初始化后**不变**（引用稳定），Hook 返回的 `client`、`session` 等是 store 上的可变属性——每次重渲染重新读取，总是最新的。

`useTimeline` 同样采用此模式：

```typescript
const [, force] = useState(0);
const tick = useCallback(() => force(n => n + 1), []);
useEffect(() => store.subscribe(tick), [store, tick]);
```

[来源](packages/app/src/hooks/useTimeline.ts#L7-L10)

## 为什么选择单监听器而非 Redux/Zustand

### 1. 零依赖，跨端共享

Store 在 `packages/app` 层，**不依赖 React**。这意味着 TUI（Ink React）和 PWA（React DOM）可以直接复用同一份 store。Redux 或 Zustand 虽然也提供独立的 store，但其生态工具链（中间件、devtools）引入了 TUI 场景不需要的浏览器假设。

```
┌─────────────────────────────────────┐
│  packages/app/stores/auth.ts        │ ← 纯 TS，无 React 导入
│  packages/app/state/navigation.ts   │ ← 纯 TS，无 React 导入
├─────────────────────────────────────┤
│  packages/app/hooks/useAuth.ts      │ ← React 层，导入 store
│  packages/app/hooks/useNavigation.ts│
├──────────────────┬──────────────────┤
│ tui/ (Ink)       │ pwa/ (React DOM) │ ← 消费相同 hooks
└──────────────────┴──────────────────┘
```

详见 [三层架构设计](三层架构设计.md) 中 core→app→tui/pwa 的依赖方向。

### 2. 简单即安全

单监听器模型的核心优势：**不会被多个订阅者意外覆盖**。由于每个 Hook 调用都通过 `useState(() => createStore())` 创建独立的 store 实例，组件树中同一 store 类型的不同实例天然隔离。

```typescript
// PostA 组件
const storeA = useState(() => createAuthStore())[0];  // 实例1
// PostB 组件  
const storeB = useState(() => createAuthStore())[0];  // 实例2
// 两者互不干扰
```

当 store 是**全局单例**（如 `createNavigation`）时，采用数组监听器支持多个订阅。当 store 是**实例化**（如 auth、timeline）时，单监听器足够且无需遍历。

### 3. 无中间件，无样板代码

Redux 的 action/reducer/dispatch 三角关系、Zustand 的 `set` 代理——这些抽象对于这个项目的需求而言是过度设计。需要的只是：

- 一个对象保存状态
- 一个方法通知变更
- 一个方法注册通知回调

三样东西，加起来不到 20 行。

## 多 Store 的隔离优势

项目目前有 7 个 store 工厂（auth、timeline、postDetail、navigation、feedConfig、viewStateStore、drafts），每个覆盖一个独立的领域：

| Store 工厂 | 职责 | 单例/实例 |
|---|---|---|
| `createAuthStore` | 认证状态、会话恢复 | 实例 |
| `createTimelineStore` | 时间线帖子列表、分页 | 实例 |
| `createPostDetailStore` | 帖子详情、线程扁平化、翻译缓存 | 实例 |
| `createNavigation` | 导航栈、AppView 类型系统 | 单例 |
| `createDraftsStore` | 草稿保存与加载 | 实例 |
| `createFeedConfig` | Feed 配置 | 实例 |
| `createViewStateStore` | 视图级状态 | 实例 |

[来源](packages/app/src/hooks/useNavigation.ts) | [来源](packages/app/src/stores/auth.ts) | [来源](packages/app/src/stores/timeline.ts)

这种细粒度隔离的好处：

- **按需加载**：组件只订阅自己需要的 store，一个时间线组件不会因为认证状态变化而重渲染。
- **独立演化**：修改 `createAuthStore` 的内部实现不影响 `createTimelineStore`，无需触发全局类型变更。
- **测试友好**：每个 store 是一个纯 TS 对象，创建即用，无需 mock 全局上下文。

## 订阅的生命周期管理

Hook 利用 `useEffect` 的清理机制确保订阅正确销毁：

```typescript
useEffect(() => store.subscribe(tick), [store, tick]);
//               │           │
//               │           └─ subscribe 返回 () => { store.listener = null; }
//               │
//               └─ useEffect 的清理函数自动调用这个返回函数
```

[来源](docs/HOOKS.md#L40-L45)

当组件卸载时，`useEffect` 的清理函数自动执行 `subscribe` 返回的取消订阅函数。对单监听器模式，这会把 `store.listener` 设回 `null`；对数组模式，会从 `listeners` 数组中移除该回调。

**注意**：`subscribe` 必须在 `useEffect` 中注册，而非在 Hook 主体中直接执行。如果直接执行 `store.subscribe(tick)`，每次渲染都会注册一个新监听器且无法正确清理，导致内存泄漏和重复渲染。

## 设计演进：从数组到单监听器（或反之）

对比 `createNavigation` 和 `createAuthStore` 可以发现，项目内部也存在设计风格的分歧。这并非架构不统一，而是**根据使用场景灵活选择**：

- 如果你的 store 是**组件局部的实例**（一个组件搭配一个 store），用单监听器——更简单，无遍历开销。
- 如果你的 store 是**全局共享的单例**（整个应用一份），用数组监听器——允许多个消费者。

两种模式共享同一组接口契约（`subscribe: (fn: () => void) => () => void`），Hook 层的消费方式完全一致。这种 **duck typing 式的接口兼容**使得切换实现风格时无需修改任何 consumer 代码。

## 与 React 渲染的关系

### 为什么不需要 `useSyncExternalStore`

React 18 引入了 `useSyncExternalStore` 用于外部 store 的并发安全订阅。这里没有使用它，原因有二：

1. **所有 store 在 Hook 内创建**：store 与组件生命周期绑定，不存在"外部"store 在并发模式下读取不一致的问题。
2. **forceUpdate 始终在微任务中触发**：store 的 `_notify()` 由 async 操作（API 请求）的 `finally` 块调用，天然位于正确的 React 批次中。

### 不变性（Immutability）的处理

没有强制 immutability。store 上的属性（如 `store.client`、`store.session`）直接原地更新。Hook 依靠 `forceUpdate` 触发重渲染后重新读取，而不是依靠引用变化来判断是否渲染。

这意味着：**如果只有纯展示组件**（props-based，不直接读取 store），它们不会自动感知 store 的变化。这恰好符合设计意图——只有调用 Hook 的容器组件（container component）才订阅 store，展示组件通过 props 接收数据。

## 总结

| 维度 | 方案 |
|---|---|
| 状态容器 | 纯 TS 工厂函数返回的普通对象 |
| 依赖 | 零第三方状态库 |
| 订阅机制 | 单监听器（实例 store）/ 数组（全局单例） |
| React 集成 | `useState` 创建实例 → `useCallback` 包装 tick → `useEffect` 注册订阅 |
| 触发渲染 | `forceUpdate`（计数器）或 `setState`（快照对象） |
| 跨端复用 | app 层无 React 依赖，TUI 和 PWA 共享同一套 store + hook |
| 测试 | 直接调用 store 方法，无需渲染组件 |

这种模式的核心哲学可以用一句话概括：**用最简单的工具表达最直接的需求**。当你的状态管理只需要"一个对象 + 一个通知 + 一个订阅"时，任何第三方库都是冗余的。
