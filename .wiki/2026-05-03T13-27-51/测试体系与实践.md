# 测试体系与实践

19 个真实 API 集成测试，零 Mock，零模拟——这是本项目测试体系的核心信条。所有测试直接调用 Bluesky 和 DeepSeek 的生产 API，从认证到 AI 工具调用，每一行断言都在真实环境中验证。

## 测试框架与配置

测试基于 **Vitest 3.x**，配置集中在 `packages/core/vitest.config.ts`：

```typescript
export default defineConfig({
  test: {
    globals: true,
    testTimeout: 60000,    // 全局 60 秒超时
    hookTimeout: 30000,    // beforeAll/afterAll 30 秒超时
  },
});
```

[来源](vitest.config.ts#L1-L9)

全局超时设为 60 秒而非 Vitest 默认的 30 秒，这是因为真实 API 调用受网络延迟和 Bluesky 索引延迟影响。个别耗时长的大用例（如 AI 多轮工具调用）还会通过第三参数进一步扩大超时：`it('...', fn, 120000)`。

测试脚本定义在 `packages/core/package.json` 中：

| 命令 | 作用 |
|------|------|
| `pnpm test` | 运行全部 4 个测试文件 |
| `pnpm test:e2e` | 仅运行 e2e.test.ts，带 verbose 输出 |
| `pnpm test:watch` | 监视模式，开发时增量运行 |

[来源](package.json#L16-L19)

## 四个测试文件的职责边界

四个文件按测试范围分层，从底层 API 到高层 AI 集成，再到全流程验证：

| 文件 | 测试数 | 覆盖范围 | 依赖 |
|------|--------|----------|------|
| `auth.test.ts` | 3 | 登录、handle 解析、获取 Profile | 仅 BskyClient |
| `feed.test.ts` | 8 | 发帖、读线程、扁平化、搜索、图片上传/下载 | BskyClient + tools |
| `ai_integration.test.ts` | 8 | AI 工具调用、翻译、润色、引导问题 | BskyClient + AIAssistant |
| `e2e.test.ts` | 10 | 全流程：认证→发帖→线程→图片→AI→翻译→润色→Profile→通知 | 全部模块 |

合计 29 个测试用例，全部通过耗时约 103 秒。[来源](TEST_REPORT.md#L9-L16)

### auth.test.ts —— 认证三连

三个测试用例构成一条完整的认证链路：

1. **创建会话**：调用 `client.login(HANDLE, APP_PASSWORD)`，验证返回的 `accessJwt` 不为空，`handle` 匹配，`did` 符合 `did:plc:` 格式
2. **解析 handle**：`resolveHandle` 返回的 DID 必须与登录会话中的 DID 一致
3. **获取 Profile**：`getProfile` 返回的 handle 和 did 与登录信息匹配

所有测试共用 `beforeAll` 中的一次登录，每个用例单独设置 15 秒超时。[来源](tests/auth.test.ts#L1-L54)

### feed.test.ts —— 帖子生命周期

覆盖 Bluesky 帖子从创建到搜索再到图片附件的完整路径。实际提交的测试包含搜索功能，而发帖、线程读取、图片上传等写操作已被注释掉（避免在 Bluesky 上残留测试数据）。运行的测试验证：

- **搜索**：搜索 "Bluesky" 关键词，验证返回结果列表
- **工具注册**：`createTools(client)` 返回的工具数量 > 20

被注释的用例展示了完整的帖子生命周期：创建测试帖（带 `[TRST 测试]` 标记）→ 读取线程 → 扁平化序列化 → 搜索验证 → 图片 blob 上传 → 嵌入图片发帖 → 提取图片 DID/CID → 下载 base64。[来源](tests/feed.test.ts#L1-L117)

### ai_integration.test.ts —— AI 自主工具调用

这是最具特色的测试文件，它验证 AI 能否自主选择并执行工具来获取信息。三个核心场景：

**搜索帖子**：AI 收到指令 "搜索包含 Bluesky 的帖子"，必须自主调用 `search_posts` 工具，然后将结果组织成回答。断言：`toolCallsExecuted >= 1`。[来源](tests/ai_integration.test.ts#L88-L108)

**获取 Profile**：AI 收到指令 "查看用户的 Bluesky 个人资料"，自主调用 `get_profile` 工具，返回显示名称和帖子数量。[来源](tests/ai_integration.test.ts#L110-L131)

**翻译与润色**：调用 `translateToChinese` 和 `polishDraft` 两个单回合 AI 函数，验证翻译结果包含中文字符（`/[\u4e00-\u9fff]/` 正则检测），润色后文本长度增长。[来源](tests/ai_integration.test.ts#L137-L175)

这些测试验证了 [AIAssistant 核心对话架构](aiassistant-核心对话架构.md) 中的工具循环执行能力——AI 不是硬编码调用某个函数，而是通过思考自主决定调用哪个工具。

### e2e.test.ts —— 10 步全流程验证

E2E 文件将整个系统的核心能力编排为 10 个顺序执行步骤，编号即顺序依赖：

| 步骤 | 测试内容 | 模块 | 超时 |
|------|----------|------|------|
| [1] | 认证：登录 + resolveHandle + getProfile | BskyClient | 15s |
| [2] | Feed：读取时间线 + 检查工具注册 | BskyClient + tools | 15s |
| [3] | 图片（已注释）：上传 blob → 发图帖 → 提取 → 下载 | BskyClient | 60s |
| [4] | AI 工具调用（已注释）：分析帖子 | AIAssistant | 120s |
| [5] | 翻译：英文→中文 | translateToChinese | 60s |
| [6] | 润色：草稿加工 | polishDraft | 60s |
| [7] | Profile：获取 bsky.app 的资料 + 关注列表 | BskyClient | 15s |
| [8] | 时间线：读取 10 条 | BskyClient | 15s |
| [9] | 通知：列出 10 条 | BskyClient | 15s |
| [10] | AI 引导问题：生成 2 个 | singleTurnAI | 60s |

[来源](tests/e2e.test.ts#L1-L141)

步骤 [1] 和 [2] 是基础，后续只读操作（[7][8][9]）独立运行，步骤 [5][6][10] 不依赖前序结果。注释掉的 [3][4] 包含写操作和 AI 工具调用，需要创建测试帖子作为前置条件。

## 零 Mock 的测试模式

本项目的测试哲学是 **集成测试优先**——所有测试直接连接真实服务，不 Mock 任何网络层。这种模式的优势和代价：

### 直接 API 测试

```typescript
// BskyClient 真实登录
const client = new BskyClient();
await client.login(HANDLE, APP_PASSWORD);
const posts = await client.getTimeline(20);
expect(posts.feed.length).toBeGreaterThan(0);
```

这种方式验证了从 HTTP 请求构造、响应解析到错误处理的全链路。关于 BskyClient 的 API 路由策略，参见 [BskyClient: Bluesky API 封装](bskyclient-bluesky-api-封装.md)。

### AI 真实调用测试

```typescript
const tools = createTools(client);
const assistant = new AIAssistant(AI_CONFIG);
assistant.setTools(tools);
assistant.addSystemMessage('你是一个深度集成 Bluesky 的终端助手。回答简练。');
const result = await assistant.sendMessage('搜索包含 "Bluesky" 的帖子');
expect(result.toolCallsExecuted).toBeGreaterThanOrEqual(1);
```

真实调用确保 AI 的提示词工程（详见[系统提示词工程](系统提示词工程.md)）在实际 LLM 上有效，工具描述（详见[31 个工具系统详解](31-个工具系统详解.md)）能被 AI 正确理解。

### 环境变量加载

测试通过 `dotenv` 从项目根目录加载 `.env` 文件：

```typescript
dotenv.config({ path: path.resolve(__dirname, '..', '..', '..', '.env') });
```

需要三个环境变量：`BLUESKY_HANDLE`、`BLUESKY_APP_PASSWORD`、`LLM_API_KEY`。配置方式详见[环境变量与认证](环境变量与认证.md)。[来源](tests/auth.test.ts#L8-L10)

## 常见问题与修复

| 问题 | 根因 | 修复方式 |
|------|------|----------|
| 测试超时（默认 30s） | 真实 API 调用受网络和索引延迟影响 | 全局超时设为 60s，大用例单独设 `15000`/`60000`/`120000` |
| Blob 下载超时 | 同步下载太快，Bluesky 同步端点未准备就绪 | 下载前加 2s 延迟；使用认证端点 + 30s 单次超时 |
| 搜索返回空结果 | AT Protocol 搜索索引有延迟 | 搜索前加 3s 等待；降低断言期望（接受 0 条结果） |
| AI 回答被截断 | 流式输出未启用，token 限制不足 | 使用非流式模式；`max_tokens: 4096`；终端显示用 Box 包裹 |
| 账户残留测试帖子 | AT Protocol 无便捷删除 API | 发帖使用 `[TRST 测试]` 标记用于识别；接受测试帖永久存在 |

[来源](docs/TESTING.md#L39-L47)

## 如何添加新测试

1. 在 `packages/core/tests/` 下创建 `.test.ts` 文件
2. 导入 `BskyClient`、`createTools`、`AIAssistant` 等核心模块（均来自 `../src/`）
3. 从项目根目录加载 `.env`
4. 编写测试——不 Mock，直接调用真实 API
5. 注意清理：写操作会在 Bluesky 上留下记录，使用标记识别而非依赖删除

[来源](docs/TESTING.md#L49-L57)

## 测试覆盖率的意义

29 个测试覆盖了 [BskyClient: Bluesky API 封装](bskyclient-bluesky-api-封装.md) 的核心端点（认证、时间线、帖子、搜索、图片、Profile、关注、通知）和 [AIAssistant 核心对话架构](aiassistant-核心对话架构.md) 的三种模式（多轮工具调用、单回合翻译、单回合润色）。零 Mock 策略确保每次 CI 运行都是真实环境的质量门禁——如果 Bluesky API 或 DeepSeek API 发生破坏性变更，测试会第一时间报警。
