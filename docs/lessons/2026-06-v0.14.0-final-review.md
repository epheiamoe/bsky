### 核心安全目标评估

**"用户知道标签服务失败了"** — **基本达成**

- 当任意 active labeler 的 API 请求最终失败时，用户通过 banner 或 toast 看到提示
- 默认配置遵循安全优先：`asukafield.xyz`（LGBTQ+ 保护）默认 `block`，官方 moderation 默认 `banner`

### 发现的问题

#### 🔴 UX 缺陷：Banner/Toast 无法关闭或重试（中等影响）

**位置**: 所有 6 个列表组件

**代码**:
```tsx
<LabelerFailureBanner failedLabelers={bannerFailures} />
// 缺少 onDismiss 和 onRetry props
```

**影响**: 用户看到 banner 后无法关闭它。如果标签服务持续失败，banner 会一直占据顶部空间。Retry 按钮在 UI 上存在但点击无效果。

**建议**: 给调用方加上 dismiss 状态（`useState`）和 retry 逻辑。

#### 🟡 边缘情况：`useModerationBatch` 整体异常完全静默（中等影响）

**位置**: `packages/app/src/hooks/useModeration.ts:259-266`

**代码**:
```tsx
}).catch(() => {
  // Silently fail — moderation is best-effort
  if (!cancelled) setResult({
    decisions: new Map(),
    failedLabelers: [],
    isLoading: false,
  });
});
```

**影响**: 如果 `resolveModerationBatch` 因非标签服务原因整体崩溃（代码 bug、内存不足等），用户会看到**完全没有 moderation 过滤的帖子**，且**没有任何失败提示**。这是比"单个 labeler 失败"更危险的状态。

**建议**: 至少记录到 console，或降级为所有 active labelers 失败的提示。

#### 🟢 `retries` 字段命名混淆（轻微）

**位置**: `packages/core/src/moderation-cache.ts:280-289`

`retries` 实际含义是"该 labeler 历史累计失败次数"，不是"已重试的次数"。不影响功能。

## 三、代码质量/bug 审查

### 🔴 Critical（需立即修复）

#### 1. FeedCache 缓存修剪逻辑缺陷 — 缓存可无限增长

**位置**: `packages/app/src/stores/feedCache.ts:54-59`

```typescript
if (combined.length > limit) {
  const excess = combined.length - limit;
  trimmed = Math.min(excess, TRIM_BATCH);
  combined.splice(0, trimmed);
}
```

**问题**: 当 `excess > TRIM_BATCH`（例如当前 1000 条，新增 50 条，`excess=50`，`TRIM_BATCH=25`），只删除 25 条，剩余 1025 条。重复 `loadMore` 时缓存会持续膨胀。

**建议修复**:
```typescript
if (combined.length > limit) {
  combined.splice(0, combined.length - limit);
}
```

#### 2. ContentWarningOverlay 被隐藏内容对屏幕阅读器完全可见

**位置**: `packages/pwa/src/components/ContentWarningOverlay.tsx:32-35`

被 `blur-sm` 覆盖的内容仍存在于 DOM 中，无 `aria-hidden` 或 `inert` 属性。屏幕阅读器用户可完整读取所有被标记为敏感的内容。

**建议修复**:
```tsx
<div className="blur-sm ..." aria-hidden="true" inert>
  {children}
</div>
```

### 🟠 High（本周内修复）

#### 3. `useModerationBatch` 每次 `posts` 变化触发全量重新查询

**位置**: `packages/app/src/hooks/useModeration.ts:233-269`

`useEffect` 依赖 `[posts, config, client]`。`posts` 每次 `loadMore` 后都是新数组引用，导致每次为**所有已加载帖子**重新创建 `LabelCache` 并批量查询。

**影响**: 加载 20 条新帖时可能触发 1000+ URI 的标签查询。

**建议**: 在 `useModerationBatch` 内部用 `useRef` 维护跨渲染的 `LabelCache`，只查询新出现的 URI。

#### 4. ComposePage 组件卸载时 blob URL 泄漏

**位置**: `packages/pwa/src/components/ComposePage.tsx`

`URL.createObjectURL` 创建的预览 URL 只在用户手动点击"移除"时 `revokeObjectURL`。如果用户直接导航离开，所有 blob URL 泄漏。

**建议**: 添加 `useEffect(() => () => { /* cleanup all blob URLs */ }, [])`。

#### 5. `useTimeline` 订阅未返回 cleanup 函数

**位置**: `packages/app/src/hooks/useTimeline.ts:54`

```tsx
useEffect(() => store.subscribe(tick), [store, tick]);
```

忽略了 `subscribe` 返回的 cleanup 函数。

**建议**:
```tsx
useEffect(() => store.subscribe(tick), [store, tick]);
```

#### 6. `FeedTimeline` IntersectionObserver 因 `posts.length` 依赖频繁重建

**位置**: `packages/pwa/src/components/FeedTimeline.tsx:149-158`

`useEffect` 依赖 `posts.length`，每次加载更多后 observer 被断开并重新创建，可能导致重复触发 `loadMore`。

**建议**: 从依赖数组中移除 `posts.length`。

### 🟡 Medium（排期内修复）

- LabelCache 无自动清理机制（`prune()` 存在但从未自动调用）
- `useModeration` 依赖数组引用不稳定导致频繁重渲染
- FeedTimeline 虚拟列表 ref callback 每次渲染重建
- `LabelCache` 失败的 URI 无负缓存，可能持续发送失败请求

### 🟢 Low（可选）

- `HiddenBanner` 缺少 ARIA `aria-expanded`/`role="region"`
- `ModerationLabelBar` "了解详情" 按钮缺少 `aria-expanded`/`aria-controls`
- `FeedTimeline` useEffect 依赖 `scrollRef` 无意义（ref 引用永不改变）
